## 📖 핵심 도메인별 상세 (Domain Deep-Dive)

### 📖-1. Problem (현상 및 데이터)

**현재 아키텍처의 문제점:**
- 웹훅 처리를 **AWS EC2**로 수행 → 인스턴스 사양에 성능이 종속
- 카페24 프로 업체 증가로 웹훅 처리량 급증 → EC2 사양 2배 스케일업 → 비용 증가
- 수천 개 쇼핑몰의 웹훅이 **병렬로 처리**되지만, 최종 저장 플로우가 **한 곳에 집중** → 병목 취약 구조

**Redis 파티셔닝 문제:**
- Redis(ElastiCache)에 웹훅 데이터가 `shop_id`를 **1000 단위**로 그룹화하여 저장
- 특정 쇼핑몰에서 웹훅 대량 발생 시 → **같은 그룹 내 모든 쇼핑몰에 처리 지연이 전파**
- shop_id가 1만 개 증가할 때마다 리스트 10개씩 **수동 추가** 필요
- shop_id 1만 개 추가 주기가 점차 짧아지는 추세 → 수동 운영 부담 가속화

**비용 구조:**
- 웹훅 데이터 캐시 서버 = **서버리스 인스턴스** (용량 유동적, 고정 사양보다 비쌈)
- 병목으로 인해 Redis에 데이터가 장기간 적재 → 용량 기반 **비용 증가**
- 2025-11-10: EC2 인스턴스 사양 2배 증가 → 평균 CPU 사용률 감소했으나 비용 부담 가중

---

### 📖-2. 기존 vs 개선 아키텍처

**기존 플로우:**
```
코어 시스템 서버 → AWS SQS → AWS Step Functions(Lambda) → AWS ElastiCache(Redis) → AWS EC2 → 데이터베이스
```

**개선 플로우:**
```
코어 시스템 서버 → AWS SQS → AWS Lambda → 데이터베이스 (직접 저장)
```

**핵심 변경:** Lambda → ElastiCache → EC2 구간을 **Lambda → DB 직접 저장**으로 단축
- ElastiCache(Redis) 제거 → 서버리스 Redis 비용 삭제
- EC2 역할 축소 → 인스턴스 하향 조정 가능
- Lambda에서 즉시 DB 저장 → 파티셔닝 병목 원천 제거

### 📖-4. 제약사항

| 항목 | 요구사항 |
|---|---|
| 데이터베이스 연결 | 동시 호출되는 여러 Lambda와 연결되는 DB에 **부하가 발생해선 안 됨** |
| 소스코드 형상관리 | Lambda 소스코드는 **서버와 같은 Repository**에서 관리 |
| Lambda 이미지 | **ECR 이미지** 기반 Lambda 세팅 |
| 인프라 구축 | 기존 리소스 변경이 아닌 **새로운 AWS 리소스를 Terraform으로 구축** |
| A/B 테스트 | 일부 업체를 제한하여 도입하고 조절할 수 있는 환경 |

### 📖-5. 리스크 Q&A

**Q1. 다수 Lambda에서 DB 동시 연결 시 문제는 없는가?**
- A. 리서치 및 테스트 필요. (Unknown 상태에서 검증 예정)

**Q2. Lambda가 DB 저장 실패 시 웹훅 데이터 유실 위험은?**
- A. Lambda 처리 실패 시 → 앞단 **SQS에서 메시지가 삭제되지 않음**
- SQS는 Lambda가 메시지를 가져와 **처리 완료해야만 삭제**
- 데이터 보존 기간: **최대 14일**
- ⚠️ 기존 Redis는 메모리에 데이터를 보유했으므로, 실패 시 데이터가 Redis에 남아있었음. 새 구조에서는 SQS가 이 역할을 대체
