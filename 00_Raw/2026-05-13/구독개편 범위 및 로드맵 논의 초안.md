# 구독개편 범위 및 로드맵 논의 초안

> 출처: Claude Code 대화 (2026-05-13)
> 컨텍스트: 이현준님의 "요금제 개편이 꼭 필요할까요" 아젠다(2026-05-04)에 대한 구체적 범위·로드맵 논의

## 개편의 범위 — 3개 레벨

기존 문서의 4단계를 실행 가능성 기준으로 3레벨로 재구성.

| 레벨 | 이름 | 범위 | 변경 대상 | 리스크 |
|------|------|------|-----------|--------|
| L1 | 다듬기 (Polish) | 기존 플랜 구조 유지, UX·금액만 손질 | 금액 조정, 청구서 UX 개선, 요금제 안내 페이지 신설, 종량제 과금 방식 개선 | 낮음 |
| L2 | 통폐합 (Consolidate) | 프로덕트별 플랜/티어 기준 일관성 확보 | 티어 수 축소(10단계→4단계), 주문수 기준 통일, 결합할인 정책 확정 | 중간 |
| L3 | 통합 재설계 (Redesign) | 에이전틱 알파앱스에 맞는 새 구독 체계 | 사용량 기반 단일 플랜 or 통합 플랜+애드온 모델 | 높음 |

추천: L1 + L2 선행, L3는 에이전틱 알파앱스 MVP 이후

## 개편 페이즈 / 로드맵

### Phase 0: 의사결정 (2주, ~5월 말)
- 개편 레벨 확정 (L1/L2/L3 중 어디까지)
- 새 요금 체계 방향 결정 (4가지 모델 중 택1 or 혼합)
- 기존 고객 마이그레이션 원칙 합의 (강제 전환 vs 자연 전환 vs 병행)

### Phase 1: L1 — 다듬기 (6~8주, 6월~7월)
- 금액 조정: 플랜별 가격 재산정, 할인율 정리
- UX 개선: 청구서 가독성, 요금제 안내 페이지 신설
- 종량제 개선: 알림톡/문자 과금 방식 정비, 메시지 포인트 정책 단순화
- 산출물: 변경된 가격표, 새 안내 페이지, 개선된 청구서 UI

### Phase 2: L2 — 통폐합 (8~10주, 8월~10월)
- 티어 통폐합: 프로덕트별 10단계 → 4~5단계로 축소 (Free / Starter / Growth / Business / Enterprise)
- 기준 통일: 전 프로덕트 주문수 기반으로 동일 구간 적용
- 결합할인 확정: 2개 10%, 3개 20% → 정책 확정 및 구현
- 마이그레이션: 기존 고객 자동 매핑 로직 + 불이익 없는 전환 보장
- 산출물: 통합 플랜표, 마이그레이션 스크립트, CS 가이드

### Phase 3: L3 — 통합 재설계 (12~16주, 에이전틱 알파앱스 MVP 이후)
- 선결 조건: 통합 알파앱스 3축(세팅/운영/성장) MVP 런칭
- 새 과금 모델: 사용량 기반 or 통합 플랜+애드온
- 과금 엔진 리빌드: 기존 상태머신 단순화, 프로덕트 단위 → 기능 단위 과금
- 산출물: 신규 구독 체계, 마이그레이션 완료, 기존 체계 일몰(sunset)

## 핵심 논의 포인트

| # | 질문 | 선택지 |
|---|------|--------|
| 1 | 통합 알파앱스 런칭 시점에 새 요금제가 꼭 필요한가? | A) 2분기 메뉴 수정엔 L1이면 충분 / B) 3분기 통합 런칭에 맞춰 L2까지 필요 |
| 2 | 최소 리소스로 동일 임팩트를 낼 방법은? | L1만으로도 요금제 안내 페이지 + 가격 조정으로 고객 불만 80% 해소 가능 |
| 3 | L3는 언제가 적기인가? | 에이전틱 알파앱스가 프로덕트 구분을 해체할 때 비로소 의미 있음 |
| 4 | 마이그레이션 전략 | Grandfather(기존 고객 가격 유지) + 신규만 새 체계 vs 전면 전환 |
| 5 | 2~3달 안에 가능한가? | L1: 가능(6~8주). L2: 빠듯(8~10주). L3: 불가능(12~16주+) |

## 리스크 및 제약

- 현행 상태머신 복잡도: 일할계산, 위약금, is_shared_day 등 레거시 로직이 깊이 얽혀 있어 L3 수준 변경 시 사이드이펙트 큼
- 계약구독 고객: 계좌이체·커스텀 조건 고객은 자동 마이그레이션 불가, 수동 대응 필요
- CS 부담: 요금제 변경 시 CS팀 교육·고객 안내 리소스 필수
- 결합할인 미확정: 현재 정책 미확정 상태 → L1 단계에서 우선 확정 필요
