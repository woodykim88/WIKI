### 읽기 전에 알아야 할 기본 개념

데이터베이스는 엑셀 시트와 비슷합니다. "테이블"은 엑셀의 시트 하나, "필드(컬럼)"는 시트의 열, "레코드(행)"은 시트의 한 줄 데이터에 해당합니다.

테이블끼리 연결되는 방식은 세 가지입니다.

1:1 (One-to-One): 하나의 Shop에 하나의 ShopDetail이 대응. 보조 정보를 분리 저장할 때 사용합니다.

1:N (One-to-Many): 하나의 Shop에 여러 개의 Subscription이 달림. 가장 흔한 관계입니다.

N:M (Many-to-Many): 하나의 테이블 레코드가 다른 테이블의 여러 레코드와 연결되고, 반대도 마찬가지인 관계입니다.

"조인(Join)" 이란 이렇게 연결된 테이블을 합쳐서 보는 것입니다. 예를 들어 "이 쇼핑몰의 구독 정보"를 보려면 Shop 테이블과 Subscription 테이블을

shop\_id

로 조인합니다.중요: 이 시스템에서 대부분의 테이블은

shop\_id

필드를 가지고 있습니다. 따라서 특정 쇼핑몰의 데이터를 볼 때는 거의 모든 테이블을

shop\_id

로 조인할 수 있습니다. 쇼핑몰 ID만 알면 관련 데이터를 쉽게 찾을 수 있습니다.

이 문서에서 사용하는 표기법은 다음과 같습니다.

테이블명.필드명

형태로 표기합니다 (예:

Shop.main\_status

)

숫자 코드가 있는 필드는 코드값과 한글 의미를 함께 표기합니다

FK는 다른 테이블을 가리키는 연결 고리(Foreign Key)를 의미합니다

#### 데이터 변경 이력에 대한 중요 사항

데이터베이스는 현재 상태만 저장합니다. 특정 필드 값이 언제, 무엇에서 무엇으로 바뀌었는지에 대한 이력은 기본적으로 남지 않습니다.

예를 들어, 구독(Subscription)의

main\_status

가 "대기(100)" → "사용중(201)"으로 변경되었을 때, 현재 값이 201이라는 것은 알 수 있지만 언제 201이 되었는지는 알 수 없습니다. 매달 새로운 구독 레코드가 생성되므로 "어떤 구독이 존재하는지"는 확인 가능하지만, 각 구독의 상태가 변경된 정확한 시점은 추적이 불가능합니다.일부 테이블에는

updated\_at

필드가 있지만, 이것도 주의가 필요합니다.

updated\_at

은 레코드의 아무 필드라도 수정되면 갱신되므로, 관심 있는 필드가 바뀐 것인지 다른 필드가 바뀐 것인지 구분할 수 없습니다. 따라서

updated\_at

만으로 특정 상태 변경 시점을 정확히 추적하는 것은 불가능합니다.

별도의 이력(History) 테이블이 있는 경우에만 과거 추적이 가능합니다. (예: DefferedAmountHistory, PaidCashHistory, SubscriptionExecuteHistory 등)

#### 데이터 삭제 정책

코어 데이터베이스는 기본적으로 데이터를 삭제(DELETE)하지 않습니다. 상태 변경(예: main\_status 변경)이나 소프트 딜리트(

is\_deleted

플래그)를 통해 데이터의 유효성을 관리하며, 실제 레코드는 보존됩니다.

예외 — 하드 딜리트가 발생하는 경우:

관리용 Management Command: 운영/데이터 정리 목적의 Django management command에서 하드 딜리트가 수행될 수 있습니다 (예:

delete\_orders

,

delete\_product

,

delete\_customer

,

delete\_category

등). 이들은 일반적인 서비스 흐름이 아닌 관리자가 직접 실행하는 명령어입니다.CASCADE 삭제: 일부 FK 관계에

on\_delete=CASCADE

가 설정되어 있어, 부모 레코드가 삭제되면 연관 레코드도 함께 삭제됩니다. 예를 들어 Shop이 삭제되면 해당 쇼핑몰의 Product, Order, Customer 등이 연쇄 삭제됩니다. 단, Shop 자체를 삭제하는 일은 정상적인 운영 흐름에서 발생하지 않습니다.이커머스 플랫폼 데이터: 플랫폼에서 동기화되는 데이터(Product, Customer, Category 등)는

is\_deleted

필드를 이용한 소프트 딜리트로 관리됩니다. 자세한 내용은 3장을 참고하세요.

정리하면, 일반적인 서비스 흐름(API, 배치 작업)에서는 데이터가 삭제되지 않으며, 삭제가 필요한 경우에도 소프트 딜리트를 우선으로 사용합니다.

## 1장. 쇼핑몰 관련 데이터

### 1.1 회사-쇼핑몰 관계

하나의 회사(Company)가 여러 쇼핑몰(Shop)을 운영할 수 있습니다 (멀티쇼핑몰).

Company 테이블:

| 필드 | 설명 |
| --- | --- |
| mall\_id | 몰 ID |
| platform\_id | 이커머스 플랫폼의 회사 ID |

### 1.4 앱(App)과 서비스(Service) — 왜 분리되어 있는가

앱(App) 은 우리 제품의 큰 분류이고, 서비스(Service) 는 앱 내부의 세부 기능 단위입니다. 이 둘이 분리되어 있는 이유는 역할이 다르기 때문입니다.

앱(App)의 역할: 제품 단위의 기능 접근 제어와 설치 추적. "이 쇼핑몰이 알파리뷰를 쓰고 있는가?"를 확인할 때 앱 단위로 봅니다. 웹훅 라우팅, 스크립트 설치, 대시보드 권한 등이 앱 단위로 관리됩니다.

서비스(Service)의 역할: 과금 단위와 세부 기능 구분. 하나의 앱 안에 여러 서비스가 있을 수 있고, 각 서비스별로 가격이 다르게 책정됩니다. 구독/청구서/설치요청은 서비스 단위로 연결됩니다.

#### ShopApp (쇼핑몰-앱 연결) — 시점 필드

ShopApp은 특정 쇼핑몰(Shop)에 특정 앱(App)이 연결된 상태를 나타내는 테이블입니다. 조인:

Shop.id = ShopApp.shop\_id

,

App.id = ShopApp.app\_id

| 필드 | 설명 | 업데이트 시점 | 값의 의미 |
| --- | --- | --- | --- |
| installed\_at | 최초로 앱이 설치된 시점 | 해당 앱의 메인 서비스가 "프로덕트온보딩중" 상태로 전환될 때 최초 1회만 기록됨 (이미 값이 있으면 갱신하지 않음) | 이 쇼핑몰에 이 앱이 처음 설치된 날짜. 재설치해도 값이 바뀌지 않으므로 "최초 도입일"로 해석 가능 |
| main\_status | 메인 상태 | 해당 앱의 메인 서비스(ShopService)의 상태 변경에 연동되어 자동 갱신됨 | ShopService의 main\_status와 동일한 값 체계 사용 |
| removed\_at | 마지막으로 앱이 삭제된 시점 | 현재 사용되지 않음 (레거시 필드) | \- |
| is\_deleted | 앱 삭제 여부 | 현재 사용되지 않음 (레거시 필드) | \- |
| deleted\_at | 앱 삭제 시점 | 현재 사용되지 않음 (레거시 필드) | \- |

참고:

installed\_at

은 쇼핑몰의 타임존(예: "Asia/Seoul") 기준으로 기록됩니다(DB에는 UTC가 저장됨).

#### ShopService (쇼핑몰-서비스 연결) — 시점 필드

ShopService는 특정 쇼핑몰(Shop)에 특정 서비스(Service)가 연결된 상태를 나타내는 테이블입니다. 조인:

Shop.id = ShopService.shop\_id

,

Service.id = ShopService.service\_id

| 필드 | 설명 | 업데이트 시점 | 값의 의미 |
| --- | --- | --- | --- |
| installed\_at | 마지막으로 서비스가 설치된 시점 | 서비스가 "프로덕트온보딩중" 상태로 전환될 때마다 갱신됨 | 가장 최근에 이 서비스가 설치(온보딩 시작)된 날짜. 재설치 시 덮어씌워지므로 "마지막 설치일"로 해석 |
| live\_at | 마지막으로 서비스가 라이브된 시점 | 서비스가 "라이브" 상태로 전환될 때마다 갱신됨 | 가장 최근에 이 서비스가 정상 운영(라이브) 상태가 된 날짜. 재라이브 시 덮어씌워짐 |
| stopped\_at | 마지막으로 서비스가 중단된 시점 | 서비스가 "서비스중단" 상태로 전환될 때마다 갱신됨 | 가장 최근에 이 서비스가 중단된 날짜. 재중단 시 덮어씌워짐 |

ShopService의 상태(main\_status) 흐름과 시점 필드 관계:

구독없음 (100) │ ▼ 프로덕트온보딩중 (200) ──── installed\_at 갱신 │ ▼ 라이브 (300) ──────────── live\_at 갱신 │ ▼ 서비스중단 (500) ────────── stopped\_at 갱신

주의사항:

모든 시점 필드는 쇼핑몰의 타임존(예: "Asia/Seoul") 기준으로 기록됩니다(DB에는 UTC 값이 저장됨).

ShopApp의

installed\_at

은 최초 1회만 기록되지만, ShopService의

installed\_at

,

live\_at

,

stopped\_at

은 해당 상태 전환이 발생할 때마다 덮어씌워집니다. 따라서 ShopService의 시점 필드는 "마지막" 시점만 확인 가능하고, 과거 이력은 남지 않습니다.

상태 전환은 반드시 정해진 순서대로만 가능합니다.

| 코드 | 플랜명 | 설명 |

| 1 | 무료 | 무료 플랜 |
| 2 | 체험 | 체험 플랜 |
| 300 | 라이트 | 소규모 쇼핑몰 대상 |
| 400 | 베이직 | 기본 유료 플랜 |

### 1.6 설치/제거 요청(InstallRemoveRequest)

쇼핑몰에 서비스를 설치하거나 제거할 때 생성되는 요청 기록입니다. InstallRemoveRequest는 Service를 FK로 참조하므로, 어떤 서비스에 대한 요청인지

service\_id

로 확인할 수 있습니다.조인 방법:

Shop.id = InstallRemoveRequest.shop\_id

,

Service.id = InstallRemoveRequest.service\_id

| 필드 | 설명 |
| --- | --- |
| shop\_id | 쇼핑몰 FK |
| service\_id | 설치/제거 대상 서비스 (FK → Service) |
| type | 요청 유형 (아래 참조) |
| main\_status | 요청 상태 (아래 참조) |
| is\_main | 메인 요청 여부 |
| reason | 설치/제거 사유 (선택) |
| data | 추가 데이터 (JSON, 선택) |
| assistant\_bill\_id | 연결된 보조 청구서 FK (선택) |
| ticket\_id | 연결된 티켓 FK (선택) |
| created\_at | 요청 생성일 |
| updated\_at | 최종 수정일 |

#### 요청 유형(type) — 언제 어떤 유형이 생성되는가

| 코드 | 유형 | 언제 생성되는가 |
| --- | --- | --- |
| 101 | 자동설치 |  |
| 102 | 직접설치 | 관리자(어드민)가 직접 설치 버튼을 눌렀을 때. 즉시 진행중 상태로 전환됨 |
| 103 | 설치요청 | 쇼핑몰 측에서 설치를 요청했을 때. 알파리뷰 설치 시 주로 이 유형이 생성됨. 관리자가 "접수"할 때까지 대기 상태 |
| 201 | 자동제거 | 시스템이 자동으로 제거할 때 (구독 종료 등). 즉시 완료 처리됨 |
| 202 | 직접제거 | 관리자가 직접 제거할 때. 즉시 진행중 상태로 전환됨 |
| 203 | 제거요청 | 쇼핑몰 측에서 제거를 요청했을 때. 관리자가 "접수"할 때까지 대기 상태 |

#### 요청 상태(main\_status) — 상태 전이 흐름

대기(100) ─── "접수" 버튼 ───→ 진행중(200) ─── 작업 완료 ───→ 완료(300) │ │ ↑ │ ├── 작업 실패 ──→ 실패(600) ──┘ (재시도 가능) │ │ │ └── "취소" 버튼 ──→ 취소(400) │ ├── "취소" 버튼 ──→ 취소(400) │ └── "삭제" 버튼 ──→ 제거(500) (요청 자체를 삭제, 연관 청구서도 취소됨)

| 코드 | 상태 | 설명 |
| --- | --- | --- |
| 100 | 대기 | 요청이 생성되어 관리자의 접수를 기다리는 상태 |
| 200 | 진행중 | 관리자가 "접수"하여 스크립트 설치/데이터 마이그레이션 등이 진행 중 |
| 300 | 완료 | 설치 또는 제거가 정상 완료됨 |
| 400 | 취소 | 관리자가 요청을 취소함. 설치요청 취소 시 이미 설치된 스크립트도 삭제됨 |
| 500 | 제거 | 대기 상태에서 요청 자체를 삭제함. 연관된 보조 청구서도 함께 취소됨 |
| 600 | 실패 | 진행 중 데이터 마이그레이션 등이 실패함. 재시도하여 완료로 전환 가능 |

#### "접수" 시 일어나는 일

관리자가 "접수"를 누르면 (대기 → 진행중):

## 2장. 구독/결제 관련 데이터

### 2.1 핵심 구조: Bill → Subscription → Payment

이 시스템에서 결제의 기본 구조는 Bill(청구서)이 가장 상위이고, Subscription(구독)이 Bill을 참조하는 형태입니다.

Bill (청구서) ← 먼저 생성됨, 독립적으로 존재 가능 │ ├─ Subscription (구독) ← Bill을 참조 (bill\_id FK) │ ├─ SubscriptionSection (구독 기간 구간) │ └─ SubscriptionMonthlySystemCycle (월간 사이클 연결) │ ├─ BillServiceItem (서비스 항목별 금액) │ └─ ServiceItem → Service + Plan (서비스-플랜 조합) │ ├─ BillDiscount (적용된 할인) │ └─ Payment (결제 기록) ← 1:1 관계 └─ CardPaymentHistory (카드 결제 상세)

왜 Bill이 상위인가? Bill(청구서)이 먼저 생성되고, 그 뒤에 Subscription이 해당 Bill을 참조하면서 생성됩니다. Bill은 구독 없이도 존재할 수 있지만(보조 청구서, 임의 청구서), Subscription은 반드시 Bill이 있어야 합니다.

### 2.8 월간 사이클(MonthlySystemCycle) — 주문 수와 플랜 한도

MonthlySystemCycle은 구독 기간을 월 단위로 쪼개서 관리하는 핵심 테이블입니다.

| 필드 | 설명 |
| --- | --- |
| shop\_id | 쇼핑몰 FK |
| start\_date  /  end\_date | 사이클 기간 |
| plan\_id | 이 사이클에 적용된 플랜 (FK → Plan) |
| order\_limit | 이 사이클의 주문 한도 (Plan의 order\_limit × rate) |
| order\_count | 현재까지 집계된 주문 수 |
| rate | 일할 비율 (기본 1.0) |
| main\_status | 사이클 상태 (대기, 진행중, 완료) |

현재 활성 사이클은

ShopSystemInfo.cur\_monthly\_system\_cycle\_id

로 확인.

#### 생성 시점

구독 시작 시: 첫 사이클 생성

매월 자동 갱신: 매일 00:00 배치에서 종료된 사이클을 완료하고 새 사이클 생성

#### order\_count 업데이트

매일 00:00, 10:00에 Order 테이블에서 해당 기간의 주문 수를 매번 처음부터 다시 집계 (Cafe24 주문만)

#### 플랜 한도 반영

Plan.order\_limit × rate → MonthlySystemCycle.order\_limit (한도) vs MonthlySystemCycle.order\_count (현재 주문 수)

105% 초과 시: plan\_limit\_option이 "안전"이면 알림만, "일반"이면 서비스 제한

#### Subscription과의 연결

하나의 구독에 여러 MonthlySystemCycle이 생깁니다. SubscriptionMonthlySystemCycle 연결 테이블로 관리.

## 3장. 이커머스 플랫폼에서 받아오는 데이터

현재 연동 가능한 이커머스 플랫폼은 Cafe24와 Imweb입니다. 이 장의 데이터는 모두 플랫폼에서 동기화되며, 각 테이블의

platform\_id

필드가 원본 ID입니다. 모든 테이블은

shop\_id

로 조인 가능합니다.

#### 이커머스 플랫폼 데이터의 삭제 처리 — 소프트 딜리트

이커머스 플랫폼에서 동기화되는 데이터는 소프트 딜리트(Soft Delete) 방식으로 관리됩니다. 실제로 데이터베이스에서 레코드를 삭제하지 않고,

is\_deleted

필드를

True

로 변경하여 "삭제됨" 상태를 표시합니다.

소프트 딜리트가 적용되는 테이블: Product, Customer, CustomerGroup, Category, ProductCategory, ECommercePlatformBenefit, ECommercePlatformCoupon, ECommercePlatformPointAutoExpiration 등

동작 방식: 이커머스 플랫폼에서 상품/고객/카테고리 등을 삭제하면, 코어 시스템의 동기화 과정에서 해당 레코드의

is\_deleted = True

로 갱신됩니다. 데이터 조회 시

is\_deleted = False

조건으로 필터링하면 현재 유효한 데이터만 볼 수 있고, 삭제된 데이터의 이력도 보존됩니다.

## 부록: 자주 쓰는 조회 시나리오 모음

#### "이 쇼핑몰의 현재 상태가 궁금해요"

Shop (main\_status, sub\_status 확인) + ShopDetail (상세 정보) + ShopStatistics (주문 통계)

#### "이 쇼핑몰의 현재 플랜이 뭔지 알고 싶어요"

빠른 확인: Shop → ShopSystemInfo (plan\_id로 Plan 조인) 상세 확인: Shop → Subscription (main\_status=201) → Bill → BillServiceItem → ServiceItem → Plan

#### "이 쇼핑몰이 어떤 서비스를 쓰고 있고, 얼마를 내고 있는지"

Shop → ShopSystemInfo.cur\_subscription\_id → Subscription → Bill → BillServiceItem → ServiceItem → Service + Plan

#### "이 쇼핑몰의 서비스 설치/제거 이력"

Shop → InstallRemoveRequest (type으로 설치/제거 구분, main\_status로 처리 상태) → Service (service\_id로 어떤 서비스인지 확인)

#### "이 쇼핑몰의 결제 실패 여부"

Shop → Subscription → Bill → Payment (main\_status: 200=성공, 400=실패, 402=미수)

#### "주문 수가 한도를 넘었는지"

Shop → ShopSystemInfo.cur\_monthly\_system\_cycle\_id → MonthlySystemCycle (order\_count vs order\_limit, plan\_limit\_option 확인)

#### "후불비용(미결제 금액)"

Shop → DefferedAmountHistory (사용/충전/소멸 이력)

#### "적용된 쿠폰"

Shop → ShopCoupon (main\_status=201이면 적용 중) → Coupon

#### "특정 상품의 판매 내역"

Product (sale\_total로 빠른 확인) 또는 Product → OrderItem (정밀 확인)

#### "특정 고객의 주문 이력"

Customer → Order → OrderItem + OrderReceiver

이 문서는 Core-System 코드베이스를 기반으로 작성되었습니다. (2026-04-07)
