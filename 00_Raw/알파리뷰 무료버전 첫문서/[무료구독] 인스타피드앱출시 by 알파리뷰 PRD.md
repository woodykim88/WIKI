* **쿼터 정책 명확화**: 막연한 "N건/월" → 명확한 **최초 30건 / 이후 월 5건**

* **FE 유/무료 판단 3축 정립**: A(쿼리파라미터) / B(온보딩 JSON) / C(sub_status.라이브유형)
* **SQS 큐 라우팅 규칙 명문화**: C 우선 → 없으면 B → 무료/유료 큐 분리

# 목적

* PRD v4 변경사항을 반영하여 FSD 구조를 User Story 기반으로 재정리한다.
* 각 기능을 **누가 / 왜 / 무엇을 원하는지** 관점으로 표현한다.
* 각 기능마다 검증 가능한 **Acceptance Criteria (인수조건)** 를 명시한다.
* PRD의 정책/제약/운영 가드레일까지 유지한 상태에서 디자인 / FE / BE / 코어가 바로 협업 가능한 형태로 정리한다.

# Phase1 제품 목표

* 무료구독 사용자가 `설치 → 무료구독 본 온보딩(카드등록 포함, 건너뛰기 가능) → 소셜 리뷰 수집/스태프 리뷰/재방문 배너 → CAFE24 역이관 → 통합 쿼터 도달 → 2차 온보딩 → 무료 체험(14일) → 유료 전환 검토`까지 한 사이클을 경험할 수 있어야 한다.
* 무료구독은 충분한 운영 경험을 제공하되, 알파리뷰 전용 위젯·푸시·업셀 등 심화 기능은 14일 체험/유료 진입 시점에만 노출하여 단계적 가치 제안을 만든다.
* 리뷰 노출은 알파리뷰 전용 리뷰 위젯이 아니라 **CAFE24 역이관(무제한)** 중심으로 해결한다. (단, 인스타그램 피드위젯은 온전히 제공)
* 알림톡 비용은 **온보딩에서 등록받은 카드로 후불 자동 정산(매월 1일)** 처리한다.

# 공통 원칙

* 기존 기능 최대 재사용, 구독/정책/제한 레이어에 집중
* 무료구독은 통합 쿼터 적용: **최초 사용 시 30건 / 이후 월 5건** (PRD v4 확정값)
* **CAFE24 역이관은 무제한 / 카운트 제외**로 무료구독에서도 제공
* 소셜 리뷰는 자동 역이관, 스태프 / 재방문 / 리뷰요청 작성분은 비동기 자동 역이관
* 대량 업로드 기능은 무료구독에서 사용 불가
* 구매 고객 대상 일반 리뷰 유도 알림톡은 Phase2 때 진행
* 알파리뷰 전용 리뷰 위젯은 Phase1 무료구독 본 온보딩/사용 구간 제외 (2차 온보딩에서 처음 등장)
* 포인트 자동 지급은 제외, 선불 포인트 10,000P 지급 구조도 폐지
* 제한은 프론트 비활성만이 아니라 BE 정책 검증까지 포함해야 한다.

# 핵심 정책 정리

## 무료구독 / 카드등록 / 체험 / 유료 정책

* 온보딩 중 카드 등록은 **선택 (건너뛰기 가능)** 이다.
* **카드 등록 ≠ 무료체험(14일) 진입**: 카드는 후불 정산용으로만 보관되고, 무료구독은 그대로 유지된다.
* 무료체험(14일) 진입 트리거는 **쿼터 한도 도달 또는 Premium 기능 클릭** 시점이다.
* 카드 미등록 사용자는 리뷰요청 **기능만 잠금**상태가 되며, 클릭 시 카드등록 모달이 노출된다.
* 카드 미등록 사용자도 소셜 리뷰 수집 / 스태프 리뷰 / 재방문 배너는 자유롭게 사용 가능하다.
* 무료구독의 통합 쿼터는 **최초 30건 / 이후 월 5건**이며, 코어 구독 월간 시스템(매월 1일) 주기에 맞춰 리셋된다.
* 유료 전환 시 쿼터 제한이 즉시 제거된다.
* **'유료 전환 검토 중(과금 전환 넛지)'은 별도 백엔드 상태값이 아니다.** 쿼터 한도 도달 또는 Premium 기능 클릭 시점에 사용자에게 과금 전환을 제안하는 모달이 노출되는 **트리거 시점**을 의미하며, 백엔드 구독 상태는 여전히 '무료 구독 사용 중'이다.
* **다운그레이드/해지는 사용자 클릭 즉시 적용되지 않는다.** CSM 컨택 + TP팀 검수/위젯 정리를 거쳐 최종 확정된다.
* **체험 중 다운그레이드 요청**은 14일 체험 종료 전에 발생할 수 있으며, 체험 구독은 끝까지 제공된 뒤 종료 시점에 무료구독으로 복구된다.

## 알림톡 후불 자동 정산 정책

* 리뷰요청 알림톡 발송 건당 **12원** (데이터11 검증 후 확정)
* **매월 1일 시스템 주기**에 직전 월 발송 건수 합계 × 단가를 등록 카드로 자동 청구
* 카드 미등록 사용자는 청구 대상에서 자동 제외 (리뷰요청 자체가 잠금이므로)
* 알림톡 발송 자체는 쿼터에서 차감되지 않으며, 카드등록 사용자에게만 활성. 월 발송 건수만큼 후불 정산된다.

# User Story + Acceptance Criteria

## 1\. 신규 앱 출시: 알파리뷰\_인스타피드

### User Story

* 초기 상점 운영자로서, 무료로 시작할 수 있는 별도 진입점(인스타그램 피드)을 통해 부담 없이 알파리뷰를 처음 써보고 싶다.
* 운영팀으로서, 사용자가 어떤 깡통 앱(설치 경로)을 통해 알파앱스 통합앱을 설치했는지 알고 싶다. 그래야 유입 경로별 온보딩과 성과를 구분할 수 있다.

### Acceptance Criteria

* 카페24 앱스토어에 사용자 시점에서는 '알파리뷰\_인스타피드' 앱이 노출된다.
* 사용자에게는 별도 앱처럼 보이지만, 실제로는 **알파앱스 통합앱이 설치되고 알파리뷰 무료구독이 활성화** 되는 구조다.
* 앱 소개(서브카피)에 "인스타그램 피드 및 소셜리뷰(네이버, 인스타) 수집을 무료로 이용하세요" 가치가 명확히 표현된다.
* 앱 설치 시 shop_id가 생성되고, 어떤 깡통 앱(설치 경로) 으로 설치되었는지 식별값이 저장된다.
* 설치 경로에 따라 이후 랜딩 또는 온보딩 진입 화면 분기가 가능하다.
* 설치만으로는 무료구독이 라이브되지 않는다. **무료구독 라이브는 온보딩 완료 시점에 시작**된다 (재설치 우회 방지).

## 2\. 무료구독 본 온보딩 플로우 ✅

### User Story

* 신규 사용자는 앱 설치 후 무엇을 해야 하는지 단계적으로 안내받고 싶다. 그래야 서비스 시작까지 이탈하지 않고 진행할 수 있다.
* 사용자는 무료구독의 핵심 가치(소셜 리뷰 수집)를 인스타 연결로 먼저 체감한 뒤, 카드 정보를 입력하고 싶다. 그래야 카드 입력에 대한 거부감이 줄어든다.
* 사용자는 카드 등록을 바로 하지 않더라도 무료구독을 먼저 경험하고 싶다. 그래야 초반 진입 장벽이 낮아진다.

### Acceptance Criteria

* 설치 후 사용자는 무료구독 본 온보딩 첫 단계로 진입할 수 있다.
* 온보딩 단계는 정확히 다음 순서로 제공된다: `기본정보 입력 → 계정정보 입력 → 인스타그램 계정 연결 → 카드정보 입력(건너뛰기 가능) → 알파리뷰 무료구독 소개 → 완료`
* 카드 등록 단계는 명시적으로 '건너뛰기' 가능하다.
* 인스타그램 계정 연결 단계도 건너뛸 수 있으며, 건너뛴 경우 인스타 소셜리뷰 가져오기 메뉴는 비활성 상태로 표시되고 사용 시점에 다시 연결 모달이 노출된다.
* '리뷰 운영상태(설문)', '서비스 소개', '위젯 설치', '푸시/업셀'은 무료구독 본 온보딩에서 **노출되지 않는다** (2차 온보딩에서 처음 등장).
* 온보딩 완료 후 무료구독 라이브 상태로 전환되며, 통합 쿼터 **최초 30건 / 이후 월 5건**이 활성화된다.
* 온보딩 중 현재 단계와 다음 행동이 사용자에게 명확히 표시된다.
* 온보딩 완료 여부, 카드 등록 여부, 인스타 연결 여부가 시스템에 각각 저장된다.
* 무료구독 / 무료체험 / 유료 진입 시 튜토리얼/안내 메시지가 구분되어야 한다.

## 3\. 카드 등록 정책 ✅

### User Story

* 사용자는 무료구독을 시작할 때는 카드 등록 부담 없이 진입하고 싶다. 하지만 알림톡 발송 비용을 자동 정산받기 위해, 또는 유료 전환이 필요한 순간에는 자연스럽게 결제 수단 등록으로 이어지고 싶다.
* 운영팀은 무료 진입 장벽은 낮추되, **알림톡 후불 정산용 카드를 선제적으로 확보**하고 향후 유료 전환 마찰을 줄이고 싶다.

### Acceptance Criteria

* 온보딩 중 카드 등록은 **선택 (건너뛰기 가능)** 으로 제공된다.
* 카드 등록 없이도 온보딩 완료 및 무료구독 시작이 가능하다.
* 카드 등록만으로 무료체험(14일)이 자동 시작되지 않는다 (분리 정책).
* **카드 미등록 사용자가** 리뷰요청 **메뉴를 클릭하면 카드등록 모달이 노출**되고, 등록 완료 후 즉시 발송 가능 상태로 전환된다.
* 카드 미등록 사용자도 소셜 리뷰 수집 / 스태프 리뷰 / 재방문 배너는 자유롭게 사용할 수 있다.
* 사용자가 쿼터 한도 도달 또는 Premium 기능 클릭으로 무료체험/유료 전환을 시도할 때, 카드가 미등록 상태이면 카드등록 단계가 자동 삽입된다.
* 카드 등록 완료 여부가 시스템에 저장된다.
* **온보딩 카드등록 이탈률**을 운영 지표로 추적할 수 있다
* 선불 포인트 10,000P 지급 구조는 v3에서 폐지되었으므로, 카드 등록 보상 형태로 노출되지 않는다.
* 알파리뷰에서 무료↔유료 구독 유형 전환이 발생할 때마다, 슈퍼관리자 시스템에 TP팀 작업용 티켓이 자동 생성된다. 티켓에는 상점 식별자, 현재 구독 유형, 전환 이전/이후 상태, 설치/제거 자동 시도 결과 등이 포함된다. 티켓 생성 시 슈퍼관리자 시스템의 기본 알림 채널을 통해 TP팀에 자동 알림이 발송된다.

## 4\. 구독 상태 흐름

### User Story

* 운영팀은 사용자의 현재 구독 상태를 일관되게 관리하고 싶다. 그래야 무료구독, 무료체험, 유료, 종료 상태별 정책을 정확히 적용할 수 있다.
* 사용자는 자신의 현재 상태에 맞는 경험만 노출받고 싶다. 그래야 혼란 없이 서비스를 이해할 수 있다.

### Acceptance Criteria

* 시스템은 다음 **백엔드 상태**를 구분할 수 있다: `방문자 / 앱 설치됨 / 온보딩 진행 중 / 온보딩 완료 / 무료구독 사용 중 / 쿼터 한도 도달 / 무료 체험 중(14일) / 유료 구독 중 / 무료 구독 / 구독 종료`
* **'유료 전환 넛지(과금 전환 검토)' 는 백엔드 상태값이 아닌 트리거 시점**이다. 쿼터 한도 도달 또는 Premium 기능 클릭 시 사용자에게 과금 전환을 제안하는 모달이 노출되는 순간을 의미하며, 구독 상태는 여전히 '무료 구독 사용 중'이다.
* **구독 상태와 온보딩/라이브 상태는 분리 관리**된다 (재설치 시 즉시 라이브 방지).
* 무료구독 사용 중 상태에서 **쿼터 한도(최초 30건 또는 이후 월 5건) 소진 또는 Premium 기능 클릭** 시 유료 전환 넛지 모달이 노출된다.
* 사용자가 넛지 모달에서 유료 전환을 선택하면, 카드 미등록 시 카드등록 단계가 자동 삽입된 후 **2차 온보딩 진입 → 무료 체험 중(14일)** 상태가 된다.
* 무료 체험 중 상태에서 (a) 자동 과금 성공 시 '유료 구독 중'으로 자연 전환된다.
* **무료 체험 중 다운그레이드 요청**: 14일 체험 종료 전에 사용자가 다운그레이드를 요청할 수 있으며, 체험 구독은 **끝까지 제공**된다. 체험 종료 시점에 CSM/TP팀이 설치 위젯·설정값을 정리하여 무료 구독으로 복구한다 (즉시 무료로 바뀌지 않음).
* '무료 구독' 상태에서도 등록된 카드는 **후불 정산용으로 계속 보관**된다.
* **유료 구독 중 → 무료 구독 다운그레이드는 즉시 적용되지 않는다**. 사용자의 클릭은 "요청" 단계이며, CSM팀이 컨택하여 쇼핑몰 상태를 검수하고 TP팀이 기설치된 위젯 제거 등 정리 액션을 완료한 후 최종 다운그레이드가 확정된다. 과금구독에서 사용했던 설정옵션값은 기억되지 않고 무료구독 쿼터제 시스템으로 활성화된다.
* **구독 해지 요청도 즉시 적용되지 않는다**. 사용자의 해지 클릭은 "요청"으로 처리되며, CSM팀 컨택 + TP팀 검수 및 정리 액션 완료 후 최종 해지가 확정된다.
* 앱 삭제 시 '구독 종료' 상태가 된다.
* 동일 Shop ID 재설치 시 기존 이력(쿼터 히스토리 포함)을 복원한 상태로 '무료 구독' 상태로 재시작한다 (즉시 라이브 X, 온보딩 완료 후 라이브).
* '무료구독 사용 중'(최초)과 '무료 구독'(다운그레이드/체험 종료 후) 상태는 운영상 구분 가능해야 한다.
* **유료→무료→유료→무료 어뷰징 시나리오**에 대해, 동일 월 내 재전환 시 1회차 무료의 잔여 쿼터를 이어 사용하도록 처리한다.
* 유료 구독 중 → 무료 구독 전환 시, 프론트엔드에서 위젯 자동제거(설치해지) 요청을 1차 수행한 후, 성공 시에만 구독 상태가 '무료 구독'으로 변경된다. 자동제거 실패 시 TP팀 티켓이 생성되며, 사람이 위젯 제거를 완료한 뒤 '무료 구독' 상태로 전환한다
* 무료 구독 → 유료 구독 전환 시, 프론트엔드에서 위젯 자동설치 요청을 1차 수행한 후, 성공 시에만 구독 상태가 '유료 구독 중'으로 변경된다. 자동설치 실패 시 TP팀 티켓이 생성되며, 사람이 위젯 설치를 완료한 뒤 '유료 구독 중'으로 전환한다.

## 5\. 소셜 리뷰 수집 및 CAFE24 역이관

### User Story

* 사용자는 인스타그램이나 네이버 블로그의 리뷰를 쉽게 가져오고 싶다. 그래야 기존에 흩어진 후기를 빠르게 상품 리뷰 자산으로 활용할 수 있다.
* 사용자는 가져온 리뷰가 CAFE24 상품 상세에 보이길 원한다. 그래야 별도 위젯 없이도 리뷰 노출 효과를 얻을 수 있다.

### Acceptance Criteria

* 사용자는 인스타그램 또는 네이버 블로그 계정을 연결할 수 있다.
* 인스타그램 연결은 온보딩 3단계에서 사전 진행 가능하며, 미연결 시 메뉴 진입 시점에 재연결 모달이 노출된다.
* 사용자는 연결된 소셜 채널에서 리뷰를 수집할 수 있다.
* **리뷰 수집 1건 성공 시 통합 쿼터가 1건 차감**된다 (가져오기 실패 시 차감되지 않음).
* 수집된 소셜 리뷰는 **CAFE24 리뷰 게시판으로 자동 역이관**된다.
* **CAFE24 역이관 자체는 통합 쿼터에서 카운트되지 않으며, 무료구독에서도 무제한 제공**된다.
* 역이관 성공/실패 상태를 시스템이 기록한다.
* 역이관 실패 건은 **재시도 큐**를 통해 자동 재시도되며, 운영자는 수동 재이관 UI를 통해 다시 시도할 수 있다.
* 쿼터가 모두 소진된 경우 추가 수집이 차단된다 (BE 정책 검증).
* CAFE24 역이관은 비동기 처리 기준이며, 큐 적체 / 실패율 / 처리 지연이 시스템 안정성 기준으로 모니터링된다 (월 한도 차단이 아님).

## 8\. 리뷰요청 (카드등록 필수) ✅

### User Story

* 사용자는 서포터즈나 체험단 같은 비구매자에게 리뷰를 요청하고 싶다. 그래야 실제 구매 이력이 없어도 초기 리뷰를 확보할 수 있다.
* 운영팀은 알림톡 발송 비용을 사용자가 즉시 결제하지 않고도, 등록된 카드로 후불 자동 정산하고 싶다.

### Acceptance Criteria

* 리뷰요청 **기능은 카드 등록 사용자에게만 활성화**된다.
* 카드 미등록 사용자가 리뷰요청 메뉴를 클릭하면 카드등록 모달이 노출되고, 등록 완료 후 즉시 발송 가능 상태가 된다.
* 사용자는 비구매자를 대상으로 리뷰요청 알림톡을 발송할 수 있다.
* **알림톡 발송 자체는 통합 쿼터에서 차감되지 않 는다** (리뷰 등록 시점에만 차감).
* 알림톡 발송 비용(건당 12원)은 등록된 카드로 **매월 1일 후불 자동 정산**된다.
* 동일 수신자에게 무료구독 기준 **최초 1회만** 발송 가능하다.
* 구매 고객 대상 일반 리뷰 유도 알림톡은 Phase1에서 제공되지 않는다.
* 리뷰요청 경유로 작성된 리뷰가 등록되는 시점에 통합 쿼터가 1건 차감되지 않으며, CAFE24 자동 역이관 대상이 된다.
* 발송 건수 / 비용 산식은 사용자에게 사전 표시된다.

## 9\. 알림톡 후불 자동 정산

### User Story

* 사용자는 발송 시점마다 결제 마찰을 겪지 않고, 매월 한 번 등록 카드로 자동 정산받고 싶다.
* 운영팀은 결제 실패 / 카드 만료 / 클레임 (첫 청구일 "왜 돈이 빠졌냐" CS) 을 사전 차단하고 싶다.

### Acceptance Criteria

* **매월 1일 시스템 주기**에 직전 월의 알림톡 발송 건수 합계 × 단가(12원/건)를 등록 카드로 자동 청구한다.
* 코어 구독 정산 주기와 동일하게 정렬되어 사용자에게 '월 1회 결제일'이라는 단일 기대치를 형성한다.
* 카드 미등록 사용자는 리뷰요청 자체가 잠금이므로, 청구 대상에서 자동 제외된다.
* **결제 실패 시 1차 재시도 → 재시도 큐 등록 (주기 별도 정의)**
* 최종 실패 확정 시 리뷰요청 기능을 비활성화하고 사용자에게 알림을 전송한다.
* 카드 갱신 안내 모달이 노출되고, 카드 갱신 후 즉시 리뷰요청이 재활성화된다.
* **첫 청구일 전 사전 알림** (예정 청구액 + 산식)이 사용자에게 발송된다 (클레임 사전 차단).
* 청구일 당일 영수증 / 내역 메일이 발송된다.
* 후불 자동 정산 성공률을 운영 지표로 추적할 수 있다 (95% 이하 시 카드 갱신 플로우 점검).
* FAQ에 후불 정산 구조가 사전 게시된다.
* 통합 쿼터(N건/월) 내에서만 발송 가능하므로, **월 청구 상한은 자연스럽게** `N × 12원`으로 캡 된다.

## 11\. 통합 횟수제한 시스템 ✅

### User Story

* 운영팀은 무료구독 사용량을 하나의 쿼터 풀로 통제하고 싶다. 그래야 비용을 관리하면서도 무료 경험을 제공할 수 있다.
* 사용자는 현재 얼마나 사용했고, 얼마가 남았는지 알고 싶다. 그래야 제한 전에 적절히 행동할 수 있다.
* 운영팀은 어뷰징 시나리오 (유료↔무료 반복, 동시 다발성 발송, 재설치 등) 를 차단하고 싶다.

### Acceptance Criteria

* 통합 쿼터는 무료구독 기준 **최초 사용 시 30건 / 이후 월 5건** 이다 (PRD v3 확정값, 데이터5로 추후 재조정 가능).
* **쿼터 차감 대상은 리뷰 등록 시점**: 스태프 리뷰 등록, 소셜 리뷰 (인스타/네이버) 수집 성공, 재방문 배너 경유 리뷰 등록, 리뷰요청 경유 리뷰 등록.
* **알림톡 발송 자체는 쿼터 차감 대상이 아니다** (후불 정산으로 별도 처리).
* **CAFE24 역이관도 쿼터 차감 대상이 아니다** (무제한 제공).
* 가져오기 실패 시 차감되지 않는다.
* 월간 리셋 기준은 **코어 구독 월간 시스템(1일) 주기**를 따른다.
* 월 중 설치 시 그 달의 N건이 전량 사용 가능하다 (일할 계산이 아니라 월 단위 풀 제공).
* **Shop ID 단위로 쿼터 이력이 영구 보존**된다 (탈퇴/재설치 시에도 유지).
* **쿼터 한도 도달 시점(최초 30건 또는 이후 월 5건 모두 소진)에 과금화 넛지 모달이 노출된다** (백엔드 상태값은 여전히 '무료 구독 사용 중').
* 쿼터 한도 도달 후 차감 대상 기능(스태프/소셜/재방문/리뷰요청경유 등록)이 **API 레벨에서 차단**된다.
* 잔여 쿼터를 사용자에게 표시한다 (예: "이번 달 남은 횟수 X건").
* 100% 초과 후 사용자가 유료 구독 선택 시 **이미 등록된 카드로 즉시 결제 진행**된다 (카드 미등록이면 등록 후 결제).
* **다중 액션 동시 실행 시 BE에서 현재 사용량 + 요청 건수 기준으로 검증**하여 한도 초과 요청을 거절한다.
* 무료→유료 업그레이드 시 쿼터 제한이 즉시 제거된다.
* 유료→무료 다운그레이드 시 다음 월 리셋 시점부터 무료 쿼터가 적용된다.
* 체험→유료 자동 전환 시, 체험 중 사용분은 별도 관리되고 유료 전환 시 카운터가 비활성화된다.
* **어뷰징 방지 정책 (v4)**:
  * 동일 월 내 유료→무료→유료→무료 반복 시, 2회차 무료에서는 1회차 무료의 잔여 쿼터를 이어 사용 (풀카운트 재제공 X).
  * 1회차 무료 기간 중 쿼터 미사용 시 → 2회차에도 잔여량 유지.
  * 한 달 이상 경과 후 재전환 시 → 풀카운트 제공.
  * Shop ID 평생 이력 기준이므로, 재설치만으로는 쿼터 리셋 불가.

### 작업영역

## 14\. 데이터 마이그레이션 및 보존 정책

### User Story

* 사용자는 앱을 삭제하거나 다시 설치해도 핵심 데이터와 이력이 일관되게 유지되길 원한다. 그래야 운영 이력이 사라지지 않고 정책 우회도 방지된다.
* 운영팀은 어떤 데이터가 언제, 어디로 이관되는지 명확히 관리하고 싶다.
* 사용자는 무료구독 → 무료체험 전환 시 1-2일 대기 없이 즉시 리뷰 기능을 쓰고 싶다.

### Acceptance Criteria

* 소셜 리뷰는 수집 시 자동 역이관된다 (CAFE24 무제한 / 카운트 제외).
* 스태프 리뷰, 재방문 배너 리뷰, 리뷰요청 작성분도 자동 역이관 대상이며, **무제한 / 카운트 제외**다.
* 역이관은 비동기 처리이며, 실패 시 재시도 큐 + 운영 모니터링이 동작한다.
* 앱 삭제 후에도 **리뷰 데이터는 영구 보존**된다
* **Shop ID 기준 쿼터 히스토리는 영구 보존**된다 (재설치로 제한 우회 불가).
* 동일 Shop ID 재설치 시 기존 이력이 복원된다.
* **재설치 시 즉시 라이브되지 않으며**, 온보딩 완료 시점에 무료구독이 라이브된다.
* 재설치 시 쿼터가 초기화되지 않는다.
* **무료구독 → 무료체험 전환 시 상품/주문/리뷰 데이터 이관이 필요**하다.
* 무료구독 사용 중에도 **리뷰 데이터는 백그라운드에서 미리 이관**되어, 체험 진입 시 1-2일 대기 없이 즉시 14일치 주문 기반 리뷰요청 기능이 활성화된다.
* 코어 서비스에서 **무료/유료별로 별도 파이프라인**을 통해 데이터를 전달한다 (트래픽 분리).
* 미디어 파일의 30일 하드 삭제 정책은 별도 운영정책으로 정의되어야 한다.

### 작업영역

* 디자인
  * 정책 안내 문구 정리 (보존 정책 / 재설치 / 미디어 삭제)
* FE
  * 상태 및 가이드 노출
* BE
  * 보존 정책 적용
  * 재설치 복원 로직
  * 이력 유지 (Shop ID 영구)
  * 자동 역이관 큐 (무제한)
* 코어
  * 무료/유료 별도 파이프라인 구축 (최초 이관 + 웹훅)
  * 무료구독 사용 중 리뷰 데이터 백그라운드 이관
  * 무료구독 → 무료체험 전환 시 상품/주문/리뷰 이관 트리거

## 15\. 14일 체험 / 유료 전환 시 2차 온보딩

### User Story

* 무료구독 사용자가 무료체험/유료 전환을 선택하는 순간, 무료에서 비노출되었던 알파리뷰 전용 위젯·푸시·업셀 채널을 처음으로 만나고 싶다. 그래야 단계적 가치 제안이 명확해진다.
* 운영팀은 "무료에서는 CAFE24 역이관으로 충분하고, 체험/유료부터 알파리뷰 전용 위젯과 부가 채널이 열린다"는 메시지를 명확히 전달하고 싶다.

### Acceptance Criteria

* 사용자가 '유료 전환 검토 중' 상태에서 유료 전환을 선택하면, 다음 순서로 2차 온보딩이 진행된다: `서비스 소개 → 위젯 설치 → 푸시/업셀 연계 유도화면 → 완료`
* 카드 미등록 상태에서 2차 온보딩 진입 시, **카드등록 단계가 자동 삽입**된다.
* 카드 등록 사용자는 카드 정보 확인 단계만 거친다 (재입력 X).
* 2차 온보딩 완료 후 **무료 체험(14일)** 이 시작되며, 14일간 전 기능이 개방된다.
* 2차 온보딩 진입 시 사용자에게 "체험/유료 전환에 따른 추가 단계"임을 명확히 안내한다.
* 위젯 설치 단계는 알파리뷰 전용 위젯 (게시판/포토리뷰/평점) 설치를 포함한다.
* 푸시/업셀 단계는 알파푸시 / 알파업셀 등 부가 채널 연계 유도화면을 포함한다.
* 2차 온보딩 진입 사유 (체험/유료 전환) 가 CS 대응 카피에 포함된다.

### 작업영역

  * 2차 온보딩 3단계 (서비스 소개 / 위젯 설치 / 푸시·업셀) UX 설계
  * 진입 사유 카피 / 가치 제안 카피 작성

## 16\. FE 유/무료 판단 기준 (A/B/C) 및 SQS 큐 라우팅 ✨ 신규

### User Story

* FE는 사용자가 어떤 화면을 보고 있는지, 어떤 온보딩을 거쳤는지, 현재 어떤 구독 상태인지에 따라 다른 UI를 보여줘야 한다. 이를 위해 **세 가지 독립된 값**을 일관되게 활용해야 한다.
* 코어팀은 마이그레이션(SQS) 시점에 유/무료에 맞는 큐를 분기해야 한다. 그래야 무료 상점이 폭증해도 유료 고객 처리에 영향을 주지 않는다.

### Acceptance Criteria

* FE는 다음 세 가지 값을 기준으로 유/무료 분기를 처리한다.

| 기호 | 의미 | 데이터 소스 | 비고 |
| -- | -- | -- | -- |
| **A** | 현재 접속한 대시보드가 유/무료인지 | **쿼리파라미터** | 진입 경로별 랜딩/안내 분기에 사용 |
| **B** | 현재 진행하는 또는 마지막 진행한 온보딩이 유/무료인지 | **온보딩 진행 시 core에 저장하는 JSON 필드값** | 온보딩 이탈 복구·잘못된 진입경로 안내에 사용 |
| **C** | 현재 알파리뷰 상태가 유/무료인지 | **sub_status.라이브유형** | 라이브 구독 기능/제한 분기에 사용 |

* **SQS 큐 라우팅 규칙**: 마이그레이션 시점에 **C 값을 우선** 판단하고, C가 없으면 **B 값을 참조**하여 유/무료 큐를 결정해 생성한다.
* B·C가 모두 없는 경우(설치 직후·온보딩 시작 전)에는 큐 결정을 보류하고, 온보딩 시작 시점에 B 값이 채워지면 그때 큐를 결정한다.
* 코어 서비스의 무료/유료 파이프라인(SQS 큐 분리)은 이 규칙에 따라 라우팅된다.

### 분기 시나리오 예시

| A (진입) | B (온보딩) | C (라이브) | 결과 | 큐 라우팅 |
| -- | -- | -- | -- | -- |
| 무료앱 | *  | *  | 신규 무료 온보딩 시작 | (B 채워지면) 무료 큐 |
| 무료앱 | *  | 유료 | "이미 알파리뷰 사용 중" 안내 | 유료 큐 (C 우선) |
| 무료앱 | 무료 (이탈) | *  | 무료 온보딩 이어가기 | 무료 큐 (B 참조) |
| 무료앱 | 유료 (이탈) | *  | 안내 화면 (CSM 컨택) | 유료 큐 (B 참조) |
| 유료앱 | *  | *  | 신규 유료 온보딩 시작 | (B 채워지면) 유료 큐 |
| 유료앱 | 무료 (이탈) | *  | "무료앱 설정 삭제 동의" 확인 | 이전 무료 큐 → 유료 큐로 전환 |
| 유료앱 | *  | 무료 | 무료→유료 업그레이드 (2차 온보딩) | 유료 큐 (C 우선) |
| 유료앱 | *  | 유료 | "이미 유료구독 사용 중" 안내 | 유료 큐 (C 우선) |

### 작업영역

* FE
  * 페이지 진입 시 A·B·C 값 로딩 및 분기 로직 구현
  * 컨텍스트별 안내 화면/모달 분기 처리
  * 디버깅을 위해 현재 A·B·C 상태를 운영 모드에서 확인할 수 있게 노출
* BE
  * core JSON 필드(B) 스키마 정의 및 온보딩 진행 시점 저장
  * sub_status.라이브유형(C) 필드 정의 및 상태 전이 시 갱신
  * B·C 조회 API 명세
* 코어
  * **SQS 큐 분리** (무료 큐 / 유료 큐) 구성
  * 마이그레이션 시점에 C 우선, 없으면 B 참조하여 큐 결정 라우팅 구현
  * 큐별 컨슈머 분리 및 모니터링 (적체/실패율)

## 17\. 최초설치 진입 시나리오 (5가지 분기) ✨ 신규

### User Story

* 사용자가 카페24 앱스토어에서 \*\*인스타피드앱(무료) 또는 알파리뷰앱(유료)\*\*을 클릭했을 때, 그 사용자의 기존 구독/온보딩 이력에 따라 보여야 하는 화면이 달라져야 한다.

### Acceptance Criteria

* 설치 클릭 시점에 **A(진입경로) + B(온보딩 이력) + C(라이브 구독)** 컨텍스트를 조합해 다음 5가지 시나리오 중 하나로 분기한다.

#### 시나리오 1 — 신규 (구독유형 없음)

* **진입경로**: 인스타피드앱(무료) 클릭
* **컨텍스트**: shop_id 없음 (B=없음, C=없음)
* **동작**: 알파리뷰 통합앱 설치 → 무료앱 온보딩 첫 화면 진입
* **분기**: 정보 입력 완료 → 튜토리얼(입력값 체크) → 숏컷 대시보드 / 정보 입력 미완료(건너뛰기) → 튜토리얼(입력 요청) → 숏컷 대시보드. 숏컷에서 기능 클릭 시 "정보 입력필요" 안내.

#### 시나리오 2 — 기존 과금구독 사용 중

* **진입경로**: 인스타피드앱(무료) 클릭
* **컨텍스트**: C = 유료(과금구독 라이브)
* **동작**: "이미 알파리뷰를 사용 중입니다" 안내 팝업 → 별도 온보딩 불필요, 유료구독 상태 유지

#### 시나리오 3 — 과거 알파리뷰(유료) 온보딩 중 이탈

* **진입경로**: 인스타피드앱(무료) 클릭
* **컨텍스트**: B = 유료(이탈), C = 없음
* **동작**: 알파리뷰 온보딩 중단 이력 안내 화면 노출 (CSM 컨택 가이드)

#### 시나리오 4 — 과거 인스타피드앱(무료) 온보딩 중 이탈

* **진입경로**: 알파리뷰앱(유료) 클릭
* **컨텍스트**: B = 무료(이탈), C = 없음
* **동작**: "인스타피드앱 이전 설정은 삭제되며 전체 알파리뷰 기능을 사용하시겠습니까?" 확인 안내 → 동의 시 유료 온보딩 시작

#### 시나리오 5 — 무료구독 사용 중 (인스타피드 사용 중)

* **진입경로**: 알파리뷰앱(유료) 클릭
* **컨텍스트**: C = 무료(라이브)
* **동작**: 무료→유료 업그레이드 플로우 진입 (AS-IS 알파리뷰앱 2차 온보딩: 서비스 소개 → 위젯 설치 → 푸시/업셀)

### 작업영역

* 디자인
  * 5가지 시나리오별 안내 화면/팝업/확인 모달 UX 설계
  * 시나리오 3·4의 안내 카피 및 CSM 컨택 가이드 카피
* FE
  * 설치 후 진입 시점에 A·B·C 조회 → 5가지 시나리오 분기 라우팅
  * 시나리오별 화면 컴포넌트 구현
  * [REVW-352](https://linear.app/salad-lab/issue/REVW-352/%EB%AC%B4%EB%A3%8C%EA%B5%AC%EB%8F%85-%EC%9C%A0%EB%AC%B4%EB%A3%8C-app-%EC%A7%84%EC%9E%85%EC%8B%9C-gateway-%EC%B2%98%EB%A6%AC-%EB%B0%8F-%EA%B8%B0%EC%A1%B4-review-auth-guard-%EC%B2%98%EB%A6%AC) 와 연계
* BE
  * 진입 시점 컨텍스트 조회 API (shop_id + B + C 반환)
  * 시나리오 식별 로직 (입력 컨텍스트 → 시나리오 키 반환)
* 코어
  * shop_id 생성/조회 + 온보딩 이탈 이력 보존

## 기타2. CS / 운영 준비물 (v4 반영)

### User Story

* 무료구독 사용자는 막히는 지점에서 바로 가이드를 받고 싶다. 그래야 사람 응대 없이도 스스로 문제를 해결할 수 있다.
* 운영팀은 반복 문의를 자동응답과 가이드로 흡수하고 싶다. 그래야 유료 고객 CS 품질을 유지할 수 있다.
* 운영팀은 첫 후불 청구일 클레임 ("왜 돈이 빠졌냐") 을 사전 차단하고 싶다.

### Acceptance Criteria

* 다음 시점별 CS 대응 스크립트가 준비된다:
  * 앱 설치 직후 ("뭘 어떻게 시작하나요?")
  * **온보딩 카드등록 단계** ("왜 무료인데 카드를 받나요?", "건너뛰면 어떻게 되나요?")
  * **인스타그램 계정 연결** ("인스타 연결이 안 돼요")
  * 무료구독 소개 화면 ("무료구독에서 뭘 할 수 있나요?", "위젯은 왜 없나요?")
  * CAFE24 역이관 이해 ("리뷰가 상품 상세에 어떻게 보이나요?")
  * 쿼터 한도 도달 ("갑자기 기능이 안 돼요")
  * **14일 체험/유료 전환 시 2차 온보딩** ("갑자기 위젯 설치 화면이 떠요", "푸시 연계는 뭔가요?")
  * **첫 후불 청구일** ("왜 돈이 빠졌나요?", "환불해주세요")
* 카드 등록 사유(후불 정산) + 건너뛰기 시 영향(리뷰요청 잠금) 가이드가 준비된다.
* CAFE24 역이관 구조를 설명하는 가이드가 준비된다.
* 쿼터 한도 도달 시 안내 문구가 준비된다.
* **첫 후불 청구일 전 사전 안내 (예정액 + 산식)** 메일/알림이 발송된다.
* 청구일 당일 영수증/내역 메일이 발송된다.
* FAQ에 후불 정산 구조가 사전 게시된다.
