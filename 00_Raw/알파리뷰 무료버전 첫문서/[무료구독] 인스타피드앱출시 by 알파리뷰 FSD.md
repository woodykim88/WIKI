## 🔴 기존 PRD 이슈 목록

### 이슈 1: Solution 4번 — 리뷰 요청 채널 불일치

**기존 PRD:**
> "주문 내역에서 최근 30건에 대해 이메일/문자 기반의 리뷰 요청을 할 수 있게 한다. 알림톡은 Phase1에선 막는다."

**문제:**
- 참고문서 Phase 1.5 범위에 **알림톡 기반 리뷰 요청**이 2순위로 명시됨 (카드 등록 절차 필요)
- "이메일/문자 기반"이라는 표현은 참고문서 어디에도 없음. 이메일 요청은 **Phase 2부터** 필요한 기능으로 명시
- 알림톡을 막으면 Phase 1.5 핵심 기능인 "리뷰 요청" 자체가 무력화됨

**수정 방향:**
- 리뷰 요청 채널을 **알림톡** 기반으로 변경
- 카드 등록 플로우를 선행 조건으로 추가
- 이메일/문자는 Phase 2로 이동

### 이슈 2: Solution 6~7번 — 위젯 노출이 Phase 1 범위 초과

**기존 PRD:**

**문제:**
- 참고문서에서 위젯(게시판/포토리뷰 위젯)은 **Phase 2: 게시/관리 기능 제공** 범위
- Phase 1.5는 "위젯 없이도 리뷰 제거/우선순위 조정 가능하도록 도움"이 목표
- 위젯 자동 설치, 디자인 커스터마이징 등은 Phase 2 영역

**수정 방향:**
- Phase 1에서 위젯 노출 관련 항목(6, 7번) 제거
- 인스타 피드만 예외적으로 제공 (별도 앱 형태, 참고문서 Phase 1 범위)
- 위젯은 Phase 2 No-Gos로 이동

### 2. Appetite — 수정

### 3. Solution — 수정

**수정된 핵심 아이디어:**

> 인스타·블로그·스태프 리뷰를 한 곳에 모으고, 알림톡 기반 리뷰 요청과 기본 관리 기능을 제공하는 '리뷰 수집·관리 입문 패키지'다. 위젯 노출 없이도 리뷰가 쌓이고 정리되는 경험을 제공하며, 통합 횟수제한과 카드등록 플로우를 통해 유료 전환의 토대를 만든다.

**수정된 Breadboarding:**

> ① 앱 설치 → ② 온보딩(무료버전) → ③ 인스타 피드 연결 → ④ 소셜/스태프 리뷰 수집 → ⑤ (카드등록) → ⑥ 알림톡 리뷰 요청 → ⑦ 리뷰 관리(고정/필터) → ⑧ 유료 넛지

**수정된 Solution 항목:**

**① 앱 설치 & 온보딩**
- 카페24 앱스토어에서 **"인스타 피드 & 소셜리뷰"** 앱으로 신규 출시 (무료 진입점 역할)
- 무료버전 전용 온보딩 플로우: 기본정보(도메인, 카테고리, 규모) → 리뷰 운영 상태 확인 → 소셜 계정 연결 유도
- 체험/과금/무료 상태 전이 플로우 설계 필요 (`추가 개발`)
  - 체험 → 무료버전, 과금 → 무료버전, 무료버전 → 체험, 무료버전 → 과금 전이 정의

**② 인스타 피드 (1순위)**
- 인스타그램 계정 연결 → 리뷰성 게시물 선택 → 상품 연결
- 상품 상세에 인스타 피드 형태 노출 (그리드/슬라이드 기본 1종)
- 이 기능만으로도 "미끼" 가치 제공 가능 (별도 앱으로도 출시 가능했던 기능)

**③ 소셜 리뷰 수집 — 인스타 & 네이버 블로그 가져오기 (1순위)**
- 인스타그램/네이버 블로그 URL → 대표 이미지/제목/요약 자동 추출 → 상품에 연결
- 통합 횟수제한 카운트에 포함

**④ 스태프 리뷰 (1순위)**
- 운영자/스태프가 직접 작성하는 리뷰 입력 기능
- 제한: 통합 횟수제한 내에서 사용 (50건/월 공유)
- 초기 리뷰 확보용 ("깔아두기")으로 주로 사용될 것으로 예상

**⑤ 리뷰 요청 — 알림톡 기반 (2순위)**
- **선행 조건: 카드 등록 필수**
  - 카드 등록 시 알림톡 발송 기능 활성화
  - 건당 100원 할증 (유료 시 12원), 최초 포인트 3,000원 제공
- 최근 주문 내역 기반 리뷰 요청 알림톡 발송
- 채널 설정: 첫 번째 리뷰 요청만 사용 가능 (회원가입 유도, 리마인더 불가)
- 작성 유형: 텍스트/포토리뷰만 허용, 나머지 비활성화
- 통합 횟수제한 카운트에 포함 (리뷰 요청 발송 = 1회, 리뷰 작성 = 1회 → 합산 2회 카운트)

**⑥ 리뷰 관리 (Phase 1.5 범위)**
- 리뷰 리스트 기본 관리:
  - 게시/비게시 처리
  - 기본 필터링 (별점, 키워드)
  - 리뷰 위치 선정 (상/하단 고정) 가능
  - 리뷰 연결 1개
- **제한:**
  - 자동 댓글 불가
  - 리뷰 게시 필터링은 별점+키워드만 (리워드 조건 모두 불가)
  - 포인트 자동 지급 불가
  - 포인트 지급 설정: 리뷰 작성 조건 기본값 고정, 별점/텍스트/포토만 수정 가능

**⑦ 재방문 배너 1종 (3순위)**
- 주문 데이터 연동 필요
- 리뷰 작성 유도를 위한 재방문 배너 기본 1종 제공
- 리뷰 작성 경로 트래킹에 재방문배너 경로 포함

**⑧ 유료화 넛지**
- 관리 화면 곳곳에 "Premium 기능" 표시 → 클릭 시 유료 업셀 모달
- **잠긴 기능 사용불가 정책:** 옵션은 바꿔볼 수 있되, 저장 시 "유료화 유도 팝업" 노출
  - 접근 자체를 차단하거나 비활성화하지 않음
  - 체험은 가능하되 적용은 유료 전용
- 한계치 도달 전 단계별 알림 (70% → 90% → 100% 초과 시 기능 축소)

### 3-A. 무료버전 추가 개발 기능 (플랫폼 운영 레이어)

> **본 Phase 1은 무료버전 고객 경험 구현뿐 아니라, 무료 정책 운영을 위한 내부 기능 개발을 함께 포함한다.**
> 무료버전은 단순히 기존 기능을 일부 잠그는 수준이 아니라, **"무료 사용자 식별 → 사용량 측정 → 제한 적용 → 상태 전이 → 유료화 유도 → 비용 통제"**까지 포함하는 별도 운영 레이어가 필요하다.
> 따라서 고객이 보는 기능(①~⑧) 외에도, 무료 정책을 가능하게 만드는 내부 기능을 정식 스펙으로 명시한다.

---

**⑨ 통합 횟수제한 시스템** `신규 개발` `1~2순위 병행`

무료버전의 핵심 제약은 기능별 개별 제한보다, **"리뷰 확보를 위해 사용한 총 액션 수"를 하나의 공통 풀(Quota Pool)로 관리**하는 방향이다.

- **Quota 차감 대상 액션:**

  | 액션 | 차감 수 | 비고 |
  |------|---------|------|
  | 알림톡 리뷰 요청 발송 | 1 | 카드 등록 상점만 |
  | 리뷰 작성 (요청에 의한) | 1 | 요청 1 + 작성 1 = 합산 2카운트 |
  | 스태프 리뷰 작성 | 1 | |
  | 소셜미디어 가져오기 (인스타) | 1 | 건당 |
  | 소셜미디어 가져오기 (네이버 블로그) | 1 | 건당 |
  | (Phase 3 이후) 알림톡 유도 | 1 | 현재 범위 포함 X, 설계 시 고려 |

- **한도:** 50건/월 + 카드 등록 필수
- **카운터 라이프사이클:**
  - 월간 카운터: 매월 1일 00:00 리셋 (매월 1일 기준 집계 or 배치 리셋)
  - 상태 전이 시: 무료→유료 전환 시 카운터 비활성화 (무제한), 유료→무료 다운그레이드 시 해당 월 잔여분부터 적용
  - **Shop ID 단위 평생 이력:** `mall_id + multi_mall_no` 기준으로 탈퇴 후에도 한도 히스토리 영구 보존 → 재설치로 제한 우회 불가
- **한도 도달 시 동작:**
  - 70% (35건): 안내 배너 "이번 달 한도 70% 소진"
  - 90% (45건): 경고 알림 "10건 남음" + 업그레이드 넛지 모달
  - 100% 초과: 리뷰 요청·스태프리뷰·가져오기 기능 비활성화 → "이번 달 놓친 리뷰 요청 N건" 손실 프레임 노출 + 다음 달 대기 or 유료 전환
- **기술 요구사항:**
  - **카운트 테이블:** `quota_usage (shop_id, year_month, action_type, count, updated_at)`
  - **이력 테이블:** `quota_history (shop_id, mall_id, multi_mall_no, free_started_at, total_used, is_active)`
  - 한도 체크 미들웨어: 각 기능 API 호출 시 `quota_usage` 확인 → 초과 시 차단 응답 + 넛지 페이로드 반환
  - 월간 리셋: 배치 or `year_month` 기준 집계 (리셋 배치 없이 월 기준 WHERE 절로 처리 가능)
- **고민 포인트:**
  - 통합 50건이 적정한지 → 데이터5로 검증 (라이트 플랜 70% 수준 가설)
  - 기능별 개별 서브캡을 추가로 둘지 vs 통합만으로 충분한지 (예: 스태프리뷰만 50건 몰아쓰기 방지)
  - 리뷰 작성 카운트 포함 시 "요청했는데 고객이 써서 내 한도가 줄었다"는 UX 불만 가능 → 사전 안내 문구 필요
  - 월 주문건 100건 이상 시 기능 다운그레이드 정책과 통합 한도의 관계 정리 필요

**⑩ 리뷰 작성 경로 트래킹** `신규 개발` `1순위부터 점진 적용`

통합 Quota의 정확한 카운트, 그리고 **"어떤 경로에서 리뷰가 가장 쉽게 확보되는지", "어떤 경로가 유료 전환과 연결되는지"** 판단을 위한 필수 데이터 기반.

무료버전에서는 "리뷰가 몇 개 생겼는가"보다 **"어떤 경로에서 생겼는가"**가 중요하다.

- **트래킹 대상 경로 (Phase 1):**

  | source_type (enum) | 설명 | 현재 구분 가능 여부 |
  |--------------------|------|-------------------|
  | `revisit_banner` | 재방문 배너 경유 작성 | ❌ 추가 개발 필요 |
  | `review_request` | 알림톡 리뷰 요청 경유 | △ request_id 연결 필요 |
  | `staff` | 스태프 리뷰 직접 작성 | ✅ 이미 구분 가능 추정 |
  | `social_instagram` | 인스타그램 가져오기 | ✅ 이미 구분 가능 추정 |
  | `social_naver_blog` | 네이버 블로그 가져오기 | ✅ 이미 구분 가능 추정 |
  | (Phase 2+) `qr_code`, `reminder`, `widget_button`, `first_request_alimtalk` | 확장 경로 | 설계 시 enum 예약 |

- **데이터 모델 — 리뷰 엔티티 또는 이벤트 로그에 추가할 필드:**
  - `review_source_type` (enum): 위 테이블의 source_type
  - `request_id` (nullable FK): 리뷰 요청 발송 ID → 요청과 작성을 1:1 연결
  - `campaign_id` (nullable): 재방문 배너 종류, 캠페인 구분
  - `import_source` (nullable): 소셜 가져오기 시 원본 URL/ID
  - `staff_written_yn` (boolean): 스태프 작성 여부 (기존 필드 활용 가능 시 생략)

- **고민 포인트:**
  - 기존 리뷰 테이블에 컬럼 추가 vs 별도 `review_source_log` 테이블 분리 → 기존 테이블 스키마 영향도 확인 선행
  - 현재 경로 구분이 안 되는 리뷰(재방문배너)는 반드시 추가 개발 필요
  - Phase 2 확장을 고려한 enum 설계 (확장 가능한 구조)
  - 이 필드가 있어야 데이터3(리뷰 작성률), 데이터5(기능별 사용량), Quota 카운트가 정확하게 동작

**⑪ 무료버전 플로우 (상태 머신)** `신규 개발` `1순위 필수`

무료버전은 기존 체험/과금 플로우에 "얹는" 것이 아니라, **별도의 온보딩과 상태 전이 규칙이 필요한 독립 흐름**이다.

- **A. 상태 머신 설계**

  기존 `체험 ↔ 과금` 2상태에서 무료가 추가되면서 **상태 머신 수준의 설계**가 필요하다.

  ```
  [visitor] → [installed] → [free_onboarding] → [free_active]
                                                      ↓
                                               [quota_reached] → [paywall_exposed] → [trial] or [paid]
                                                      ↓
                                               (다음 달 리셋) → [free_active]
  ```

  **정규 전이:**

  | From | To | 트리거 | 동작 |
  |------|----|--------|------|
  | visitor | installed | 무료 앱 설치 | shop_id에 free_plan 상태 부여 |
  | installed | free_onboarding | 온보딩 시작 | 무료 전용 온보딩 플로우 진입 |
  | free_onboarding | free_active | 온보딩 완료 | Quota 카운터 활성화, 기능 개방 |
  | free_active | quota_reached | 월 50건 소진 | 기능 비활성화 + 업그레이드 넛지 |
  | quota_reached | free_active | 월 리셋 (매월 1일) | 카운터 초기화, 기능 재개방 |
  | free_active/quota_reached | trial | 유료 체험 시작 | 데이터 유지, Quota 해제 |
  | free_active/quota_reached | paid | 결제 완료 | 즉시 모든 기능 해제, 한도 제거 |

  **예외 전이 (⚠️ 핵심 비즈니스 판단 필요):**

  | From | To | 트리거 | 고민 포인트 |
  |------|----|--------|-----------|
  | trial | free_active | 체험 기간 만료 | **가장 위험한 전이.** 허용하면 체험→무료→방치 경로 생김. 막을지 vs 허용하되 넛지 극대화할지 결정 필요 |
  | paid | free_active | 결제 실패 확정 / 자발적 다운그레이드 | 기존 리뷰 데이터 유지, 유료 기능(위젯 등) 비활성화. 월 주문건 100건 이상이면 기능 다운그레이드 |
  | free_active | uninstalled | 앱 삭제 | 리뷰 데이터 보존 정책, Shop ID 이력 영구 보존 |
  | uninstalled | free_active | 재설치 (동일 Shop ID) | 이전 Quota 히스토리 복원, 무료 한도 리셋 X |

- **B. 온보딩 플로우 (무료 전용)**
  - 카페24 앱스토어 → 앱 설치 → 무료 전용 온보딩 시작
  - 온보딩 스텝: 기본정보(도메인, 카테고리, 규모) → 리뷰 운영 상태 확인 → 소셜 계정 연결 유도 → 완료
  - 기존 유료 온보딩과 설치 시점에 플랜 판별하여 분기
  - **고민:** 기존 온보딩 컴포넌트 재사용 범위 vs 완전 새 플로우 필요 범위

- **C. 알림톡 / 카드 등록 플로우** `2순위`
  - 리뷰 요청 기능 사용을 위한 카드 등록 → 알림톡 활성화 과정
  - **설계 원칙:** Phase 3(작성 기능 전면 개방)에서도 **동일 플로우를 재사용**할 수 있게 설계
  - 플로우: 리뷰 요청 메뉴 진입 → "카드를 등록하면 알림톡으로 리뷰를 요청할 수 있어요" 안내 → 카드 등록 → 최초 포인트 3,000원 충전 → 알림톡 발송 가능 상태
  - **카드 등록 위치:** 셋업 체크리스트의 **마지막 단계**에 배치 (앞 단계에서 가치 체감 후 등록 유도, 첫 단계에 두면 이탈률 급등)
  - **고민:** 카드 등록 = PG 연동인지, 단순 결제수단 등록인지 → 기존 결제 인프라와 연동 방식 확인 필요

**⑫ 신규 앱 출시** `1순위 병행`

무료버전이 메인 유료 제품의 축소판인지, 별도 진입 상품인지에 따라 앱스토어 포지셔닝과 온보딩 구조가 달라진다. **"신규 앱 출시 여부" 자체가 제품 스펙**이다.

- **옵션 A: 별도 앱으로 등록**
  - 앱명 (안): "인스타 피드 & 소셜리뷰" (무료 미끼 기능을 전면에 내세움)
  - 장점: 무료 진입 허들 낮음, 앱스토어 노출 면적 확대, "무료" 키워드 마케팅 가능
  - 단점: 동일 shop에 두 앱 설치 시 충돌 방지 로직 필요, 유료 전환 시 앱 전환 마이그레이션 필요
  - 필요 스펙: 앱명, 스토어 설명, 설치 후 원앱 통합 방식, 삭제/업그레이드 시 데이터 연속성

- **옵션 B: 기존 알파리뷰 앱 내 무료 플랜 추가**
  - 장점: 앱 전환 불필요, 업셀 동선 자연스러움, 데이터 연속성 자동 보장
  - 단점: "무료" 키워드 앱스토어 노출 효과 약함, 기존 유료 고객과의 UI 분기 복잡

- → **이 의사결정은 PRD 확정 전 논의 필수** (제품 구조를 근본적으로 바꾸는 결정)

- **공통 기술 요구사항:**
  - 앱 설치 시 shop_id에 "무료 플랜" 상태 자동 부여
  - 기존 알파리뷰 백엔드와 동일 인프라 사용 (별도 서버 X)
  - Shop ID(Mall_id + 멀티쇼핑몰 번호)별 **평생 적용**
  - 앱 심사/등록 리드타임: 카페24 앱 심사 통상 1~2주 → 1순위 개발과 병행하여 심사 제출

**⑬ 데이터 마이그레이션 정책** `정책 정의 (2~3순위)`

PRD에 **"무엇을 옮기고, 무엇은 안 옮길지"**를 명확히 정의해야 한다. 무료→유료 전환, 앱 교체, 스토어 이전 시 데이터 정합성 문제를 줄이기 위함.

- **원칙:**
  - 무료버전 설치 시 기존 카페24 리뷰 데이터 이관 **하지 않음** (유료와 차별화)
  - 무료버전에서 불필요한 데이터는 마이그레이션 대상에서 **명시적으로 제외**

- **데이터 유형별 마이그레이션 매트릭스:**

  | 데이터 유형 | 무료 설치 시 (← 카페24) | 무료→유료 전환 시 | 앱 삭제 시 | 재설치 시 |
  |------------|----------------------|-----------------|-----------|----------|
  | 카페24 기존 리뷰 | ❌ 이관 안 함 | ✅ 이관 시작 (유료 기능) | N/A | ❌ |
  | 스태프 리뷰 | N/A (무료에서 생성) | ✅ 그대로 유지 | ⚠️ 보존 정책 필요 | 복원 |
  | 재방문배너 경유 리뷰 | N/A (무료에서 생성) | ✅ 그대로 유지 | ⚠️ 보존 정책 필요 | 복원 |
  | 소셜 리뷰 메타데이터 | N/A (무료에서 생성) | ✅ 그대로 유지 | ⚠️ 보존 정책 필요 | 복원 |
  | 위젯 배치 정보 | N/A (Phase 2) | Phase 2에서 생성 | 삭제 | N/A |
  | 포인트 관련 데이터 | ❌ 차단됨 | Phase 3에서 활성화 | N/A | N/A |

- **무료→카페24 역이관:**
  - 재방문배너·스태프리뷰로 작성된 리뷰 → 카페24에도 반영되어야 의미 있음
  - 기존 이관 파이프라인 활용 가능한지 확인 필요

- **앱 삭제(탈퇴) 시 정책:**
  - 리뷰 데이터 보존 기간: 즉시 삭제? 30일 보관? → 정책 정의 필요
  - Shop ID 기준 Quota 히스토리: **영구 보존** (재설치 시 리셋 방지)

**⑭ 기타 제한 정책** `정책 정의 (전 순위 공통)`

무료버전의 기능 제한은 단순 "비활성화"가 아니라, **유료화 넛지와 연결된 4단계 레벨**로 설계한다.

- **제한 레벨 프레임워크:**

  | 레벨 | 방식 | UX | 유료화 넛지 |
  |------|------|-----|-----------|
  | L1. 완전 비노출 | 메뉴/UI 자체가 보이지 않음 | 존재 자체를 모름 | 없음 (인지 불가) |
  | L2. 노출 + 비활성 | 메뉴는 보이나 회색 처리, 클릭 불가 | "Premium" 뱃지 표시 | 클릭 시 기능 설명 + 업그레이드 안내 |
  | L3. 수정 가능 + 저장 차단 | 옵션을 만져보고 미리보기 가능, 저장 시 차단 | 체험은 가능 | **저장 시 유료화 유도 팝업** (가장 강력) |
  | L4. 제한 수량까지 허용 | 기능 사용 가능하나 한도 존재 | 한도 소진 알림 | 한도 도달 시 업그레이드 넛지 |

- **기능별 제한 레벨 매핑:**

  | 영역 | 무료 허용 범위 | 제한 레벨 | 차단 범위 |
  |------|---------------|----------|----------|
  | 알림톡 채널 | 첫 번째 리뷰 요청만 | L2 (나머지 비활성) | 회원가입 유도, 리마인더 |
  | 리뷰 유형 | 텍스트, 포토 | L2 | 동영상, 프리미엄 리뷰 |
  | 리뷰 게시 필터 | 별점, 키워드 | L2 | 리워드 조건 전체 |
  | 포인트 지급 설정 | 별점/텍스트/포토 금액만 수정 | **L3** (수정 가능, 저장 차단) | 작성 조건 변경, 자동 지급 |
  | 디자인 설정 - 공통 | 기본 색상만 | **L3** | 상세 컬러, 폰트, 레이아웃 |
  | 디자인 설정 - 상세보기 | - | L2 (모두 비활성) | 전체 |
  | 디자인 설정 - 작성화면 | 별점/포인트 컬러만 | **L3** | 나머지 |
  | 리뷰 작성 유도하기 | - | L2 (모두 비활성) | 전체 |
  | 자동 댓글 | - | L2 | 전면 차단 |
  | 리뷰 연결 | 1개 | L4 (1개 한도) | 2개 이상 |
  | 위젯 | 인스타 피드만 | L1 (나머지 비노출) | 포토리뷰/평점/게시판 (Phase 2) |
  | 포토리뷰 위젯 가로배열 | 1 고정 | L2 | 변경 불가 |
  | 통합 횟수 | 50건/월 | L4 | 초과 시 기능 비활성화 |

- **서비스 퀄리티 제한 (트래픽)**
  - 일정 수준 이상 API 호출 시 rate limit 적용

  - **고민:** rate limit 기준 지표 (Phase 1에서는 위젯 없으므로 API 호출 기준)

### 3-B. 리스크 관리 `Phase 1 필수`

> Phase 1 수준이라도 무료버전은 인프라·메시지·운영 비용과 유료 잠식 리스크를 발생시킨다.
> 이 프로젝트는 "출시"보다 **"제한과 롤백을 전제로 한 운영 실험"**이다.

#### 비용 리스크

**비용 가드레일 (Phase 1 범위):**

| 비용 항목 | 발생 조건 | 예상 규모 | 산출 데이터 | 가드레일 |
|-----------|----------|----------|------------|---------|
| **알림톡 발송** | 카드 등록 상점의 리뷰 요청 | 건당 100원 할증 (유료 12원 대비 88원 추가) | 데이터11 | 무료 상점당 월 메시지 원가 상한 설정 |
| **인프라 (서버/트래픽)** | 무료 상점 API 호출, 인스타피드 렌더링 | 상점당 월 비용 역산 필요 | 데이터10 | 무료 상점당 월 인프라 원가 상한 설정 |
| **CS 운영** | 무료 상점 문의 대응 | 설치 직후 CS 집중 발생 예상 | 데이터12 | 무료 CS 우선순위 하향 + 자동응답 |
| **카드등록 포인트** | 최초 3,000원 제공 | 카드 등록 상점수 × 3,000원 | 카드 등록 전환율 | 월 총 포인트 지급 상한 설정 |
| **데이터 스토리지** | 소셜 리뷰 이미지, 스태프 리뷰 | 상점당 저장량 역산 | 데이터10 파생 | 무료 상점 저장 한도 |

> **비용 손익 판단 등식:** `무료 상점 1개당 월 운영비(인프라+알림톡+CS) < 유료 전환 시 LTV × 전환율 기대값`
> 이 등식 검증을 위한 최소 데이터가 데이터10~12이며, Phase 1 착수 전 산출 완료가 이상적.
> **가드레일 초과 시:** 해당 구간 상점에 rate limit 적용 or Quota 축소 or 기능 품질 제한

**비용 리스크 시나리오:**

| 시나리오 | 트리거 | 영향 | 대응 |
|---------|--------|------|------|
| **A. 무료 상점 폭증** | 앱스토어 노출 → 예상 3배 설치 | 서버 비용↑, CS 폭주 | rate limit + 월 주문 100건↑ 다운그레이드 + 극단적 사용량 제한 |
| **B. 카드 등록률 극저** | 카드 허들 → 1순위만 사용 | 유료 전환 파이프라인 미작동, 프리라이더 양산 | 체크리스트 UX 최적화 + 카드등록 시 추가 당근 + 필요시 카드 없이 제한적 알림톡 허용 |

#### 유료 잠식 리스크

무료의 목표는 유료이며, 전체 가입자 수가 아니라 **유료 전환과 직결된 "진짜 지표"**에 집중해야 한다.

**Kill / Rollback Rule — 성패 판단은 이 지표로:**

| 지표 유형 | 지표 | 기준 |
|----------|------|------|
| ⚠️ 위험 신호 (Kill Switch) | 6주 내 체험수 감소 | **10% 이상** → 롤백 검토 |
| ⚠️ 위험 신호 (Kill Switch) | 6주 내 매출 감소율 | **10% 이상** → 롤백 검토 |
| ✅ 진짜 성공 지표 | 무료→체험 전이율 | 주간 추적 |
| ✅ 진짜 성공 지표 | 무료→과금 전환율 | 주간 추적 |
| ✅ 진짜 성공 지표 | 유료 플랜 가격 페이지 조회수 | 무료 사용자 기준 |
| ✅ 진짜 성공 지표 | 유료 기능 클릭수 (잠긴 기능) | 무료 사용자 기준 |
| 📊 보조 지표 | 과금→무료 다운그레이드 | 주간 10건 초과 시 경고 |
| 📊 보조 지표 | 체험→무료 전이 | 주간 비율 모니터링 |
| 📊 보조 지표 | 온보딩 완료율 | < 50% 시 UX 점검 |
| 📊 보조 지표 | 통합 Quota 소진율 | < 20% → 과다, > 90% → 과소 |
| ❌ 허영 지표 (참고만) | 전체 무료 가입자 수 | 의사결정 기준으로 사용 금지 |

> **핵심:** "사용자는 늘었는데 왜 실패냐" 같은 해석 충돌을 방지하기 위해, Kill/Rollback 기준을 PRD 단계에서 합의해야 한다.

**유료 잠식 시나리오:**

| 시나리오 | 트리거 | 영향 | 대응 |
|---------|--------|------|------|
| **C. 과금 상점 다운그레이드** | "무료로도 충분" 판단 | 매출 감소 → Kill Rule 도달 | 주간 모니터링 + 유료 전용 기능 리마인더 + 기능 범위 축소 |
| **D. 체험→무료 흡수** | 체험 만료 후 과금 대신 무료 선택 | 유료 전환 파이프라인 약화 | 체험→무료 전이 차단 or 강한 넛지 (⑪ 상태 머신 참조) |

#### 운영 및 CS 리스크

무료버전이 설치 실패·설정 혼란·제거 요청 증가를 유발할 수 있다.

**CS 관점 요구사항:**

| CS 발생 시점 | 예상 주요 문의 | Phase 1 대응 |
|-------------|--------------|-------------|
| 앱 설치 직후 | "뭘 어떻게 시작하나요?" | 무료 전용 온보딩 가이드 자동 안내 |
| 소셜 계정 연결 시 | "인스타 연결이 안 돼요" | 채널톡 지식 사전 추가 |
| 카드 등록 시 | "왜 카드를 등록해야 하나요?" | 안내 문구 + FAQ |
| Quota 한도 도달 | "갑자기 기능이 안 돼요" | 한도 단계별 사전 알림 (70%/90%/100%) |
| 잠긴 기능 접근 시 | "이 기능은 왜 안 되나요?" | L3 정책: 체험 가능 + 저장 시 팝업 안내 |

- 채널톡 무료버전 FAQ 가이드 문서 사전 작성 → 자동 응답 우선 제공
- 무료 상점 CS 우선순위: 유료 고객 하위 (유료 CS 품질 보호)
- **기능 QA뿐 아니라 운영 QA도 Definition of Done의 일부**

**롤백 정책:**

1. **즉시 롤백:** 시장 여론 등 정성적 판단으로 부정적 특이사항 시
2. **데이터 기반 롤백:** Kill Rule 기준 도달 시
3. **단계적 제한 강화 (롤백 전 우선 시도):**
   - 절대적 사용량 제한 도입 (전체 리뷰수 N건 / 주문건 N건)
   - 극단적 기능 제한: 포토리뷰 제한, 알림톡 전종류 제한
4. **평가 주기:** 매 2주 제한 수준 검토 → 보고. 6주 1차 평가. 3개월 정식 평가.

| **스태프 리뷰** | 운영자/스태프 직접 작성. Quota 차감. |
| **리뷰 요청 (알림톡)** | 카드등록 선행 → 첫 번째 리뷰요청만 가능 → 텍스트/포토만 → Quota 차감. 건당 100원 할증. |
| **리뷰 관리** | 게시/비게시, 별점/키워드 필터, 상하단 고정, 리뷰연결 1개. 자동댓글·리워드·포인트 자동지급 불가. |
| **재방문 배너** | 주문 데이터 연동 기반 1종. 리뷰 작성 유도. |
| **유료화 넛지** | L2/L3/L4 제한 레벨별 넛지. 한계치 단계별 알림. 잠긴 기능 체험→저장 시 팝업. |

**B. 플랫폼 운영 레이어 (⑨~⑭)**

| 항목 | 내용 |
|------|------|
| **통합 횟수제한** | 공통 Quota Pool 50건/월. 카운트 대상: 알림톡+요청+스태프+소셜+작성. Shop ID 평생 이력. `신규 개발` |
| **리뷰 작성 경로 트래킹** | `review_source_type`, `request_id`, `campaign_id`, `import_source` 필드. `신규 개발` |
| **무료버전 플로우** | 상태 머신: visitor→installed→free_onboarding→free_active→quota_reached→paywall. 예외 전이 포함. `신규 개발` |
| **신규 앱 출시** | 별도 앱 vs 기존 앱 내 플랜 추가 — 의사결정 필요. 앱 심사 1~2주. |
| **데이터 마이그레이션** | 이관 매트릭스 정의. 카페24 역이관 정책. 삭제 시 보존 정책. |

### 5. 리스크 대응을 위한 FAQ — 보강

**기존 Q1~Q3 유지 + 아래 추가:**

**Q4. 카드 등록 허들이 높아 리뷰 요청 기능을 아예 못 쓰는 상점이 많으면?**

A4. 카드 등록 없이도 1순위 기능(인스타피드, 스태프리뷰, 소셜미디어 가져오기)은 사용 가능하므로 기본 가치는 제공된다. 카드 등록은 셋업 체크리스트의 마지막 단계에 배치하여 앞 단계에서 가치 체감 후 등록하도록 유도한다. (참고문서: "첫 번째 단계에 카드 등록을 두면 이탈률 급등")

**Q5. 통합 50건/월 한도가 초기 상점에게 부족하거나 과다하면?**

A5. 데이터5(기능별 월평균 사용수, 중간값) 확인 후 조정 가능하도록 정책 플래그를 둔다. 라이트 플랜 수준의 70%를 기준으로 설정했으며, 매 2주마다 제한 수준 상향/하향 검토.

**Q6. 무료 플랜 도입 후 기존 체험/과금 상점이 무료로 다운그레이드하면?**

A6. Kill Rule로 관리. "과금→무료", "체험→무료" 전이율을 주간 추적. 6주 내 체험수 감소 10% 이상 또는 매출 감소 10% 이상 시 단계적 제한 강화 → 롤백 순서로 대응. 핵심은 전체 가입자가 아니라 유료 전환율과 매출 변화가 판단 기준이라는 점.

**Q7. CS 비용이 예상보다 크게 발생하면?**

A7. 데이터12(CS 발생 규모·시기·사유)를 Phase 1부터 태깅하여 추적. 설치 직후·카드등록·한도 도달 시점별 CS 스크립트 사전 준비. 무료 CS 우선순위를 유료 하위로 설정하고, 자동 응답·가이드 문서 링크 우선 제공. 무료 상점 CS 비용이 상점당 운영비의 주요 부분을 차지하면, FAQ 자동화 강화 or 무료 범위 축소로 대응.

**Q8. 무료 상점에서 생성된 리뷰 데이터의 소유권/이관은 어떻게 되나?**

A8. 마이그레이션 매트릭스(⑬ 참조)에 따라 정의. 소셜 리뷰는 알파리뷰 내부에만 존재, 스태프리뷰·재방문배너 리뷰는 카페24 역이관 필요. 무료→유료 전환 시 모든 데이터 유지. 앱 삭제 시 보존 기간 정책을 별도 정의 (Quota 이력은 영구 보존).

### 6. No-Gos — 수정

1. **위젯 노출 (Phase 2로 이관)**
   - 포토리뷰 위젯, 평점 위젯, 게시판 리뷰 위젯은 Phase 2 범위. Phase 1에서는 인스타 피드만 제공.
2. **상세 디자인 커스터마이징**
   - 폰트, 상세 컬러, 레이아웃 조합 등 고급 디자인은 Phase 2 이후.
3. **포인트 자동 지급 / 리워드 정교화**
   - 조건별 차등, 이벤트 보너스, 쿠폰 연동 등은 Phase 1 범위에서 제외. 포인트 설정 UI는 L3(체험 가능, 저장 차단).
4. **이메일/문자 기반 리뷰 요청 (Phase 2로 이관)**
   - Phase 1은 알림톡 기반만 제공. 이메일 요청은 Phase 2부터.
5. **프라이싱/패키지 전면 개편**
   - 행동 데이터 수집만 해 두고 리디자인은 Phase 2 이후.
6. **경쟁사 분석 기반 차등화**
   - 리뷰에이드 등 경쟁사 분석은 Phase 2부터 리서치.

### 7. Definition of Done — 수정

**기능 완료 기준:**

1. **1순위 완료 기준 (2주차):**
   - 무료 플랜으로 앱 설치 → 온보딩 완료 → 인스타 피드 연결 또는 소셜/스태프 리뷰 1건 이상 수집 완료한 상점이 최소 **20개** 이상.
2. **2순위 완료 기준 (4주차):**
   - 카드 등록 후 알림톡 리뷰 요청을 최소 1회 발송한 상점이 **10개** 이상.
   - 통합 Quota 시스템이 정상 동작 (50건/월 한도 적용, 카운트 정합성 확인).
3. **3순위 완료 기준 (6주차):**
   - 재방문 배너를 통한 리뷰 작성이 1건 이상 발생한 상점 존재.
   - 리뷰 작성 경로 트래킹에서 경로별 구분이 정상 동작.

**플랫폼 운영 레이어 완료 기준:**

4. **데이터 수집 인프라:**
   - 이벤트 스키마(공통 키 + 3계층 이벤트) 구현 완료.
   - 데이터1~9 사전 분석 완료 → Quota 한도, 제한값 최종 확정.
   - 데이터10~12 산출 완료 → 무료 상점당 월 비용 추정치 도출.
5. **모니터링 대시보드:**
   - 주간 운영 대시보드 구축 완료: 설치수, 온보딩 완료율, 전이율, Quota 소진율, CS 건수.
   - Kill Rule 지표(체험수 감소율, 매출 감소율) 자동 모니터링.
6. **운영 준비:**
   - 채널톡 무료버전 FAQ 지식 추가 완료.
   - CS 시점별(설치직후/카드등록/한도도달) 대응 스크립트 준비 완료.
   - 롤백 플래그 구현 완료 (무료 기능 전체 또는 부분 비활성화 가능).

> **기능 QA뿐 아니라 운영 QA도 Done의 일부이다.**

### 8. 알파리뷰 생애주기 Dependency

> 참고문서에서 Phase 1.5 기획 필요로 명시

무료버전 도입에 따라 아래 생애주기 각 단계에서 분기 처리가 필요하다:

| 생애주기 단계 | 무료버전 영향 | Phase 1 대응 | 상태 머신 매핑 |
|--------------|-------------|-------------|--------------|
| 앱 설치 | 무료 앱으로 별도 진입점 (신규 앱 출시) | ✅ 필수 | visitor → installed |
| 온보딩 | 무료 전용 온보딩 플로우 | ✅ 필수 | installed → free_onboarding |
| 서비스 개시 | 무료 플랜 상태로 개시 | ✅ 필수 | free_onboarding → free_active |
| Quota 한도 도달 | 기능 비활성화 + 넛지 | ✅ 필수 | free_active → quota_reached |
| 결제/카드등록 | 카드 등록 플로우 (알림톡 활성화) | ✅ 2순위 | free_active → (card_registered) |
| 유료 전환 | 체험 시작 또는 직접 과금 | ✅ 핵심 성공 경로 | free_active → trial / paid |
| 설치 요청 | 위젯 없으므로 Phase 1에서는 해당 없음 | ❌ Phase 2 | - |
| 앱 삭제 | 무료 상점 탈퇴 시 데이터 처리 정책 | ✅ 정책 정의 | free_active → uninstalled |
| 재설치 | Shop ID 이력 복원, Quota 리셋 X | ✅ 정책 정의 | uninstalled → free_active |
| 상담/문의 | 채널톡 지식 추가, 가이드 업데이트 | ✅ 운영 준비 | (전 단계) |
| 이관 도입 | 리뷰 이관 없음. 스태프/재방문 리뷰만 카페24 이관 | ⚠️ 정책 정의 | - |
