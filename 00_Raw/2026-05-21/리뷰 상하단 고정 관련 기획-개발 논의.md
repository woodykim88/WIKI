### 1\. 논의 배경

리뷰 자동 정렬 기능과 상단고정 자동화를 구현하는 방향에 대한 기술 스펙 및 데이터 처리 방식을 논의

### 2\. 주요 논의 내용

#### 2-1. 큐레이션 이벤트(EPL) 액션 정의

VIEW

: 보기만 하고 아무것도 안 하는 케이스(View만 발생, 액션 없음): 신호로 기록, 개선 방향 추적 필요

Accept

: 상단 추천을 수락(적용)하는 액션

Like

:추천된 리뷰는 좋지만, 최상단 고정을 안하는 건너뛰기(Skip)

Deny

: 이 리뷰는 완전히 추천하지마 = 완전 거부(Deny)

#### 2-2. 상단고정 자동화 범위 및 정책

상단고정(상단 배치)만 자동화하기로 결정 — 하단 배치 자동화는 이번 스펙 제외 (미결)

상단고정 자동화는 기존 테이블 재활용, 업체 노하우 상단고정 로직 사용

상단 고정 자동화 시, 기존에 정립되어있던 상단 고정 위에 리뷰를 올린다는 개념으로 접근

기존 솔트값을 삭제하는 게 아닌 맨 위로 올리는 개념

99위가 최고점이면 100위으로 설정하여 최상단 배치하는 방식

그러나, 솔트값은 실제로 바뀌게 됨

상단고정 자동화 기능은 유료 판매 고려 가능성 언급 (이번 스펙 외)

#### 2-3. 배치 처리 주기

상단고정 추천/분석: 실시간 처리 가능

상단고정 자동화(=이하 ‘자동정렬’) 배치 주기: 7일(주간) 배치로 결론

관리자가 주 단위로 일괄 처리하는 패턴이 많아 7일로 결정

1시간도 허용 가능하다는 의견 있었으나 7일로 정리

상단고정 자동화는 오늘부터 배치 필요

#### 2-4. 크론(Cron) 구조 논의

기존 리뷰 트래킹 크론(15분 주기)과의 통합 vs 별도 크론 생성 논의

문제 시나리오: 7일 배치로 상단 고정된 리뷰가 리워드 문제 등으로 게시대기 상태로 전환될 경우 업체 혼란 우려

상단 고정 상태인데 실제로 게시되지 않는 상황 방지 필요

결론: 기존 크론에 통합하면 로직이 복잡해질 수 있어, 별도 크론으로 분리하는 방향 검토

기존 크론에서는 더블체크 용도로만 활용

담당자가 코드 확인 후 최종 결정 예정

#### 2-5. API 설계

큐레이션 정보 업데이트 API: 구현 완료

추천 목록 조회 API (상단 고정 후보 가져오기): 미구현 — 태성님이 만들거나 담당자가 직접 구현

#### 2-6. 하단 배치 기준 (유니스코어 하한) 재검토

1점대 리뷰가 하단으로 갈 만큼 나쁜 품질인지 의문 제기

예시: '좋아요 좋아요' 반복 5건 — 성의 없는 리뷰이나 현재 기준으로 하단 배치

0.8점짜리 리뷰도 존재

정밀도 검증을 위해 점수 데이터 추가 분석 필요성 언급

최종 점수 기준(상단 몇 점 이상, 하단 몇 점 이하)은 담당자가 데이터 보면서 결정 예정

### 3\. 결정 사항

자동화 범위: 상단고정 자동화만 진행 (하단 배치 자동화는 이번 제외)

배치 주기: 7일(주간) 배치로 결정

크론 구조: 별도 분리 방향으로 검토, 코드 확인 후 최종 결정

추천 목록 조회 API: 태성님 또는 현희님이 구현

유니스코어 상/하단 기준은 데이터 분석 후 별도 확정
