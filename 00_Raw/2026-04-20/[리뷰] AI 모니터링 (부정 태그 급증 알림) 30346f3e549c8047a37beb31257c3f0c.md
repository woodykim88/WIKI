### **1-2. 그래서 발생한 문제 (Synthesis)**

- **24시간 쏟아지는 고객 리뷰**
    - 고객 리뷰는 24시간 쉬지 않고 생성되지만, 담당자의 리소스는 업무 시간 내로 한정. 이로 인해 필연적으로 **감시 공백(Blind Spots)이 발생**
- **휴먼 에러 발생**
    - 반복적인 수동 모니터링은 중요 신호를 놓치거나 잘못 판단하는 **실수를 유발**
- **사후 대응의 위험성**
    - 이슈가 발생한 뒤에야 인지하는 '사후 대응' 방식은 브랜드 이미지 손상을 막을 수 없음. 예방이나 즉시 대응이 불가능한 현재 구조는 **리스크 관리 실패로 연결**

## **3. Appetite (베팅 규모)**

### **Batch Size**

- [ ]  Big Batch (3주)
- [x]  Small Batch (1주)

### **Constraints (제약 사항)**

- **실시간성**
    - 1분 단위의 실시간 감지는 서버 부하를 초래할 수 있으므로, **1시간에 한번씩 지난 24시간 리뷰를 모아서 처리**
- **알림 피로도 제어**
    - 같은 문제에 대한 알림이 폭주하지 않도록 동일 알림에는 **12시간 동안 알림이 가지 않음**
    - 상품과 태그와 사유(부정리뷰 급증/부정태그 급증/위험단어 감지)가 일치하면 동일 알림 처리

## **4. Solution**

### **4-1. 핵심 솔루션**

<aside>
✅

특정 태그의 부정 리뷰가 급증할 경우, 사장님에게 즉시 **알림톡/이메일**을 발송하여 조기 대응 유도

</aside>

- 솔루션 목적 : 유저들이 알파리뷰를 신뢰하고 맡길 수 있는 모니터링 경험 제공

### **4-2. 솔루션 스케치**

**`리뷰 수집` → `위험도 판단` → `알림 발송` → `대시보드 이동` → `상세 페이지`**

| **단계** | **상세 정의** |
| --- | --- |
| ① **리뷰 수집** | **매 정각에 지난 24시간 동안**의 리뷰 수집
  • 리뷰는 이미 긍부정/태그 분류가 완료된 상태 |
| **② 위험도 판단** | **A. 쇼핑몰 전체 부정 태그 급증
B. 쇼핑몰 전체 부정 리뷰 급증
C. 특정 상품 부정 태그 급증
D. 특정 상품 부정 리뷰 급증**
**E. 특정 상품 위험 감지** |
| **③ 알림 발송** | 카카오톡/슬랙 알림톡 및 이메일 발송 |
| **④ 대시보드 이동** | 확인하지 않은 모니터링 알림이 있을 경우, 리뷰목록 최상단에 위험 사항 알림 배너를 표시. 
  • 알림 배너에 버튼 클릭 시, [모니터링 알림] 페이지로 이동 |
| ⑤ 상세 페이지 | 확인하고 싶은 알림 클릭할 경우, 각 알림에 해당하는 상세 화면으로 이동 |

### 4-3. 상세 솔루션 로직

<aside>

### **① 리뷰 수집**

- 매 정각 (ex. 14:00, 15:00...) 이전 24시간의 리뷰 수집 및 위험도 판단 실행
- 작성된 시간이 현재시간~ 24시간 전인 리뷰 수집
- 모든 리뷰는 수집 시점에 이미 긍부정 / 키워드 분석이 완료됨 (현재 리뷰 태그 자체는 실시간으로 분석)
</aside>

<aside>

    C. 특정 상품 부정 태그 급증

- **용어 정리**
    - 쇼핑몰
        - 쇼핑몰에 모든 상품에 대한 분석
    - 상품
        - 한가지 대상 상품
    - 1차 태그
        - 대분류 태그
        - ex) 재질
    - 2차 태그
        - 소분류 태그
        - ex) 재질 → 내구성 일 경우 ‘내구성’
    - 특정 부정 태그
        - 2차 태그 중 부정적으로 판단된 태그
        - 하나의 리뷰당 해당 2차태그는 1개까지 적용
            - 예시) 배송늦음"에 대한 부정 2차 태그 수 (하나의 리뷰에 해당 2차 태그가 2개라도 1개로 판단)
        - 2차태그의 부정 판단 여부는 1차 태그와 동일
            - 예시) 상품 → 재질(부정적인 리뷰) - 내구성 → 1개의 특정 부정 태그
            예시) 상품 → 재질(긍정적인 리뷰) - 내구성 → 특정 부정 태그 X
    - 특정 태그
        - 긍정과 부정을 포함한 특정 2차 태그
        - 특정 상품일 경우 해당 상품에 대한 2차태그만 계산/쇼핑몰 전체일 경우 전체 상품에 대한 2차 태그 계산
        - 하나의 리뷰당 해당 2차태그는 1개까지 적용
            - 예시) 배송늦음"에 대한 부정 2차 태그 수 (하나의 리뷰에 해당 2차 태그가 2개라도 1개로 판단)
    - 부정 리뷰
        - LLM이 부정적이라 판단한 리뷰
    - 전체 리뷰
        - 긍정과 부정을 포함한 모든 리뷰

**E. 특정 상품 위험 리뷰 감지**

- 단 1건이라도 발생하면 안 되는 브랜드 리스크 키워드 포착
- 중요도가 높고 리스크가 크기때문에 고성능 전용 LLM 활용
- LLM이 맥락을 분석하여 아래 치명적인 위험 주제에 해당하는 리뷰를 판단
- **1시간이 아닌 실시간으로 진행**
- 위험 리뷰 감지 시, 리뷰를 게시하지 않음

    ```markdown
    # 역할 (Role)
    당신은 이커머스 플랫폼의 전문 '브랜드 리스크 관리자'이자 '리뷰 분석가'입니다. 고객 리뷰에서 브랜드에 치명적인 위험(Fatal Risk)이 될 수 있는 내용을 감지하는 것이 당신의 목표입니다.
    
    # 목표 (Objective)
    제공된 고객 리뷰를 분석하여 아래 정의된 "핵심 위험 카테고리"에 해당하는지 판단하십시오. "일반적인 불만"(예: 배송 지연, 단순한 맛/디자인 불호, 경미한 결함)과 "치명적인 위험"(예: 안전 사고, 법적 분쟁, 폭력적 언어)을 명확히 구분해야 합니다.
    
    # 핵심 위험 카테고리 (Critical Risk Categories)
    1. **제품 안전 및 신체적 위험 (Safety & Liability)**
       - 제품 사용으로 인한 신체적 상해, 질병, 병원 방문, 화재, 폭발, 감전 등을 언급하는 경우.
       - 주요 키워드: 병원, 응급실, 화상, 화재, 폭발, 식중독, 부작용, 피가 났다, 다쳤다.
       - *주의:* "맛이 폭발적이다", "심장이 멎을 뻔했다(너무 좋아서)"와 같은 은유적 표현은 위험이 아닙니다.
    
    2. **법적 분쟁 및 범죄 연루 (Legal & Criminal)**
       - 짝퉁(가품) 의혹 제기, 사기 주장, 고소/고발/경찰 신고 언급.
       - 제품의 불법적인 개조나 범죄 용도(마약, 몰카 등) 사용을 암시하거나 문의하는 내용.
       - 주요 키워드: 짝퉁, 가품, 사기, 경찰서, 고소, 신고, 소비자보호원(소보원), 불법.
    
    3. **사회적 논란 및 혐오 표현 (Social & Ethical)**
       - 특정 집단에 대한 혐오 발언(인종, 성별, 지역 차별)이나 제품과 무관한 과도한 정치적 선동.
       - 주요 키워드: 차별적 비하 용어, 혐오 표현, 정치적 비방.
    
    4. **악성 클레임 및 인격 모독 (Abuse & Harassment)**
       - 판매자나 상담원을 향한 심한 욕설, 성희롱, 살해 협박, 스토킹 위협.
       - *주의:* 단순히 "불친절하다", "기분 나쁘다" 정도의 불만은 위험이 아닙니다. 위협적이거나 모욕적이어야 합니다.
    
    # 지침 (Instructions)
    1. 리뷰의 맥락을 주의 깊게 읽으십시오.
    2. 해당 리뷰가 위 4가지 카테고리 중 하나에 속하는지 판단하십시오.
    3. 단순한 불만(배송, 단순 하자, 취향 차이)인 경우 `risk_detected: false`로 분류하십시오.
    
    # 예시 데이터 (Few-Shot Examples)
    
    User: "배송이 일주일이나 걸려서 짜증나네요. 다시는 안 사요."
    Assistant: 
    {
      "risk_detected": false,
      "risk_category": null,
      "severity": "None",
      "reason": "단순 배송 지연에 대한 불만임"
    }
    
    User: "이거 바르고 피부가 다 뒤집어져서 지금 응급실 왔습니다. 진단서 끊어서 청구할 겁니다."
    Assistant: 
    {
      "risk_detected": true,
      "risk_category": "Safety",
      "severity": "High",
      "reason": "제품 사용 후 신체적 상해(피부 발진) 및 병원 방문(응급실) 언급"
    }
    
    User: "와 진짜 맛이 폭발적이네요! 매운맛 좋아하시는 분들 강추합니다."
    Assistant: 
    {

      "risk_category": null,
    ```

**4-4. 고객이 떠올릴 수 있는 사용 모습 (As-is → To-be)**

- **As-is**
    - 출근해서 커피 마시며 어제 달린 리뷰를 하나하나 읽어봄.
    - "어? 이거 왜 지퍼 불량 이야기가 3개나 있지?" 뒤늦게 파악하고 부랴부랴 물류팀에 전화함. 이미 오전 배송은 마감됨.
    - 불량제품으로 인해 브랜드 이미지 손상
- **To-be**
    - 주말 저녁, TV를 보다가 알림톡이 울림. [모달 니트 가디건] : [품질] 관련 부정 태그 급증
    - 즉시 링크를 눌러 상황 파악 후, 문제 파악 후 배송 및 광고 일시 중지 처리
    - **브랜드 이미지가 손상되지 않고 리스크 관리 성공**

### **4-4. 이로 인해 가능해지는 대표 사용자 시나리오**

1. **배송 사고 조기 진압:** 택배 파업 등으로 배송 관련 불만이 급증할 때, 30분 내에 인지하고 배송 공지를 띄워 추가 항의를 방어.
2. **제품 불량 감지:** 특정 제조 일자 상품의 '불량' 리뷰가 연속 2건 들어오면 즉시 판매를 중지 처리하여 반품 비용 절감.

## **5. FAQ**

- Q. '최근 24시간 리뷰'를 매시간(1시간 주기) 수집하면, 중복 집계되는 리뷰가 발생하지 않나요?
    - A. 네, 중복 집계를 의도했습니다. 부정 태그를 단 1시간의 데이터만으로는 모수가 작아 '급증' 여부를 판단하기 어렵다고 생각하여 24시간 으로 넓게 잡되, 감시 주기는 1시간으로 설정하여 추세를 모니터링 할 수 있도록 했습니다.
- Q. 판단 조건인 '이전 14일 평균 부정 비율'이 0%인 경우(신상품 등)에는 어떻게 되나요?
    - A. 특정 태그의 이전 24시간 부정 비율 >= (이전 14일 부정 비율 + 10%p)를 적용하여 10% 이상일 경우 알림을 전송합니다.

## **7. No-Gos (이번 배포에서 하지 않는 것)**

- **실시간(Real-time) 감지 및 알림**
    - 서버 부하 및 비용 이슈를 고려하여, 분 단위 실시간 감지는 제외합니다. 1시간 단위(매 정각)로 감지합니다.

    - 개발 공수 / 이미 유사한 메뉴가 많아 오히려 고객에게 혼란을 줄 수 있을까 우려스러워 진행하지 않음
