## #1 청구서 개요

### 청구서(Invoice) 종류

> 구독은 고객사가 프로그램 사용권을 얻기 위해 구매하는 ‘서비스’의 ‘주문서’의 개념이며, 공급자(샐러드랩) 입장에서 해당 비용을 청구하는 청구서(Invoice)의 개념이고, 시스템 입장에서 서비스가 구동되는 프로그램 코드상의 명령 또는 상태의 개념.
> 
> 서비스 청구서: 무료 버전 구독, 체험 구독, 과금 구독, 임시 구독
> 
> 보조 청구서: 메시지 포인트 자동 충전, 메시지 포인트 일회성 충전, 설치 작업 대행, 제거 작업 대행
> 
> 임의 청구서: 기타 결제, 기타 환불, 예치금 환급, 예치금 생성

## #2 기능 활성화/비활성화

#### 기능 실행/중지의 의미

서비스 청구서는 결국 고객사가 어떤 기능을 활성화/비활성화 시킬 것 인가, 어떤 페이지에 접근할 수 있는가, 어떤 기능의 실행을 원천적으로 막을 것인가에 대한 근거가 되지만, 명시적으로 어떤 기능을 켜고 꺼야 하는지를 구현해 놓은 개념은 “기능 실행/중지”라고 명함

구현 목표

이러한 기능 실행/중지에는 대시보드, 주요기능, 메시징서비스, 위젯으로 4가지로 분류 됨

슈퍼관리자 기능 실행중지 팝업을 띄웠을 때 현재 기준의 실행중지 값 표시

기능 실행중지 팝업에서 설정한 값은 최우선순위 예외로 지정하여 따로 저장

ex) 기능실행중단표에 따르면 온보딩중, 위젯 사용 불가상태이나, 슈퍼관리자의 기능실행중지를 통해 강제로 ON시킬 수 있음

기능실행중지 필터에 옵션 추가, 슈퍼관리자의 기능실행/중지에 관련된 값들을 넣는 것 모든 실행/중지 설정, 특정 기능 실행중지에 관련된 값들은 유지

프로덕트별 기능 실행/중지는 프로덕트 상태 중, 구독없음과 서비스중단에서는 상태고정이 있든 없든 무시되고, 비활성화되며, 다른 상태로 변경됐을 때 고정됐던 설정이 되살아남

#### 기타

구독종료 vs 구독중단 vs 기능 중지

구독 종료: 사이트 상태값을 서비스청구서(무료구독, 체험구독, 과금구독)을 구독 중단상태로 만들어서 어떠한 종류의 구독도 남기지 않고 구독 중단됐을 때 “구독 종료”상태로 변경(해지 전단계)

구독 중단: 청구서 상태를 중단으로 만드는 것 서비스청구서(무료구독,체험구독, 과금구독)에 각각 ‘구독중단’상태 존재

기능 중지: 기능 중지는 구독과 관계 없이 기능의 동작을 제한하는 행위를 의미

## #3 구독 생성(구독 시작/수정)

> 서비스 청구서가 새롭게 생성되는 것은 생성 명령과 갱신 명령이 있다. 그중에서 생성 명령은 사용자(고객사 및 슈퍼관리자)가 입력한 정보를 바탕으로 새로운 구독이 생성 되는 명령이라는 점에서 이전의 구독정보를 바탕으로 구독이 생성되는 갱신 명령과는 다르다.

#### 개요

슈퍼관리자에 표시되는 명칭으로 구독 시작 또는 구독 수정 시, 고객사가 수행하는 온보딩 및 프로덕트 활성화(업그레이드/다운그레이드)에서 구독 생성 명령 가능

#### 구독 생성 명령의 하위 기능

| 기능 | 고객사 | 슈퍼관리자 |
| --- | --- | --- |
| 과금 구독 생성 기능 | 온보딩 시점, 또는 업그레이드/다운그레이드 시점에 선택한 프로덕트가 명시된 정기 구독 생성 가능 | 정기구독 및 계약구독 생성 가능(가격조건, 구독 기간 조건 등을 설정할 수 있음 계약구독 참고) |
| 구독 시작일 지정 기능 | 체험구독 생성 유무에 따라 자동 결정 | \- 직접 지정하여 현재 또는 미래, 과거 시점으로 설정 가능 - 과거 시점 선택 시 |
| 체험 구독 생성 기능 | 프로덕트별 사이트 기준으로 최초 1회만 제공 가능 | \- 구독 시작일을 미래시점으로 지정하면 오늘과 그 사이에 체험구독을 생성 - 구독 시작일이 오늘 또는 과거라면, 체험기간 생성 안됨 |
| 서비스청구서 비활성화 기능 | 고객사의 행동은 해당사항 없음 | 필수로 적용 |
| 예약중 구독변경 불가 기능 (해지/구독변경) | 고객사의 행동은 해당사항 없음 | 필수로 적용, 슈퍼관리자만 수정 및 중단 가능 |

#### 프로덕트 활성화 프로세스

프로덕트 활성화(온보딩 시점, 또는 업그레이드 시점) 시 최초 사이트 기준으로 최초 1회만 체험구독 자동 생성

설치 요청서가 발생하는 프로덕트라면, “서비스 개시” 또는 “직접설치완료" 시 체험구독 생성

## #4 구독 주기(갱신, 월간 시스템 주기)

### 구독 주기

구독 주기 종류

월간 구독: 결제 일자(구독 시작 일자)에 맞춰 구독이 매월 갱신되는 유형

연간 구독: 연간 구독 결제 일자에 맞춰 구독이 새롭게 갱신되는 유형

계약구독 - 3/6/9개월 구독: 계약구독은 월간, 연간 외에 3/6/9개월 구독도 가능

구독 주기별 과금 기준

구독이 시작된 시기에 따라 구독기간이 연간 구독인 경우 365일, 366일, 월간 구독인 경우 31일, 30일, 29일, 28일로 차이가 있는데, 일수와는 상관 없이 결제주기에 따라서 구독 비용을 부과

ex) 알파리뷰 89,000원 x 3개월 구독인 경우 91일이든 92일이든 상관하지 않고 267,000원으로 구독비용 결정

### 월간 시스템 주기

구독의 결제 일자부터 월간 시스템 주기가 발생하며,

월간 구독의 경우 해당 주기와 똑같이 월간시스템주기가 설정되며, 3/6/9연간 구독을 사용하고 있다고 하더라도 내부에서 월간으로 시스템이 진행 됨

월간 시스템 주기를 따라서 각종 시스템 알람, 한도 초기화, [무료 메시지 포인트 재충전](https://www.notion.so/1ff47fd1fcd24326917a1ddeba11a90a?pvs=25#85ddf45a3a8e4858838854393eb12b37) 이 진행됨

즉, 연간구독에서 반드시 필요한 주기로 ‘월간 구독’으로 결제 된다면 발생하는 월간 구독의 주기와 동일함

### 구독 기간과 결제일자

#### 월간 구독

최초 구독 시작 시

첫번째 과금 구독이 만들어진다면, 첫 시작이 몇일이든 해당 월의 마지막 일자까지만 일할계산 하여 구독 생성

다음월 갱신 시

다음 월 1일부터 말일까지 1개월 구독이 생성

결제일자 지정

결제일자는 1일로 지정됨

ex) 15일에 첫 과금구독 시작 시 15일~말일까지의 구독이 생성되고 다음 월(M+1)부터는 1일~말일로 갱신

(아키이브) 결제일자 1일 고정 전 기획

#### 연간 구독

시작일자별 구독 기간 결정

(가격이 365일과 다르지 않음)

23.03.01 ~ 24.02.29(366) vs 23.03.01 ~ 24.02.28(365) → 366일이 맞음

윤달의 경우

윤달의 2월 28일(또는 29일이 아닌 날짜) 결제 시작 시, 20xx.2.28~20xx.2.27(366일)

2월29일~2월 27일 365일 → 29일 결제일 불가 → 28일을 결제일로 지정

윤년 2월 29일: x년 29일 ~ 2월 28일 365일 → 28일로 결정

월간시스템주기 이슈

연간 구독이 시작됨과 동시에 월간시스템주기는 ‘월간구독주기’와 같은 정책으로 산정되지만, 연간 구독의 마지막 월간시스템주기에서 이슈가 있음

만약 3월 28일에 연간결제가 시작되면 첫번째 월간시스템주기는 “3월 28일~4월31일”

하지만 마지막 월간시스템주기는 “n+1년 3월1일~3월31”일이 아니라 구독의 마지막 날인 n+1년 3월1일~3월27일로 지정되어야 함.

구독변경 시 월간시스템주기 이슈

연간구독 중간에 구독변경이 일어나면, 해당 날짜로부터 기존 월간시스템의 마지막 종료일까지 1개월 미만의 첫 번째 월간시스템주기가 생성됨 (241211)

#### 집계 기간

[월간 시스템 주기](https://www.notion.so/cdff839defda43c8acaaf87a0bcacd92?pvs=25#9a38b3c6abb748e59747ec89b9814422) 의 기간을를 따름

한도 사용량은 집계 기간 동안, 매일 자정 업데이트

집계 기간 종료 후 다음 월간 시스템 주기 시작 시 한도는 0으로 초기화 됨(한도 사용률의 기준이 되는 집계 기간이 새롭게 잡히니까 당연한 말임)

구독 변경 시 한도 사용률 계산

구독 변경 시 기존 구독의 플랜 한도 사용률은 구독변경 당일까지의 집계 기간 내에 플랜 한도를 해당 집계 기간의 일수로 일할 계산하여 반올림 환산

ex) 기존 구독의 집계 기간 4.1 ~ 4.30 → 변경 후 기간 4.1~4.10(10일): 33.333% 플랜 한도: 100개 → 33개, 한도 사용량 10개 / 33개 = 30.333%

의 이슈

구독 변경 후 새로운 구독의 플랜 한도를 해당 구독 기간의 일수로 일한 계산하여 적용, 0의 자리 반올림

ex) 새로운 구독의 집계 기간 4.10~4.30(21일): 70% 플랜 한도: 100개 → 70개

구독 변경 후 기존 구독은 → 변경된 구독으로 바뀌고, 새로운 구독이 생성됨

#### 청구서 수정과 개정

#### 구독 변경 원리

구독 변경이 일어나면 “기존 구독”을 모두 환불하고 이 때, 크레딧, 예치금, 현금이 모두 환불되며 현금 또한 예치금으로 환불

“변경된 구독” 가격을 환불한 크레딧, 예치금으로 결제한다. 만약 [위약금](https://www.notion.so/1a667a31679247fb8ab957339519dbc7?pvs=25#123c69c6f4bf48ff849def1d5152047e) 으로 인해 기존 구독의 결제 내역보다 변경된 구독의 비용이 더 비싸질 수 있는데, 해당 위약금은 “새로운 구독”로 이월되어 결제됨

“새로운 구독”에 대한 비용을 결제하며 “ [위약금](https://www.notion.so/1a667a31679247fb8ab957339519dbc7?pvs=25#123c69c6f4bf48ff849def1d5152047e) ”이 있는 경우 비용에 포함하여 결제 진행

위약금: [위약금 발생 예시](https://www.notion.so/cdff839defda43c8acaaf87a0bcacd92?pvs=24#0d0d8fd5e226420eba4a3d5031e1f656) [💳과금 시스템 - 미납금/위약금](https://www.notion.so/1ff47fd1fcd24326917a1ddeba11a90a?pvs=24#123c69c6f4bf48ff849def1d5152047e)

“기존 구독”은 원본으로 보관되며 → “변경된 구독”이 변경된 날의 구독기간에 맞춰서 비용, 할인액 등이 조정되어 생성되며, 새로운 조건에 맞춰 “새로운 구독”이 생성

변경된 구독과 새로운 구독에는 각각 변경 유형에 맞춰서 변경

#### “변경된 구독”의 비용 산출 방법

is\_shared\_day 옵션으로 변경된 구독의 하루치 금액을 받을 것인가 안받을 것인가를 결정할 수 있음

아카이브: 기존 당일 사용분 까지 부과 250523

"구독률"은 사용일수 / 구독 기간 (째 자리에서 반올림)

① 기존구독의 구독기간 (4.25~5.24, 사용기간 30일) ② 5월 8일에 구독 변경 진행 시 (4.25 ~5.7, 사용기간 13일) ③ 구독률 13/30 = 0.433330\[…\] (소수점 6째 자리에서 반올림)

다수의 프로덕트를 이용한다면 각각 위 과정을 진행하여 전체 서비스 비용을 산출하게 되며, 할인 금액 또한 가격이 재산정 되는데, 위 ④를 기준으로 할인 금액이 재산정 됨

#### “새로운 구독”의 가격 산출방법

예시) ① 변경 후의 새로운 구독 기간 5/8~5/24 가정 ② 8일~24일은 17일 / 5월은 31일이나 이전 구독의 일수가 30일이고, 30일로 결정 → 17/30 = 56.666% ③ 정가 x 56.666%

아카이브: 기존 새로운 구독의 분모 결정 방식 231212

### 체험 구독

체험구독의 특징

고객사가 구독 최초 생성, 프로덕트 업그레이드로 인한 구독 변경 시 사이트 기준으로 최초로 프로덕트를 사용하게 됐을 때 자동 생성

슈퍼관리자의 구독수정을 통해서 추가로 강제 생성 가능

정기구독과 다르게 구독주기가 없음

과금이 되지 않아 영수증이 없음

플랜 한도는 무제한

프로덕트 체험구독과 패키지 체험구독

체험구독은 프로덕트마다 따로 생성되며 패키지의 체험 구독도 존재함

프로덕트의 체험구독은 하위 패키지의 사용 권한도 함께 제공하여 기능상 범위가 더 넓음

체험구독의 유일무이성(희소성)

체험구독은 최초 1회만 제공됨

프로덕트와 패키지 각각 체험구독 발생 가능

프로덕트의 체험구독에는 하위 패키지를 포함함

따라서 프로덕트 체험 구독을 이미 사용했다고 하더라도, 각 하위 패키지에 대하여 별도로 체험구독 생성 가능

어뷰징을 막기 위해 최초 과금 구독일로부터 1개월 뒤 부터 활성화 되는 기능 등 필요

체험구독의 예약중 상태 과금구독 생성/수정 기능

체험구독이 생성 될 때 예약중 상태의 과금구독(정기구독, 계약구독)이 생성됨

예약중 구독이 생성 될 때 기존에 라이브중인 프로덕트와 기준을 포함해서 생성되어야 하고, 예약중인 구독은 플랜 안심 한도 옵션에 의해, 구독기간이 시작될 때 플랜은 업그레이드될 수 있음

같은 날 2개 이상의 프로덕트의 체험 구독이 종료된다면 예약구독은 1개만 존재(즉, 통합되어 한번의 예약 명령만 남아서 구독변경 명령을 1번만 하게 된다는 뜻)

체험구독이 체험기간 연장 쿠폰이나 프로덕트 추가 프로모션에 의해서 수정될 때 과금구독(정기구독, 계약구독)의 시작일도 그에 맞춰 수정됨

기간 변동 가능

구독기간은 기본 14일이지만, 체험 기간 연장 쿠폰이나 프로덕트 추가 프로모션에 의해 유동적으로 변화 가능

체험구독은 과금 구독을 전제로 진행 되며, 체험 기간 종료 후 즉시 해당 기능이 추가 된 내역으로 구독 변경

프로덕트별 체험기간 생성 → 정기 구독 변경 원리
