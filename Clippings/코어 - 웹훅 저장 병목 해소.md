### 1\. Problem

> 지금 우리고객은 어떤 문제를 겪고 있을까. 구체적인 사례와 데이터를 포함한다.

유동적인 트래픽에 대응하기 힘든 경직된 처리 구조 탓에, 고객사가 늘어날수록 병목 현상에 따른 데이터 신뢰도 하락과 인프라 비용 부담이 가중되는 악순환을 해결한다.

현재 웹훅처리는 AWS EC2라는 서비스를 이용하여 처리하고 있으며, 이 방식은 EC2 인스턴스의 사양에 따라 성능이 크게 좌우됩니다. 최근 카페24 프로 업체들로 인해 웹훅 처리량이 증가하면서, 기존 사양의 처리 속도로는 한계를 느껴 인스턴스 크기를 2배로 늘려 단위 시간내 처리량은 늘어났지만 비용 또한 증가하였습니다. 뿐만 아니라 수 천개 쇼핑몰의 웹훅이 병렬로 처리되는 반면 최종적으로 저장되는 플로우가 한 곳에 모여있으므로 이로 인한 병목이 발생하기 취약한 구조입니다.

11월 10일 오후, EC2 인스턴스 사양을 2배 늘려 평균 CPU 사용률이 감소한 상황

EC2에선 Redis 캐시 서버에 저장된 웹훅 데이터를 소모하여 저장하고 있습니다. Redis 캐시 서버 내에는 웹훅 데이터가 shop\_id를 1000 단위로 나누어 그룹지어 저장됩니다. 이는 특정 쇼핑몰에서 웹훅 데이터가 많이 발생할 경우, 특정 그룹 내에 모든 쇼핑몰에게 웹훅 처리 지연이 공유된다는 문제가 있습니다. 뿐만 아니라, 앞으로 쇼핑몰이 늘어나면서 shop\_id가 크게 늘어날 때마다 수동으로 리스트를 추가적으로 나눠야 하는 문제도 있습니다.

각 shop\_id가 생성된 시점. 1만개가 추가되는 주기가 점차 짧아지고 있음

웹훅 데이터가 저장되는 캐시 서버는 데이터 용량이 유동적이기 때문에 서버리스 인스턴스를 사용하고 있습니다. 이 존건은 사양을 구체적으로 정하였을 때 보다 더 비싸며, 병목으로 인해 저장된 용량이 많은 채로 오래 지속될수록 비용이 증가합니다.

### 2\. Appetite

> 문제를 해결하기 위해 우리는 얼마의 시간을 "베팅"할 것인가

Big Batch (3주): 새로운 람다 설정 및 테스트, 부하 테스트, A/B 테스트 및 동시에 호출되는 여러 람다와 데이터베이스간의 연결, 세션 관리

Small Batch (1주)

제약 사항

데이터베이스 연결: 동시에 호출되는 여러 Lambda와 연결되는 데이터베이스에 부하가 발생해선 안됩니다.

소스코드: 데이터를 처리하고 저장하는 람다의 소스코드는 서버와 같은 Repository에서 형상관리가 이루어져야 합니다.

### 3\. Solution

> 문제 해결을 위한 핵심 아이디어를 적는다.

현재 웹훅 처리 플로우 중 AWS Lambda → ElastiCache → EC2 부분을 변경하여 Lambda에서 즉시 데이터베이스에 저장하도록 변경하여 병목을 해소하고 리소스 사용으로 인한 리소스를 감소시킴

플로우 변화

기존: 코어 시스템 서버 → AWS SQS → AWS Step Functions(Lambda) → AWS ElastiCache(Redis) → AWS EC2 → 데이터베이스

개선: 코어 시스템 서버 → AWS SQS →

### 4\. Map the Scopes

> Scope는 프로젝트 시작 전 완벽하게 정할 수 없다. 진행하며 점진적으로 구체화한다.

#### Unknown (아직 알아보는 단계)

람다에서 데이터베이스 저장까지 실행할 경우 증가하는 람다 실행 시간과 이로 인한 새로운 병목

#### Known (개발시작을 위한 범위세팅)

| 항목 | 내용 |
| --- | --- |
| 람다 정의 | ECR 이미지를 이용한 람다 세팅 |
| AWS 리소스 재정의 | 기존에서 변경이 아닌 새로운 AWS 리소스를 테라폼을 이용해 구축 |
| A/B 테스트 테스트 환경 구축 | 위 개발내용을 일부 업체를 제한하여 도입할 수 있고 이를 조절할 수 있는 환경 구축 |

> 미래 예상되는 (기술/정책) 리스크 관련, 현재의 대응계획을 미리 적는다

Q1. 수 많은 람다에서 데이터베이스에 연결을 요청할 경우 발생하는 문제가 없을까요?

A. 이와 관련하여 리서치 및 테스트가 필요합니다.

Q2. 기존 구조(Redis)는 메모리에라도 데이터를 들고 있었던 것 같은데요. 변경된 구조에서 Lambda가 DB 저장에 실패하면 해당 웹훅 데이터는 어디로 가는걸까요? 실패한 웹훅을 따로 저장했다가 재처리하는 과정이 없으면 주문 누락 등의 사고가 예상됩니다

A. Lambda에서 동작이 실패할 경우 앞단에서 데이터를 전달해준 SQS에서 메시지가 삭제되지 않아 데이터가 최대 14일까지 보존됩니다. SQS에서는 Lambda가 메시지(데이터)를 가져와 처리가 완료되어야만 삭제한다고 이해하시면 됩니다.
