# [[구독개편 범위 및 로드맵]]

## 📌 한 줄 통찰 (The Karpathy Summary)

> 구독개편은 "다듬기(L1) → 통폐합(L2) → 통합 재설계(L3)"의 3레벨 프레임으로 범위를 선택하되, L3는 에이전틱 알파앱스가 프로덕트 경계를 해체한 뒤에야 의미 있으므로 L1+L2 선행이 최적 전략이다.

## 📐 도메인 모델 개요 (Domain Model Overview)

| # | 도메인 | 핵심 개념 |
|---|--------|-----------|
| 1 | 개편 배경 및 의사결정 프레임 | 이현준 제안 4단계 → 실행 가능성 기준 3레벨 재구성, 통합 알파앱스 런칭 시점과의 정합성 |
| 2 | L1: 다듬기 (Polish) | 기존 플랜 구조 유지, 금액·UX·종량제만 손질. 상태머신 비접촉 |
| 3 | L2: 통폐합 (Consolidate) | 프로덕트별 10단계 티어 → 4~5단계 축소, 주문수 기준 통일, 결합할인 확정. 마이그레이션 필요 |
| 4 | L3: 통합 재설계 (Redesign) | 사용량 기반 단일 플랜 or 통합 플랜+애드온. 과금 엔진 리빌드, 프로덕트→기능 단위 과금 전환 |
| 5 | 페이즈별 로드맵 | Phase 0(의사결정 2주) → Phase 1(L1, 6~8주) → Phase 2(L2, 8~10주) → Phase 3(L3, 12~16주) |
| 6 | 리스크 및 제약 | 상태머신 복잡도, 계약구독 수동 대응, CS 부담, 결합할인 미확정 |

## 📖 핵심 도메인별 상세 (Domain Deep-Dive)

### 📖-1. 개편 배경 및 의사결정 프레임

**원본 아젠다 (이현준, 2026-05-04):**

이현준 팀리더가 제기한 핵심 질문 5가지:

| # | 질문 | 함의 |
|---|------|------|
| 1 | 요금제 개편이 왜 필요한가? | 개편의 당위성 검증 |
| 2 | 안 하면 무슨 일이 생기는가? | 얻는 것/잃는 것/미뤄지는 것 평가 |
| 3 | 통합 알파앱스 런칭 시점에 꼭 새 요금제가 필요한가? | ⚠️ 진정한 통합 런칭은 3분기 예상, 2분기는 단순 메뉴 수정 수준 |
| 4 | 최소 리소스로 동일 임팩트를 낼 수 있는 방법은? | 비용 대비 효과 최적화 |
| 5 | 2~3달 안에 가능한가? | 일정 실현 가능성 |

**원본 4단계 수준 (이현준 제안):**

| 수준 | 내용 |
|------|------|
| 1 | 기존 요금제 큰 틀 유지, 금액·UX 다듬기 |
| 2 | 프로덕트 내 플랜/티어 필요한 만큼만 수정 |
| 3 | 프로덕트간 플랜/티어 일관성 있게 통합/통일 |
| 4 | 완전히 새로운 구독 체계 (사용량 기반 단일 플랜, 통합 플랜+애드온, 서비스별 통합 플랜, 서비스별 종량제 등) |

**→ 실행 가능성 기준 3레벨 재구성:**

원본 수준 1을 L1, 수준 2~3을 L2로 통합, 수준 4를 L3로 매핑. 수준 2와 3은 프로덕트 내/간의 차이일 뿐 기술적 작업 범위가 유사하므로 하나의 페이즈로 묶는 것이 효율적.

| 레벨 | 원본 수준 | 이름 | 상태머신 영향 | 마이그레이션 | 리스크 |
|------|-----------|------|---------------|--------------|--------|
| **L1** | 수준 1 | 다듬기 (Polish) | 비접촉 | 불필요 | 낮음 |
| **L2** | 수준 2+3 | 통폐합 (Consolidate) | 부분 접촉 | 자동 매핑 필요 | 중간 |
| **L3** | 수준 4 | 통합 재설계 (Redesign) | 전면 리빌드 | 전면 마이그레이션 | 높음 |

---

### 📖-2. L1: 다듬기 (Polish)

**범위:** 기존 플랜 구조(라이트~엔터6) 그대로 유지. 과금 상태머신, 일할계산 로직, 위약금 로직 일체 미접촉.

**변경 대상:**

| 항목 | 현재 문제 | 개선 내용 |
|------|-----------|-----------|
| 금액 조정 | 플랜별 가격이 경쟁력 대비 재검토 필요 | 플랜별 가격 재산정, 할인율 정리 |
| 청구서 UX | 청구서 보기 어려움 (이현준 언급) | 청구서 가독성 개선 |
| 요금제 안내 | 안내 페이지 부재 | 요금제 안내 페이지 신설 |
| 종량제 과금 | 알림톡/문자 과금 방식 복잡 | 메시지 포인트 정책 단순화, 과금 방식 개선 |

**소요 기간:** 6~8주
**산출물:** 변경된 가격표, 새 안내 페이지, 개선된 청구서 UI

⚠️ **L1 단독으로도 고객 불만의 약 80%를 해소할 수 있음** — 가장 적은 리소스로 가장 큰 체감 임팩트를 내는 구간

---

### 📖-3. L2: 통폐합 (Consolidate)

**범위:** 프로덕트별로 상이한 플랜/티어 구분 및 기준을 일관성 있게 축소·통일.

**핵심 변경 사항:**

| 항목 | 현재 상태 | 목표 상태 |
|------|-----------|-----------|
| 티어 수 | 알파업셀 기준 10단계 (라이트~엔터6) | 4~5단계 (Free / Starter / Growth / Business / Enterprise) |
| 플랜 기준 | 프로덕트별 주문수 한도 기준 상이 | 전 프로덕트 동일 주문수 구간 적용 |
| 결합할인 | 정책 미확정 (2개 10%, 3개 20% 설계만 존재) | 정책 확정 및 구현 |
| 한도 체계 | 프로덕트별 한도 기준 상이 | 통일된 한도 기준 적용 |

**마이그레이션 원칙:**
- 기존 고객: 자동 매핑 로직으로 가장 가까운 상위 플랜에 매핑 (불이익 없는 전환 보장)
- ⚠️ 계약구독(계좌이체) 고객: 커스텀 조건이므로 자동 마이그레이션 불가 → 수동 대응 필수
- Grandfather 옵션: 기존 고객은 기존 가격 유지, 신규 고객만 새 체계 적용 (vs 전면 전환)

**소요 기간:** 8~10주
**산출물:** 통합 플랜표, 마이그레이션 스크립트, CS 가이드

---

### 📖-4. L3: 통합 재설계 (Redesign)

**범위:** 에이전틱 알파앱스에 맞는 완전히 새로운 구독 체계. 프로덕트(리뷰·업셀·푸시) 구분 자체를 해체.

**선결 조건:** 통합 알파앱스 3축(세팅/운영/성장) MVP 런칭 완료

**4가지 모델 후보 (이현준 제안 원문):**

| 모델 | 설명 | 벤치마크 |
|------|------|----------|
| 사용량 기반 단일 플랜 | 하나의 통합 플랜, 사용량에 따라 요금 변동 | Make.com, Mixpanel |
| 통합 플랜 + 애드온 | 단일 기본 플랜 + 기능/서비스별 애드온 추가 과금 | Channel.io, Zapier |
| 서비스별 통합 플랜 | 전체 티어 구분 동일, 하위 서비스별 요금만 상이 | Grafana, JetBrains |
| 서비스별 종량제 플랜 | 서비스별 플랜 구성 및 요금 모두 상이 | Klaviyo, Datadog |

**과금 엔진 변경 범위:**
- 기존 상태머신 단순화 (현행: 사이트→프로덕트→구독→결제 4중 상태 연쇄)
- 프로덕트 단위 과금 → 기능 단위 과금 전환
- 일할계산·위약금·is_shared_day 등 레거시 로직 전면 재설계

**소요 기간:** 12~16주 (마이그레이션 기간 별도)
**산출물:** 신규 구독 체계, 마이그레이션 완료, 기존 체계 일몰(sunset)

⚠️ **L3를 에이전틱 알파앱스 MVP 전에 진행하면, 프로덕트 경계가 아직 살아있는 상태에서 과금 체계만 먼저 해체하게 되어 정합성 문제 발생. 2번 작업하게 될 리스크 높음.**

---

### 📖-5. 페이즈별 로드맵

| 페이즈 | 기간 | 레벨 | 핵심 산출물 | 선결 조건 |
|--------|------|------|------------|-----------|
| **Phase 0** | 2주 (~2026-05-31) | 의사결정 | 레벨 확정, 모델 방향 결정, 마이그레이션 원칙 합의 | 이현준 아젠다 논의 완료 |
| **Phase 1** | 6~8주 (2026-06 ~ 2026-07) | L1 | 가격표, 안내 페이지, 청구서 UI | Phase 0 완료 |
| **Phase 2** | 8~10주 (2026-08 ~ 2026-10) | L2 | 통합 플랜표, 마이그레이션, CS 가이드 | Phase 1 완료 |
| **Phase 3** | 12~16주 (에이전틱 알파앱스 MVP 이후) | L3 | 신규 구독 체계, 기존 체계 sunset | 통합 알파앱스 3축 MVP 런칭 |

**핵심 의사결정 포인트:**

| # | 질문 | 선택지 |
|---|------|--------|
| 1 | 통합 알파앱스 런칭 시점에 새 요금제가 꼭 필요한가? | A) 2분기 메뉴 수정엔 L1이면 충분 / B) 3분기 통합 런칭에 맞춰 L2까지 필요 |
| 2 | 최소 리소스로 동일 임팩트를 낼 방법은? | L1만으로도 "요금제 안내 페이지 + 가격 조정"으로 고객 불만 80% 해소 가능 |
| 3 | L3는 언제가 적기인가? | 에이전틱 알파앱스가 "프로덕트 구분"을 해체할 때 비로소 의미 있음 |
| 4 | 마이그레이션 전략 | Grandfather(기존 고객 가격 유지) + 신규만 새 체계 vs 전면 전환 |
| 5 | 2~3달 안에 가능한가? | L1: 가능(6~8주) / L2: 빠듯(8~10주) / L3: 불가능(12~16주+) |

### 📖-6. 리스크 및 제약

| 리스크 | 영향 레벨 | 상세 | 대응 |
|--------|-----------|------|------|
| 상태머신 복잡도 | L3 | 일할계산, 위약금, is_shared_day, 구독률 공식 등 레거시 로직이 4중 상태(사이트→프로덕트→구독→결제)로 깊이 얽혀 있음 | L1·L2 선행으로 점진적 단순화 |
| 계약구독 고객 | L2~L3 | 계좌이체·커스텀 조건 고객은 자동 마이그레이션 불가 | 수동 대응 리스트 사전 작성, CS팀 협의 |
| CS 부담 | L1~L3 | 요금제 변경 시 CS팀 교육·고객 안내 리소스 필수 | Phase별 CS 가이드 선제 작성 |
| 결합할인 미확정 | L1 | 현재 "2개 10%, 3개 20%" 설계는 존재하나 정책 미확정 | L1 Phase에서 우선 확정 필요 |
| L3 조기 착수 리스크 | L3 | 프로덕트 경계가 살아있는 상태에서 과금 체계만 해체 시 정합성 붕괴, 2번 작업 발생 | 에이전틱 알파앱스 MVP 이후로 L3 지연 |

## ⚠️ 모순 및 업데이트 (Contradictions & RL Update)

- **[2026-05-13] 초안 작성:** 이현준 아젠다(2026-05-04)의 4단계 수준을 3레벨로 재구성. 수준 2(프로덕트 내)와 3(프로덕트 간)을 L2로 통합한 것은 기술적 작업 범위 유사성 기반 판단. 이후 논의에서 분리가 필요할 수 있음.
- **[2026-05-13] 결합할인 정책 상태:** 알파솔루션 구독 및 과금 시스템 종합 브리핑(2026-04-19)에서 "정책 미확정" 명시. 이 초안에서 L1 단계 확정 필요로 분류.
