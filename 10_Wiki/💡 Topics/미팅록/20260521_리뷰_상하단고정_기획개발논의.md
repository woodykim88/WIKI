## 2. 주요 논의 내용

### 2.1. 큐레이션 이벤트 (EPL) 액션 정의
* **VIEW**: 보기만 하고 아무것도 안 하는 케이스 (View만 발생, 액션 없음) ➔ 신호로 기록하며 개선 방향 추적 필요
* **Accept**: 상단 추천을 수락(적용)하는 액션
* **Like**: 추천된 리뷰는 좋지만, 최상단 고정을 안 하고 건너뛰는 skip 액션
* **Deny**: 이 리뷰는 완전히 추천하지 말라는 완전 거부(Deny) 액션

### 2.2. 상단고정 자동화 범위 및 정책
* 상단고정 자동화는 기존 테이블을 재활용하며, 업체 노하우가 축적된 상단고정 로직을 그대로 사용한다.
* 상단 고정 자동화 시, 기존에 정립되어 있던 상단 고정 위에 리뷰를 올린다는 개념으로 접근한다.
  * 기존 솔트값을 삭제하는 게 아닌 맨 위로 올리는 개념을 도입한다.
  * 예: 99위가 최고점이면 100위로 설정하여 최상단에 배치하는 방식. (이때 솔트값은 실제로 바뀌게 됨)
* 상단고정 자동화 기능은 향후 유료 판매 플랜으로 고려할 가능성을 언급함 (이번 베팅 스펙 외).

### 2.3. 배치 처리 주기
* 상단고정 추천/분석: 실시간 처리 가능
* **상단고정 자동화(이하 '자동정렬') 배치 주기**: **7일(주간) 배치**로 최종 결론
  * 관리자가 주 단위로 일괄 처리하는 패턴이 많아 7일로 결정.
  * 1시간 배치도 허용 가능하다는 의견이 있었으나 최종적으로 7일 주간 배치로 정리.
  * 상단고정 자동화는 오늘부터 배치에 적용 필요.

### 2.4. 크론 (Cron) 구조 논의
* **핵심 질문**: 기존에 리뷰 게시상태를 변경하는 크론 작업이 돌아간 이후에 리뷰 추천에 대한 크론이 돌아야 하는지에 관한 연계 문제 분석.
* **통합 vs 별도 분리**: 기존 리뷰 트래킹 크론(15분 주기)과의 통합할 것인가, 혹은 별도 크론을 생성할 것인가의 문제.
  * **문제 시나리오**: 7일 배치로 상단 고정된 리뷰가 리워드 문제 등으로 '게시대기' 상태로 전환될 경우, 업체가 실제 화면에 노출되지 않아 혼란을 겪을 우려 존재.
  * **결론**: 기존 크론에 통합하면 로직 복잡도가 기하급수적으로 늘어날 수 있으므로, **별도 크론으로 분리하는 방향**을 강력 검토한다.
  * 기존 크론에서는 더블체크 용도로만 활용하며, 현희님이 코드를 직접 확인 후 최종 구조를 결정할 예정.

### 2.5. API 설계
* 큐레이션 정보 업데이트 API: 구현 완료
* 상단 고정 후보 가져오기 API: 미구현 상태

### 2.6. 하단 배치 기준 (유니스코어 하한선) 재검토
* 1점대 리뷰가 하단으로 갈 만큼 나쁜 품질인지에 대한 의문 제기.
  * 예시: '좋아요 좋아요'가 5건 반복된 리뷰의 경우, 성의 없는 리뷰이지만 현재 기준(uni_score 1점대 이하)에서는 하단에 배치됨.
  * 실제 운영 중인 0.8점짜리 리뷰도 존재함.
* 정밀도 검증을 위해 점수 데이터의 추가적인 분석 필요성을 확인.
* 최종 점수 기준(상단 몇 점 이상을 추천하고 하단 몇 점 이하를 하단 고정할지)은 정태성 PO가 실제 데이터를 보며 확정할 예정.

## 3. 최종 결정 사항
* **자동화 범위**: 상단고정 자동화만 진행 (하단 배치 자동화는 이번 범위에서 제외)
* **배치 주기**: 7일(주간) 배치로 결정
* **크론 구조**: 별도 분리 방향으로 검토, 코드 확인 후 현희님이 최종 결정
* **추천 목록 조회 API**: 태성님 또는 현희님이 신규 구현 예정
* **유니스코어 상/하단 기준**: 데이터 분석 후 PO가 별도 확정
