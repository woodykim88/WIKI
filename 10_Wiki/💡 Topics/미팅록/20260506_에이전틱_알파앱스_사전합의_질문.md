# 에이전틱 알파앱스 사전합의 질문 및 의견 (2026-05-06)

## 📌 한 줄 통찰 (The Karpathy Summary)

> 에이전틱 알파앱스 런칭 전 합의해야 할 4대 질문(진짜 문제인가 / Phase 1 스코프 / 협업 정의 / 투자 각오)에 대한 사전 의견 정리. 핵심 결론: "상품력을 매출로 전환하는 실행 엔진"으로 경계를 세우고, B세그먼트(소형팀)에 리뷰 자동화 1개 축부터 시작하며, Q3는 전사 올인이 아닌 선발대 PoC로 진행.

## 질문 1-1. 이거 진짜 문제인가? 왜 고객들은 돈을 잘 못벌까?

### 문제 진단 재점검

컨셉 문서의 문제 정의는 **"인지 범위 초과 → 도구에서 결과로"**인데, 이건 **우리 제품의 문제**이지 **고객이 돈을 못 버는 문제**가 아니다. 두 질문을 분리해야 한다.

**고객이 돈을 못 버는 진짜 이유 (이커머스 셀러 관점):**

| 원인 계층 | 구체적 현상 | 알파앱스가 건드릴 수 있나? |
|-----------|------------|--------------------------|
| **전략 부재** | "뭘 해야 할지 모름". 데이터는 있는데 해석 못함, 다음 액션을 도출 못함 | **해결 가능** — 3C 분석 + 에이전트 제안 |
| **실행 역량 부족** | 배너 디자인, 광고 소재, CRM 세그먼트 설계를 할 줄 모르거나 할 시간이 없음 | **해결 가능** — 자동 생성 + 자동 집행 |
| **도구 복잡성** | 도구가 있어도 세팅이 어렵고, 여러 도구를 조합해야 효과가 남 | **해결 가능** — 통합 + 자동 세팅 |
| **상품력/가격 경쟁력** | 상품 자체가 매력이 없거나 가격이 안 맞음 | **해결 불가** — 마케팅 도구의 영역 밖 |
| **채널/트래픽 부족** | 자사몰에 유입 자체가 적음 | **부분 해결** — 애즈/채널링으로 일부, 그러나 핵심은 아님 |

### 의견: "해결할 수 있는 문제"의 경계를 먼저 선언해야 한다

경계를 세우지 않으면 "에이전틱 알파앱스를 쓰면 매출이 오른다"는 과대 약속이 되고, 고객 기대치와 실제 결과의 갭에서 이탈이 발생한다.

**권장 프레이밍:**

> "상품력은 있는데 운영 역량이 부족하여 매출이 정체된 셀러"가 타겟.
> 알파앱스는 **상품력을 매출로 전환하는 실행 엔진**이다. 상품력 자체를 만들어주진 않는다.

이 경계를 세우면 "씨딩" 영역의 모호함도 자연스럽게 정리된다 — 씨딩은 상품력이 검증된 후의 확산 수단이지, 상품력을 만드는 수단이 아니다.

## 질문 1-2. 첫 런칭 대상 타겟은 누구인가?

### 3개 후보 비교

| 세그먼트 | 특징 | 자동화 니즈 | ARPU | 세일즈 난이도 | 프리셋 방향 |
|----------|------|-------------|------|--------------|-------------|
| **A. 1인 셀러 (월 주문 ~500건)** | 대표=기획=CS=마케터. 시간이 가장 부족 | 극도로 높음 — "전부 해줘" | 낮음 (3~5만원대) | 낮음 (셀프서브) | 풀오토 프리셋. 판단도 대행 |
| **B. 소형팀 (2~5인, 월 주문 500~3,000건)** | 마케팅 담당자 1명 있으나 전문성 부족 | 높음 — "세팅 해주고, 제안해줘" | 중간 (10~30만원대) | 중간 | 추천형 프리셋 + 사람 승인 |
| **C. 중형 (5~20인, 월 주문 3,000건+)** | 마케팅팀 존재. 전략은 있으나 실행 리소스 부족 | 중간 — "실행을 대행해줘" | 높음 (30~100만원+) | 높음 (영업 필요) | 커스터마이징 가능 프리셋 |

### 의견: B 세그먼트 (소형팀) 우선 런칭

1. **"마케팅 담당자 1명"이 존재**하므로, 에이전트의 제안을 평가하고 승인할 사람이 있음. 1인 셀러는 판단까지 대행해야 하는데, 초기 AI 품질로는 리스크가 큼
2. **ARPU와 세일즈 난이도의 균형**이 가장 좋음. A 세그먼트는 ARPU가 너무 낮아 에이전트 운영 비용을 커버 못할 수 있고, C 세그먼트는 세일즈 사이클이 길어 초기 모멘텀을 만들기 어려움
3. **현재 알파앱스 과금 구조와 정합** — 알파리뷰 최소 39,000원/월 + 업셀/푸시 패키지를 고려하면 B 세그먼트의 지불 의사(10~30만원대)와 매칭
4. **프리셋 난이도가 적절** — "추천 → 승인" 모델은 에이전트가 틀려도 사람이 교정 가능. 이 과정에서 에이전트 품질 데이터가 축적됨

**B 세그먼트 기반 프리셋 방향:**
- 초기 진단: 카페24 연동 시 주문수·상품수·리뷰수·객단가·재방문율 자동 스캔 → "당신 쇼핑몰의 성장 병목은 ___입니다" 진단 카드
- 프리셋 제안: 병목에 따라 "리뷰 확보 우선" / "객단가 향상 우선" / "리텐션 강화 우선" 3가지 트랙 제시
- 1주일 후: 에이전트가 첫 주간 보고서 + 다음 주 추천 액션 제안

## 질문 2. Phase 1에 대해서 합의합시다

### PRD Phase 구분에 대한 재고

PRD에서 제안한 Phase 구분은 **기술 아키텍처 관점**이다. 고객 가치 관점에서 재구성하면 Phase 1은 다르게 정의되어야 한다.

**제안: Phase 1 = "첫 만남에서 첫 성과까지" (Time-to-First-Value)**

| 구분 | PRD 원안 | 재제안 |
|------|----------|--------|
| Phase 0 | 공통 온보딩, 인증 통합 | 동일 (필수 인프라) |
| Phase 1 | 운영 자동화 전체 (리뷰+상담+팝업+대시보드+보고서) | **축소**: 자동 세팅 + 진단 + 리뷰 자동화 1개 축만 |
| Phase 2 | 성장 에이전트 | 운영 자동화 확장 + 성장 기초 |

### Phase 1 재정의 상세

```
[Phase 1 스코프]
1. 자동 세팅 (10분 완료) ← 세팅 축의 핵심
2. 쇼핑몰 진단 카드 ← "왜 매출이 안 나는지" 첫 인사이트
3. 리뷰 자동 운영 ← 운영 축 중 가장 성숙한 1개만
   - 리뷰 자동 게시/미게시 판단
   - 리뷰 자동 답변 (긍정/부정 분기)
   - 주간 리뷰 요약 보고서
```

### 왜 리뷰만 먼저인가

1. 알파리뷰가 가장 기능이 풍부하고 데이터가 많음 (review_type 4종, creater_type 9종, 위젯 20개+)
2. 리뷰는 **성과가 가장 가시적** — "리뷰 100개 → 전환율 X% 상승"이 눈에 보임
3. 리뷰 자동화의 난이도가 상대적으로 낮음 (룰 기반 + AI 답변, 이미 AI 분석 인프라 존재)
4. CRM/광고는 에이전트 판단 오류의 비즈니스 임팩트가 큼 (돈이 나감). 리뷰는 실수해도 수정 가능

### Phase 1 성공 기준

- 온보딩 10분 완료율 > 70%
- "진단 카드를 보고 유료 전환" 비율 측정 (새 퍼널 지표)
- 리뷰 자동 처리율 > 60% (수동 개입 40% 이하)
- Time-to-First-Value < 24시간 (가입 → 첫 자동 액션 발생)

## 질문 3. 온보딩 이후 1주일 여정에서의 '협업' 정의

### "협업"이 아닌 "위임의 단계"로 정의해야 한다

"협업"이라고 하면 사람과 에이전트가 동등하게 일하는 것처럼 들리는데, 실제로는 **사람이 에이전트에게 위임하는 범위를 점진적으로 넓히는 과정**이다.

### 1주일 여정 — 위임의 3단계

```
Day 0        Day 1-2          Day 3-5         Day 6-7
───────────────────────────────────────────────────────
[관찰 모드]   [제안 모드]       [자동 모드]      [보고 모드]
 에이전트가    에이전트가        승인된 영역에서   첫 주간 보고서
 쇼핑몰 분석   "이렇게 하면     에이전트가       + 다음 주 제안
 + 진단 카드   어떨까요?" 제안   자동 실행
```

| 단계 | 에이전트 행동 | 사람 행동 | 알림 방식 |
|------|-------------|-----------|-----------|
| **Day 0: 관찰** | 카페24 데이터 스캔, 리뷰 현황 분석, 위젯 상태 진단 | 진단 카드 확인, "맞다/아니다" 피드백 | 카카오 알림톡 1회 "진단 완료" |
| **Day 1~2: 제안** | "미답변 리뷰 23건 발견. 이렇게 답변하면 어떨까요?" | 제안된 답변 검토, 수정/승인 | 대시보드 + 알림톡 "제안 3건 대기 중" |
| **Day 3~5: 자동** | 승인 패턴 학습 완료. 유사 리뷰에 자동 답변. 예외만 에스컬레이션 | 에스컬레이션된 건만 처리 | 에스컬레이션 시에만 알림 |
| **Day 6~7: 보고** | 첫 주간 보고서: 자동 처리 N건, 에스컬레이션 N건, 리뷰 작성률 변화, 다음 주 추천 액션 | 보고서 리뷰. 다음 주 위임 범위 결정 | 알림톡 "첫 주 리포트 도착" |

### 핵심 설계 원칙 3가지

1. **"절대 먼저 실행하지 않는다" (Day 0~2):** 처음 48시간은 제안만. 이 기간에 고객이 에이전트의 판단 품질을 체감하고 신뢰를 형성
2. **"틀려도 되는 영역부터 자동화한다":** 리뷰 답변은 수정 가능. 가격 변경이나 프로모션 집행은 Phase 1에서 절대 자동화하지 않음
3. **"사람이 위임 범위를 결정한다":** 에이전트가 스스로 권한을 넓히지 않음. Day 7 보고서에서 "다음 주에는 이것도 자동화할까요?"로 명시적 제안

## 질문 4. 3분기에 OKR 수준으로 전사 투자할 각오가 되어 있는가?

### 각오가 되어 있는지보다, "각오가 될 수 있는 조건"을 먼저 정리해야 한다

오늘(5/6) 코어팀 조직개편 미팅에서 나온 논의와 직접 연결된다.

### 현실 체크 — 3분기에 가능한가?

| 조건 | 현재 상태 | Gap |
|------|-----------|-----|
| 인력 | 코어팀 확장 논의 중 (주니어 PO/PD 채용 진행). 개발자 추가 채용 계획 | 채용 → 온보딩 → 생산성 확보까지 최소 2~3개월. **Q3 시작 시점에 전력 투입 불가능** |
| 조직 구조 | 코어팀 12명 단일 조직 → 병목 우려 (미팅록 분석) | 에이전틱 알파앱스는 세팅·운영·성장 3축. **1팀이 3축을 동시에 할 수 없음** |
| 기술 기반 | AI 리뷰 분석(NLP, 감성 분류) 존재. 하지만 "에이전트 판단 아키텍처"는 미구축 | 에이전트 인프라 0 → 1 구축이 필요 |
| 제품 완성도 | 알파애즈 미개발, 통합 대시보드 미존재, 공통 온보딩 미존재 | Phase 0 인프라부터 만들어야 함 |

### 의견: "전사 올인"은 시기상조. "전사 정렬 + 선발대 투입"이 현실적

**Q3 투자 구조 제안:**

```
선발대 (3~4명, 풀타임):
├── PM 1명: Phase 0+1 스코프 오너
├── 백엔드 1명: 공통 온보딩 + 에이전트 인프라 기초
├── 프론트엔드 1명: 통합 대시보드 + 진단 카드 UI
└── AI/ML 1명: 리뷰 자동 답변 + 에스컬레이션 룰 엔진

지원군 (기존 팀, 파트타임 20%):
├── 알파리뷰팀: 리뷰 자동화 API 제공
├── 코어팀: 카페24 인증 통합, 온보딩 레이어
└── 디자인: 진단 카드 + 대시보드 UX
```

### 투자 결정을 위한 선행 조건 (6월 말까지)

| # | 선행 조건 | 왜 필요한가 |
|---|-----------|-------------|
| 1 | **Phase 1 스코프 확정** (질문 2) | 스코프 없이 OKR을 쓸 수 없음 |
| 2 | **B세그먼트 고객 5곳 인터뷰** | "진짜 이게 필요한지" 검증. 컨셉 문서는 대표님 구상이지 고객 검증이 아님 |
| 3 | **기술 PoC: 리뷰 자동 답변 품질** | 에이전트 판단 정확도가 70% 미만이면 Phase 1 자체를 재설계해야 함 |
| 4 | **조직 구조 결정** (코어팀 미팅 후속) | 선발대 3~4명을 어디서 빼올 것인가. 기존 제품 로드맵과의 충돌 해소 |

### Q3 OKR 제안 (선발대 수준)

| Objective | Key Result |
|-----------|------------|
| 에이전틱 알파앱스 Phase 0+1 MVP 출시 | KR1: 자동 세팅 → 10분 온보딩 프로토타입 완성 (8월 말) |
| | KR2: 진단 카드 → B세그먼트 10곳 파일럿 피드백 수집 (9월) |
| | KR3: 리뷰 자동 답변 정확도 80%+ 달성 (내부 테스트 기준) |

전사 올인은 **이 3개 KR이 달성된 후 Q4에 결정**하는 것이 안전하다. Q3에 전사가 달라붙었는데 PoC 결과가 안 좋으면, 매몰 비용과 조직 사기 저하가 크다.
