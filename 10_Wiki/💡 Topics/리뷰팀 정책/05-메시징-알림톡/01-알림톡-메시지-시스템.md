# 알림톡 및 메시지 시스템

## 1. 도메인 개요

### 1.1 비즈니스 문제

알파리뷰는 Cafe24 쇼핑몰 고객에게 **카카오 알림톡**을 통해 리뷰 작성을 요청하는 시스템을 운영한다. 상품 배송 상태에 따라 적절한 시점에 리뷰 작성 요청 메시지를 자동 발송하며, 비회원 포인트 지급 안내, 리마인더 발송, 서비스 온보딩 알림 등 다양한 유형의 알림톡을 관리한다.

핵심 기능은 다음과 같다:
- 배송 상태 변경 시 자동으로 리뷰 작성 요청 알림톡 발송
- 쇼핑몰별 발송 시점/리마인더/발송 제한 설정 관리
- 카카오 비즈메시지(알림톡) 및 SMS 대체 발송(fallback) 지원
- 커스텀 템플릿을 통한 업체별 맞춤 메시지 발송
- 발송 로그 관리, 재발송, 예약 취소 기능
- 부정리뷰 분석 리포트 발송 (일간/주간)

### 1.2 핵심 개념 및 용어 정의

| 용어 | 설명 |
|------|------|
| **알림톡** | 카카오 비즈메시지를 통해 발송되는 알림 메시지. 카카오톡 수신 실패 시 SMS로 대체 발송된다. |
| **비즈톡** | 카카오 알림톡 발송을 위한 외부 API 서비스. 메시지 전송 및 결과 조회를 담당한다. |
| **발신 프로필(Sender Profile)** | 알림톡 발송 시 표시되는 카카오톡 채널 프로필. 쇼핑몰별 자체 프로필 또는 알파리뷰 공용 프로필을 사용한다. |
| **발신 프로필 그룹** | 여러 쇼핑몰의 발신 프로필을 하나의 그룹으로 묶어 기본 템플릿을 공유하는 구조. |
| **템플릿 검수** | 카카오 정책에 따라 알림톡 템플릿은 사전 검수(승인)를 거쳐야 발송 가능하다. |
| **chip (치환 변수)** | 메시지 템플릿 내 `#{변수명}` 형태로 정의된 동적 데이터 삽입 지점. 예: `#{고객명}`, `#{상품명외n건}`, `#{url}` |
| **빠른리뷰** | 만족도 버튼(만족/보통/별로)을 통해 별점을 즉시 선택하여 작성하는 간편 리뷰 유형. |
| **일반리뷰** | '리뷰 작성하기' 버튼을 통해 텍스트/사진/영상 등을 포함하여 작성하는 일반적인 리뷰 유형. |
| **리마인더** | 첫 번째 리뷰 요청 이후 일정 기간이 지난 뒤 재차 보내는 리뷰 작성 독려 메시지. |
| **message_check** | 주문 아이템(OrderItem)에 저장되는 10자리 비트 플래그 문자열. 각 자릿수가 특정 발송 단계의 완료 여부를 나타낸다. |
| **shop_id=1** | 알파리뷰 기본(공용) 프로필. 쇼핑몰 자체 프로필 사용이 불가능한 경우 fallback으로 사용된다. |

## 2. 데이터 모델

### 2.1 AlimtalkMessage (쇼핑몰 메시지 템플릿)

쇼핑몰별 알림톡 메시지 템플릿을 관리하는 모델이다. 하나의 쇼핑몰은 여러 유형의 템플릿을 보유한다.

| 필드 | 타입 | 설명 |
|------|------|------|
| `shop_id` | IntegerField | 쇼핑몰 ID |
| `type` | CharField(100) | 메시지 템플릿 유형 (`AlimtalkMessage__Type` 참조) |
| `message_kakao` | TextField | 카카오톡 메시지 내용 (치환 변수 포함) |
| `message_sms` | TextField | SMS 대체 메시지 내용 (치환 변수 포함) |
| `use` | BooleanField | 사용 여부 (기본값: True) |
| `target` | CharField(200) | 리뷰 대상. 기본값 `"all"`. **Deprecated 예정** |
| `check_at` | DateTimeField | 상태 변경 일시 (auto_now) |
| `inspection_status` | CharField(20) | 템플릿 검수 상태 (`AlimtalkMessage__InspectionStatus` 참조) |
| `template_code` | CharField(50) | 카카오 템플릿 코드 |
| `template_name` | CharField(100) | 템플릿 이름 |

**인덱스**: `[shop_id]`

### 2.2 AlimtalkSetting (쇼핑몰 메시지 설정)

쇼핑몰별 알림톡 발송 정책을 관리하는 모델이다. `shop_id`에 대해 unique 제약 조건이 있어 쇼핑몰당 1개의 설정만 존재한다.

| 필드 | 타입 | 설명 | 기본값 |
|------|------|------|--------|
| `shop_id` | IntegerField (unique) | 쇼핑몰 ID | - |
| `is_alimtalk_active` | BooleanField | 첫 번째 리뷰 요청 알림톡 사용 여부 | `True` |
| `is_non_member_active` | BooleanField | 비회원 회원가입 유도 알림톡 사용 여부 | `True` |
| `is_reminder_active` | BooleanField | 리뷰 작성 리마인더 사용 여부 | `False` |
| `auto_delivered_at` | IntegerField | 발송 시점 구분 | `3` |
| `reminder_date` | IntegerField | 리마인더 발송까지의 일수 (day 단위) | `30` |
| `send_mail_limit` | IntegerField | 1일 최대 발송 메시지 개수 | `1` |
| `send_mail_limit_type` | IntegerField | 1일 최대 발송 제한 유형 | `모두전송(100)` |

**인덱스**: `[shop_id]`

#### auto_delivered_at 발송 시점 규칙

| 값 | 의미 |
|----|------|
| `-1` | 배송완료 시점에 발송 |
| `0` | 배송대기 시점에 발송 |
| `1 이상` | 배송중 상태 전환 후 N일 뒤 발송 |

기본값은 `3`으로, 배송중 상태 전환 후 3일 뒤에 알림톡이 발송된다.

### 2.3 AlimtalkMessageLog (쇼핑몰 메시지 로그)

실제 발송된 알림톡 메시지의 이력을 기록하는 모델이다.

| 필드 | 타입 | 설명 |
|------|------|------|
| `shop_id` | IntegerField | 쇼핑몰 ID |
| `customer_id` | IntegerField | 고객 고유 ID |
| `customer_member_id` | CharField(200) | 쇼핑몰 내 고객 아이디 |
| `customer_name` | CharField(200) | 고객 이름 |
| `customer_phone` | CharField(50) | 고객 전화번호 (숫자만 저장) |
| `msgid` | CharField(20) | 비즈메시지 일련번호 |
| `log_no` | IntegerField | 로그 번호 (shop 단위 순번) |
| `send_type` | IntegerField | 메시지 유형 (`AlimtalkMessageLog__SendType` 참조) |
| `text` | CharField(500) | 실제 발송된 메시지 내용 |
| `message_type` | CharField(20) | 메시지 발송 상태/채널 (`AlimtalkMessageLog__MessageType` 참조) |
| `resend` | IntegerField | 재발송된 알림톡의 로그 번호 (재발송한 적 없으면 null) |
| `data` | TextField | 발송 요청 시 사용한 입력 데이터 (JSON) |
| `response` | TextField | 비즈톡 API 응답 데이터 |
| `created_at` | DateTimeField | 로그 생성일 |
| `cancel_at` | DateTimeField | 예약 취소일 |
| `sending_at` | DateTimeField | 실제 발송 완료일 |
| `reserved_at` | DateTimeField | 예약 발송 예정일 |

**인덱스**: `[shop_id, log_no]`, `[shop_id, send_type]`

**저장 시 비즈니스 로직**: `save()` 메서드에서 `customer_phone` 필드의 비숫자 문자를 모두 제거하여 숫자만 저장한다.

### 2.4 AlimtalkMessageLogOrderItemRelation (알림톡-주문 아이템 연결)

알림톡 메시지 로그와 주문 아이템 간의 다대다 관계를 관리하는 연결 테이블이다. 하나의 알림톡 메시지가 여러 주문 아이템과 관련될 수 있다.

| 필드 | 타입 | 설명 |
|------|------|------|
| `message` | ForeignKey(AlimtalkMessageLog) | 알림톡 메시지 로그 (related_name: `order_item_set`) |
| `order_item` | ForeignKey(OrderItem) | 주문 아이템 (related_name: `message_set`) |
| `send_type` | IntegerField | 메시지 유형 (AlimtalkMessageLog의 send_type 중복 저장) |
| `created_at` | DateTimeField | 생성일시 (auto_now_add) |
| `updated_at` | DateTimeField | 수정일시 (auto_now) |

**제약조건**: `(message, order_item)` 쌍에 대한 UniqueConstraint.

### 3.2 템플릿 검수 상태 (InspectionStatus)

카카오 정책에 따른 템플릿 검수 상태 흐름이다:

| 상태 | 코드 | 설명 |
|------|------|------|
| 검수요청_이전 | `None` | 아직 검수를 요청하지 않은 상태 |
| 검수요청_등록 | `REG` | 검수 요청이 등록된 상태 |
| 검수요청_심사요청 | `REQ` | 심사가 요청된 상태 |
| 검수요청_승인 | `APR` | 검수 승인 완료 (발송 가능) |
| 검수요청_반려 | `REJ` | 검수 반려 (재수정 필요) |

**중요 규칙**: `inspection_status`가 `APR`(승인)이 아닌 템플릿은 발송에 사용할 수 없으며, 이 경우 알파리뷰 기본(공용) 프로필(shop_id=1)의 템플릿으로 대체된다.

### 3.3 메시지 발송 유형 (SendType)

`AlimtalkMessageLog__SendType` 열거형이다:

| 유형 | 값 | 코드 | 설명 |
|------|----|------|------|
| 첫번째요청 | `100` | `first_send` | 최초 리뷰 작성 요청 |
| 리마인더 | `200` | `reminder` | 리뷰 작성 리마인더 재요청 |
| 회원가입 | `300` | `nonmember` | 비회원 회원가입 유도 |

### 3.4 메시지 발송 상태/채널 (MessageType)

`AlimtalkMessageLog__MessageType` 열거형이다:

| 상태 | 코드 | 설명 |
|------|------|------|
| 예약발송 | `R` | 예약 발송 대기 중 |
| 메시지발송 | `M` | SMS로 대체 발송됨 |
| 카카오톡발송 | `K` | 카카오톡으로 발송됨 |
| 예약발송취소 | `C` | 예약 발송이 취소됨 |
| 발송실패 | `F` | 발송 실패 |

### 3.5 발송 제한 유형 (SendMailLimitType)

`AlimtalkSetting__SendMailLimitType` 열거형이다:

| 유형 | 값 | 설명 |
|------|----|------|
| 모두전송 | `100` | 발송 제한 없이 모든 대상에게 전송 |
| 알림톡종류구분 | `200` | 고객당 알림톡 종류별로 `send_mail_limit` 적용 (by_type) |
| 알림톡종류무관 | `300` | 고객당 알림톡 종류 무관하게 `send_mail_limit` 적용 (by_all) |

### 3.6 메시지 타입 결정 규칙

`AlimtalkMessage__Feature.get_message_type()` 메서드의 규칙:

- 다음 유형은 **텍스트 알림톡 (`AT`)** 으로 발송된다:
  - `비회원_포인트지급`, `서비스시작_무료체험O`, `서비스시작_무료체험X`, `온보딩이탈`, `온보딩완료`, `부정리뷰_일간리포트`, `부정리뷰_주간리포트`
- 그 외 모든 리뷰 요청 유형은 **이미지 알림톡 (`AI`)** 으로 발송된다.

### 3.7 템플릿 선택 로직 (발신 프로필 Fallback)

`AlimtalkMessage__Feature.get_alimtalk_message()` 메서드의 규칙:

1. 해당 `shop_id`와 `type`에 해당하는 가장 최신 템플릿(`-id` 정렬)을 조회한다.
2. 템플릿이 존재하지 않거나 `use=False`이면 `None`을 반환한다.
3. `shop_id=1`(알파리뷰 기본 프로필)인 경우, 가장 오래된 기본 템플릿을 반환한다.
4. 일반 쇼핑몰의 경우 다음 조건을 **모두** 만족해야 해당 쇼핑몰의 템플릿이 사용된다:
   - 카카오 발신 프로필 상태(`kakao_profil_status`)가 `정상(A)`
   - 카카오 채널 상태(`kakao_channel_status`)가 `활성화(A)`
   - 템플릿 검수 상태(`inspection_status`)가 `검수요청_승인(APR)`
5. 위 조건 중 하나라도 실패하면 **알파리뷰 기본 프로필(shop_id=1)** 의 템플릿으로 대체한다.

### 3.8 치환 변수 (Chip) 시스템

메시지 템플릿은 `#{변수명}` 형태의 치환 변수를 포함하며, 발송 시 실제 데이터로 교체된다.

| 치환 변수 | 카카오톡 처리 | SMS 처리 |
|-----------|---------------|----------|
| `#{쇼핑몰명}` | 원본 그대로 | 단축 URL 사용 시 9자 제한, 미사용 시 6자 제한. 초과분은 `-`으로 처리 |
| `#{고객명}` | 고객 이름 (없으면 "고객") | 동일 |
| `#{상품명외N건}` / `#{상품명외n건}` / `#{리뷰x상품명}` | 첫 번째 상품명 + "외 N건" | 상품명 10자 초과 시 `..` 처리. 99건 초과 시 "외 99+건" |
| `#{송장번호}` / `#{운송장번호}` | "택배사명 : 운송장번호" | 동일 |
| `#{url}` | 원본 URL | 단축 URL (실패 시 원본 URL) |
| `#{지급혜택}` | 포인트 금액 (9만 초과 시 "9만 ↑") | 동일 |
| `#{지급신청바로가기}` | 원본 URL | 단축 URL |
| `#{지급사유}` | "리뷰 혜택 지급" | 동일 |
| `#{SNS리뷰작성가능여부}` | SNS/동영상 리뷰 작성 가능 여부에 따른 안내 문구 + URL | 동일 (단축 URL 사용) |
| `#{주문번호}` | 주문 아이템 플랫폼 ID (16자 제한) | - |
| `#{주문목록바로가기}` | 원본 URL | 단축 URL |
| `#{남은혜택}` | 빈 문자열 | 빈 문자열 |

#### URL 구조

| 용도 | URL 패턴 |
|------|----------|
| 리뷰 작성 페이지 | `{REVIEW_WRITE_PAGE_HOST}/l{customer.short_code}` |
| SNS 리뷰/주문목록 | `{REVIEW_WRITE_PAGE_HOST}/o{customer.short_code}` |
| 포인트 지급 신청 | `{REVIEW_WRITE_PAGE_HOST}/a{order.short_code}` |
| 리뷰 작성 (버튼) | `{REVIEW_WRITE_PAGE_HOST}/w{order_item.short_code}` |
| 배송 조회 | `m.search.naver.com/search.naver?query={택배사명}{운송장번호}` |

#### URL 단축 처리

1. 코어 서비스의 URL 단축 API를 호출하여 단축 URL을 생성한다.
2. URL 단축 실패 시 원본 URL을 그대로 사용한다 (fallback).
3. 원본 URL 사용 시 SMS 글자 수 확보를 위해 쇼핑몰명 제한 길이가 9자에서 6자로 줄어든다.

### 3.9 예약 발송 시간 규칙

`AlimtalkMessage__Feature.change_reserved_time()` 메서드의 규칙:

- 현재 시각 기준 **19시 이전**이면: **당일 19시 + 랜덤 분/초**에 발송 예약
- 현재 시각 기준 **19시 이후**이면: **다음날 19시 + 랜덤 분/초**에 발송 예약
- 랜덤 분/초를 사용하여 동시에 대량 발송되는 것을 방지한다.

### 3.10 발송 제한 및 조건

#### 알림톡 비활성화 업체

`ALIMTALK_DISABLED_SHOP` 리스트에 등록된 쇼핑몰은 알림톡 발송이 완전히 차단된다.

#### 메시지 중복 발송 방지

`AlimtalkMessageLog__Feature.check_exist_duplicate_message_log()` 메서드의 규칙:

- **같은 날짜**에 **동일한 메시지 내용**, **동일한 고객명**, **동일한 전화번호**로 이미 발송된 로그가 존재하면 중복으로 판단하여 발송하지 않는다.
- 단, `cancel_at`이 설정된(취소된) 알림톡 로그는 중복 판단에서 제외한다.

#### message_check 비트 플래그

`OrderItem__MessageCheckType`은 주문 아이템 단위로 메시지 발송 이력을 10자리 비트 플래그 문자열(`"0000000000"`)로 관리한다:

| 위치 (인덱스) | 의미 | 적용 메서드 |
|---------------|------|-------------|
| 0 | 주문 직후 처리 완료 | `apply_after_order` |
| 1 | 송장 입력 직후 처리 완료 | `apply_after_invoice` |
| 2 | 배송 완료 알림톡 발송 완료 | `apply_after_delivered_alimtalk` |
| 3 | 비회원 알림톡 발송 완료 | `apply_after_nonmember_alimtalk` |
| 4 | SNS 리뷰 혜택 제공 완료 | `apply_sns_review_rewarded` |
| 8 | 리마인더 알림톡 발송 완료 | `apply_reminder` |

- `"0000000000"`: 전체 발송 가능 (초기 상태)
- `"1111111111"`: 전체 발송 불가 (모든 단계 완료 또는 테스트용)

#### 050 안심번호 필터링

`cancel_050_reserved_alimtalk` 관리 명령어를 통해, `050`으로 시작하는 안심번호로 예약된 알림톡을 일괄 취소할 수 있다. 안심번호는 실제 고객 전화번호가 아니므로 발송 실패 가능성이 높다.

#### 발송 불가 발신번호

특정 발신번호(`025531008`, `07050662122` 등)는 SMS 발송에서 제외된다.

### 3.11 커스텀 업체 템플릿

특정 쇼핑몰은 기본 템플릿 대신 맞춤형 템플릿을 사용한다.

- `ALIMTALK_MESSAGE_CUSTOM_SHOP` 리스트에 등록된 쇼핑몰 ID가 대상이다.
- `CustomAlimtalkMessageQuickReview__Type`: 커스텀 빠른리뷰 템플릿 유형
- `CustomAlimtalkMessage__Type`: 커스텀 일반리뷰 템플릿 유형
- `CUSTOM_ALIMTALK_TEMPLATES_RAW` 딕셔너리에 커스텀 템플릿 원본이 정의된다.
- 버튼 생성 시 `get_buttons_data()`에서 커스텀 업체 여부를 판단하여 적절한 템플릿 소스를 선택한다.

### 3.12 버튼 유형

알림톡 메시지에 포함되는 버튼은 다음과 같다:

| 버튼 유형 | 설명 | URL 치환 변수 |
|-----------|------|---------------|
| 만족도 버튼 (빠른리뷰) | 만족/보통/별로 3단계 | `#{write_url}/5`, `#{write_url}/3`, `#{write_url}/1` |
| 리뷰 작성하기 (일반리뷰) | 리뷰 작성 페이지 이동 | `#{write_url}` |
| 배송 조회 | 네이버 배송 조회 | `#{tracking_url}` |
| 자세히보기 (비회원 포인트) | 포인트 지급 신청 페이지 | `#{benefit_url}` |

모든 버튼은 `WL` (웹 링크) 타입이며, `url_mobile`과 `url_pc`를 동일하게 설정한다.

### 3.13 알림톡 발송 배치 처리

`AlimtalkMessage__Feature.request_send_message_list()` 메서드의 규칙:

- 메시지 리스트를 **100개씩 분할**하여 비즈톡 API에 전송한다.
- 각 배치는 단건으로 나누어 POST 요청을 보내며, 요청마다 0.01초의 sleep을 추가한다.
- 전송 실패 시 예외를 캐치하고 Datadog 로그를 기록한다.

### 3.14 개인정보 마스킹

`AlimtalkMessageLog__ListSerializer`의 로그 목록 조회 시:

- 쇼핑몰의 `is_used_personal_data_masking` 설정이 활성화된 경우, `customer_name`과 `customer_phone`에 마스킹 처리를 적용한다.
- `Common__MaskUtil.mask_name()` 및 `Common__MaskUtil.mask_phone_number()` 유틸리티를 사용한다.

## 4. 상태 전이

### 4.1 메시지 발송 상태 전이 (MessageType)

```
[발송 요청] ──> 예약발송(R)
                  │
                  ├──> 카카오톡발송(K)  ← 카카오톡으로 정상 발송됨
                  │
                  ├──> 메시지발송(M)    ← SMS로 대체 발송됨
                  │       │
                  │       └──> 발송실패(F)  ← SMS 발송 실패 (코드 != M000)
                  │
                  └──> 예약발송취소(C)  ← 사용자 또는 시스템에 의해 취소됨
```

상태 전이 규칙:
- 최초 발송 요청 시 `예약발송(R)` 상태로 생성된다.
- 1시간 주기 배치(`every_1hour_kakao_message_refresh`)에서 비즈톡 API를 조회하여 실제 전송 결과를 반영한다.
- SMS 발송 시 응답 코드가 `M001`이면 이통사 결과 대기 중이므로 `sending_at`을 다음 사이클로 넘긴다.
- SMS 발송 시 응답 코드가 `M000`이 아니면 `발송실패(F)`로 전환한다.

### 4.2 템플릿 검수 상태 전이 (InspectionStatus)

```
검수요청_이전(None) ──> 검수요청_등록(REG) ──> 검수요청_심사요청(REQ)
                                                      │
                                                      ├──> 검수요청_승인(APR)  ← 발송 가능
                                                      │
                                                      └──> 검수요청_반려(REJ)  ← 재수정 필요
```

- 발신 프로필 그룹에 등록 시, 기본 템플릿은 그룹의 이미 승인된 상태를 상속받아 자동으로 `검수요청_승인(APR)`으로 설정된다.

```
    ├──5. shop_id=1의 빠른리뷰_배송완료 템플릿으로 메시지 생성
    │
    └──6. 비즈톡 API를 통해 알림톡 발송
```

### 5.8 알림톡 로그 목록 조회

```
[GET /api/v1/shops/{shop_id}/alimtalk/logs]
    │
    ├──1. 필터 파라미터 처리
    │     ├── send_type: 메시지 유형 필터
    │     ├── start_date / end_date: 생성일 기간 필터
    │     └── search: 고객 이름/전화번호 검색 (OR 조건)
    │           └── 숫자+하이픈 포함 시 하이픈 제거 (전화번호 검색 최적화)
    │
    ├──2. log_no 내림차순 정렬
    │
    ├──3. 페이지네이션 적용 (PageNumberCustomPagination)
    │
    └──4. 쇼핑몰 개인정보 마스킹 설정 확인
          └── is_used_personal_data_masking=True이면 이름/전화번호 마스킹
```

### 5.9 알림톡 설정 조회/수정

```
[GET /api/v1/shops/{shop_id}/alimtalk/setting]
    ├── selected_fields 파라미터로 조회 필드 선택 가능 (쉼표 구분)
    └── DynamicFieldsModelSerializer를 사용하여 선택적 필드 반환

[PATCH /api/v1/shops/{shop_id}/alimtalk/setting]
    ├── 유효성 검증:
    │     ├── auto_delivered_at: 최소값 -1
    │     ├── reminder_date: 최소값 1 (일 단위)
    │     └── send_mail_limit: 최소값 1
    └── audit_log 데코레이터를 통한 변경 이력 기록 (AlimtalkSettingLogFormatter)
```
