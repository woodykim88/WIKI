# 통계 및 리포트 시스템

## 1. 도메인 개요

통계 및 리포트 시스템은 알파리뷰 플랫폼에서 수집된 리뷰 데이터를 집계/분석하여 쇼핑몰 운영자에게 인사이트를 제공하는 도메인이다. 크게 세 가지 하위 시스템으로 구성된다.

1. **대시보드 통계 시스템 (`ShopStatDaily`)** - 쇼핑몰 전체 리뷰/주문/적립금 등의 일간 통계를 집계하여 대시보드 그래프, 스코어카드, 랭킹 등의 UI를 지원한다.
2. **VoC 리포트 시스템 (`ReportStatDaily` + `ReviewAnalysisReport`)** - 알파리뷰에서 작성된 리뷰에 한정하여 부정 리뷰(VoC) 중심의 통계를 별도로 집계하고, 주간/일간 리포트를 카카오톡 알림톡 또는 슬랙으로 자동 발송한다.
3. **AI 리뷰 분석 시스템 (`ReviewAnalysisConfig`, `ReviewPhrase`, `ReviewPhraseTagRelation`)** - 리뷰 내용을 AI가 문장 단위(phrase)로 분해하고, 태그를 부여하여 VoC를 체계적으로 분류한다. 태그 분석 결과는 대시보드 및 Pre-sales 리포트(CSV)로 활용된다.

### 핵심 용어

| 용어 | 설명 |
|------|------|
| **ShopStatDaily** | 쇼핑몰 전체 일간 통계 로그. 대시보드 그래프/스코어카드에 사용 |
| **ReportStatDaily** | 알파리뷰 작성 리뷰 한정 일간 통계. VoC 리포트 메시지 구성에 사용 |
| **ReviewAnalysisReport** | 리포트 수신 설정 엔티티 (채널, 타입, 수신번호 등) |
| **ReviewAnalysisReportMessage** | 리포트 발송용 메시지 페이로드 저장 엔티티 |
| **VoC (Voice of Customer)** | 부정 리뷰 가운데 의미 있는 사유가 담긴 리뷰 |
| **ReviewPhrase** | AI가 리뷰 본문에서 추출한 개별 문장(구문) |
| **ReviewTag / ReviewPhraseTagRelation** | 구문에 부여된 태그 및 그 매핑 관계 |
| **ReviewNegativePhrase** | 부정 문장으로 판별된 리뷰의 부정 구문 |

## 3. 비즈니스 규칙

### 3.1 대시보드 일일 통계 집계 (`ShopStatDaily`)

**집계 시점**: 매일 00시 크론 작업 (`daily_00am_logging_stat_daily`)

**집계 대상**: 코어 서비스에 등록된 모든 쇼핑몰

**집계 규칙**:

1. **마지막 생성 데이터 확인**: 가장 최근 `ShopStatDaily.created_at`를 조회하여, 그 다음 날부터 오늘까지의 데이터를 생성한다.
2. **초기 데이터 없는 경우**: 기존 데이터가 없으면 오늘 기준 5일 전부터 생성한다.
3. **리뷰 필터링**: `creater_type`이 관리자/복사/원본인 리뷰는 제외한다.
4. **리뷰 유형별 카운트**: 전체 리뷰, 알파리뷰(플랫폼=알파리뷰 AND order_item 존재), 빠른/일반/SNS, 포토/영상/설문, 회원/비회원 리뷰를 각각 집계한다.
5. **부정 리뷰 통계**: `review_type=일반`, `review_origin=null(원본)`, `is_deleted=False` 조건의 리뷰 중 `review_negative_set`이 존재하는 리뷰를 부정 리뷰로 카운트한다. 관리자/복사 리뷰는 제외한다.
6. **주문/적립금**: 해당 일자 주문 수, 주문 상품 수, 환불 상품 수, 지급 포인트 합계를 집계한다.
7. **Bulk 생성**: 집계 결과를 `bulk_create`로 일괄 저장한다.

**캐싱 전략**: 대시보드 조회 시 캐시를 12시간(`cache_expiry = 60 * 60 * 12`) TTL로 저장한다. 캐시 키는 `stats_graph_{shop_id}_{interval}_{attr}_{start}_{end}` 형식이다.

### 3.2 VoC 리포트 통계 집계 (`ReportStatDaily`)

**집계 시점**: 매일 00시 크론 작업 (`daily_00am_create_report_stat_daily`)

**집계 대상**: `ReviewAnalysisReport.use=True`이면서 채널이 등록된(카카오톡 전화번호 또는 슬랙 채널이 존재하는) 업체만

**집계 규칙**:

1. **기존 데이터 존재 시**: 마지막 생성 일자의 다음 날부터 오늘까지 생성한다. 단, 너무 오래된 데이터(지난주 월요일 이전)인 경우 지난주 월요일부터 생성한다.
2. **기존 데이터 미존재 시**: 지난주 월요일부터 오늘까지 생성한다.
3. **리뷰 범위**: `platform=알파리뷰`, `order_item_fk` 존재, `review_origin=null(원본)`, `is_deleted=False`, `creater_type`이 관리자/복사가 아닌 리뷰만 대상으로 한다.
4. **중복 방지**: 동일 `(shop_id, date)`에 이미 데이터가 있으면 생성하지 않는다.
5. **평점 집계**: 빠른+일반 리뷰에 대해서만 평점 합계 및 1~5점별 리뷰 수를 집계한다. SNS 리뷰는 평점 집계에서 제외된다.
6. **부정 리뷰 집계**: 일반 리뷰 중 `review_negative_set`이 존재하는 리뷰를 부정 리뷰로 카운트한다.

### 3.3 리포트 설정 관리 규칙

**최대 수신번호 제한**: 각 리포트 타입별로 카카오톡 수신번호는 **최대 3개**까지 설정 가능하다. 초과 시 `EXCEED_KAKAOTALK_PHONE_LIMIT` 에러가 발생한다 (HTTP 409).

**수신번호 중복 방지**: 동일 리포트 타입 내에서 카카오톡 수신번호 중복 저장이 불가하다. DB UniqueConstraint `(shop_id, report_type, kakaotalk_phone)` 위반 시 `DUPLICATE_KAKAOTALK_PHONE` 에러가 발생한다 (HTTP 409).

**슬랙 채널 등록**: 슬랙 채널을 등록/수정할 때마다 코어 서비스의 `shops/{shop_id}/slack/join-channel` API를 호출하여 알파앱스 앱을 해당 채널에 등록한다. 등록 실패 시 `FAIL_JOIN_CHANNEL` 에러가 발생한다 (HTTP 500).

**Bulk 처리**: 설정 수정(PATCH)은 생성/수정/삭제를 한 트랜잭션 내에서 bulk 처리한다.
- ID가 없는 데이터: 생성
- 기존에 존재하는 ID: 수정
- 요청에 포함되지 않은 기존 ID: 삭제

### 3.4 VoC 리포트 메시지 생성 규칙

**생성 시점**: 매일 00시 크론 작업 (`daily_00am_create_report_message`)

**리포트 종류별 규칙**:

| 리포트 타입 | 생성 주기 | 기준 기간 | 리포트 기준 일자 |
|-------------|-----------|-----------|------------------|
| 주간리포트 (100) | **매주 월요일만** | 지난주 월~일 (7일) | 지난주 일요일 |
| 일간리포트 (200) | **매일** | 어제 하루 | 어제 |

**메시지 생성 규칙**:

1. `ReviewAnalysisReport.use=True`이면서 채널이 등록된 업체가 대상이다.
2. 업체(shop_id) + 리포트 타입(report_type)별로 그룹화하여 처리한다.
3. `ReportStatDaily` 데이터에서 통계 치환 변수(chip)를 연산한다.
4. **리뷰 0건 스킵**: 해당 기간에 알파리뷰 리뷰(`review_alph`)가 0건이면 리포트를 생성하지 않는다.
5. **중복 방지**: 동일 `(shop_id, report_date, report_fk_id)`에 이미 메시지가 존재하면 생성하지 않는다.
6. 수신 채널별로 메시지 페이로드를 구성한다:
   - **카카오톡(알림톡)**: 템플릿 코드 기반 메시지, 수신번호별 UUID msgid 생성, 즉시 전송 설정 (`reserved_time: "00000000000000"`)
   - **슬랙**: Block Kit 형식 메시지, 채널 ID별 블록 데이터 구성

**통계 치환 변수 연산 (`calculate_common_metrics`)**:

| 치환 변수 | 연산 공식 |
|-----------|-----------|
| `review_alph_media_sum` | 포토 리뷰 수 + 영상 리뷰 수 |
| `review_alph_low_ratings_sum` | 평점 1점 + 2점 + 3점 리뷰 수 |
| `review_alph_ratings_avg` | 누적 평점 합계 / (빠른 리뷰 수 + 일반 리뷰 수) |
| `review_alph_negative_ratio` | (부정 리뷰 수 / 일반 리뷰 수) * 100 |

### 3.5 VoC 리포트 메시지 발송 규칙

**발송 시점**: 매일 09시 크론 작업 (`daily_09am_send_report_message`)

**발송 규칙**:

1. **어제 날짜**에 해당하면서 `sending_at=null`(미발송)인 메시지가 대상이다.
2. 업체(shop_id) + 메시지 타입(알림톡/슬랙)별로 그룹화하여 코어의 알람센터에 발송한다.
3. 알람센터 이벤트 타입: `AlarmEvent__Type.부정리뷰알람 (130)`
4. 발송 완료 후 `sending_at`에 현재 시각을 기록한다.

### 3.6 VoC 인사이트 규칙

**주간리포트 인사이트**:
- **VoC 비율이 높은 상품 TOP3**: 지난주 기간 내 부정 리뷰 비율이 높은 상품 3개를 추출한다.
- **VoC 비율이 낮은 상품 TOP3**: 동일 기간 내 부정 리뷰 비율이 낮은 상품 3개를 추출한다.
- 상품명이 30자를 초과하면 30자 + "..." 으로 잘라서 표시한다.

**일간리포트 인사이트**:
- **부정 리뷰 TOP3**: 어제 작성된 부정 리뷰 중 상위 3건의 내용을 표시한다.
- 리뷰 내용이 100자를 초과하면 100자 + "..." 으로 잘라서 표시한다.

### 3.7 대시보드 통계 표시 자동 ON/OFF 규칙

**실행 시점**: 매월 1일, 15일 크론 작업 (`monthly_1st15th_set_stat_daily`)

**규칙**: 쇼핑몰의 `stat_simple_status`가 `AUTO`인 경우에만 동작한다.
- 최근 1개월간 리뷰 작성률이 3% 이하이거나, 알파리뷰 리뷰 수가 50건 미만이면 `stat_simple=False` (대시보드 통계 비표시)
- 그 외에는 `stat_simple=True` (대시보드 통계 표시)

### 3.8 AI 리뷰 분석 규칙

**VOC 분석 흐름** (현재 주석 처리 - SP-008 이후 별도 서비스로 분리됨):

1. 리뷰 본문을 AI(OpenAI)에게 전달하여 문장(phrase) 목록을 추출한다.
2. 각 phrase에 대해 부정 여부(`is_negative`)를 판별한다.
3. phrase에 대해 태그를 분류한다.
   - **기본 태그(default)**: 이슈(파손, 불량 등), 제품특징(기능, 디자인 등), 기타의견(CS, UX 등)
   - **커스텀 태그**: 쇼핑몰별 설정한 태그
4. 결과를 `ReviewPhrase`, `ReviewPhraseTagRelation` 테이블에 저장한다.

**태그 상태 관리** (`TagStatus`):
- `INACTIVE (100)`: 비활성
- `ACTIVE (200)`: 활성
- `DELETING (300)`: 삭제 진행중

**배치 분석** (`BatchAnalysisRequest`):
- 여러 리뷰를 한 번에 분석 요청할 수 있다.
- 상태: `PROCESSING (100)` -> `COMPLETED (200)`
- `finish_count`로 완료 진행률을 추적한다.

**AI 중복 이미지 검사** (`every_15min_ai_process`):
- 15분마다 SQS 큐에서 메시지를 수신하여 AI 처리를 수행한다.
- dHash 알고리즘으로 이미지 해시를 계산하고, 해밍 거리(threshold=5) 이내이면 중복으로 판정한다.
- 중복 이미지가 발견되면 리뷰의 게시/보상 상태를 조정한다.

## 4. 상태 전이

### 4.1 리포트 메시지 발송 상태

```
[생성됨]
  sending_at = null
       |
       | daily_09am_send_report_message 실행
       v
[발송 완료]
  sending_at = 발송 시각
```

### 4.2 배치 분석 요청 상태 (`BatchAnalysisRequest`)

```
[PROCESSING (100)]   요청 생성 시 초기 상태
       |
       | finish_count 증가 (개별 리뷰 분석 완료 시)
       |
       v
[COMPLETED (200)]   target_review_count == finish_count 달성 시
```

### 4.3 개별 리뷰 분석 상태 (`ReviewAnalysisLog`)

```
[요청_처리중 (100)]   분석 요청 생성 시
       |
       +---> [요청_완료 (200)]   AI 분석 성공 시 (completed_at 기록)
       |
       +---> [요청_실패 (300)]   AI 분석 실패 시 (failed_at 기록)
```

### 4.4 태그 상태 (`TagStatus`)

```
[INACTIVE (100)] <---> [ACTIVE (200)]   활성화/비활성화 토글
                           |
                           v
                   [DELETING (300)]     삭제 요청 시 (deletion_requested_at 기록)
                           |
                           v
                       [삭제됨]          실제 삭제 완료
```
