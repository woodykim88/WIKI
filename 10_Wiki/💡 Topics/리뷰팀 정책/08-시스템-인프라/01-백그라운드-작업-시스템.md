# 백그라운드 작업 및 시스템 인프라

## 1. 도메인 개요

Alpha Review 시스템은 비동기 작업 처리, 크론잡 스케줄링, ECS 태스크 기반 장기 작업, 그리고 독립 마이크로서비스 등 다층적인 백그라운드 작업 인프라를 운영한다. 이러한 구조는 리뷰 수집, AI 분석, 알림톡 발송, 데이터 동기화 등 다양한 비즈니스 요구사항을 안정적으로 처리하기 위해 설계되었다.

핵심 구성요소는 다음과 같다:

- **AsyncRunner**: SQS 기반 비동기 작업 처리 시스템 (경량 작업용)
- **크론잡 (Management Commands)**: 주기적으로 실행되는 데이터 갱신 및 트래킹 작업
- **SystemCycle (ECS Task Runner)**: AWS ECS Fargate 기반 장기 실행 작업 (리뷰 이관 등)
- **마이크로서비스**: AI 리뷰 분석, 키워드 추출, 웹 크롤링, 리뷰 영속화 등 독립 서비스

## 2. 데이터 모델

### 2.1 AsyncRunner 관련 모델

AsyncRunner는 별도의 Django 모델을 갖지 않으며, SQS 큐를 통한 메시지 기반 통신으로 동작한다. 핵심 데이터 구조는 Pydantic 모델로 정의된다.

#### AsyncRunnerPayload

| 필드 | 타입 | 설명 |
|------|------|------|
| `event_type` | `str` | 작업 유형 식별자 (레지스트리에 등록된 태스크 이름) |
| `shop_id` | `int \| None` | 대상 쇼핑몰 ID |
| `args` | `list` | 태스크 함수에 전달할 위치 인자 |
| `kwargs` | `dict` | 태스크 함수에 전달할 키워드 인자 (`shop_id` 자동 포함) |

- 정의 위치: `projectreview/asyncrunner/registry.py`

### 2.2 SystemCycle - RunnerData 모델

ECS 태스크 기반 장기 작업의 실행 데이터를 저장하는 모델이다.

#### RunnerData

| 필드 | 타입 | 설명 |
|------|------|------|
| `shop_id` | `IntegerField` | 대상 쇼핑몰 ID |
| `task_type` | `IntegerField` | 작업 유형 (`Task__Type` choices) |
| `data` | `JSONField` | 작업에 필요한 원본 데이터 |
| `created_at` | `DateTimeField` | 생성 시각 (auto_now_add) |
| `updated_at` | `DateTimeField` | 수정 시각 (auto_now) |

- 정의 위치: `projectreview/systemcycle/models.py`

### 2.3 Task__Type 열거형

| 값 | 이름 | 설명 |
|----|------|------|

## 3. 비즈니스 규칙

### 3.1 AsyncRunner 작업 처리 규칙

AsyncRunner는 AWS SQS FIFO 큐를 소비하여 비동기 작업을 처리하는 상주 프로세스이다. `AsyncBase` 기반 클래스 계층 구조로 구현되어 있다.

#### 클래스 계층 구조

```
AsyncBase__Feature (메시지 송수신)
  └── AsyncRunner__Feature (리뷰 전용 SQS 큐 연결)

AsyncBase__Registry (작업 등록 및 큐/락 관리)
  └── AsyncRunner__Registry (4개 워커, asyncrunner 로그 타입)

AsyncBase__Worker (큐 데이터 처리 워커)
  └── AsyncRunner__Worker (payload 기반 태스크 실행)
```

#### 메시지 발행 규칙
- `AsyncBase__Feature.push_message()` 메서드를 통해 SQS에 메시지를 전송한다.
- `message_group_id`는 기본값으로 `{event_type}-{shop_id}` 형식을 사용하며, FIFO 큐의 메시지 그룹 순서를 보장한다.
- `message_deduplication_id`는 기본값으로 `{event_type}-{shop_id}-{uuid4()}` 형식이며, 중복 메시지를 방지한다.
- `shop_id`는 항상 자동으로 `kwargs`에 포함된다.
- 기본 지연 시간(delay_seconds)은 60초이다.

#### 메시지 소비 규칙
- `AsyncRunner__Util.consume_messages()`가 SQS에서 최대 10개씩 메시지를 가져온다.
- 메시지 소비 후 0.5초 지연을 둔다.
- 메시지가 없으면 빈 큐와 락을 정리(clear_queues)한다.

#### 메시지 분배 규칙 (identifier 기반 병렬 처리)
- 각 메시지는 `(event_type, shop_id)` 조합을 구분자(identifier)로 사용하여 별도 큐에 분배된다.
- 동일 identifier 내 메시지는 순차적으로 처리된다 (Lock 기반).
- 서로 다른 identifier 간에는 병렬로 처리된다.
- 기본 워커 수는 4개이다 (AsyncRunner__Registry.num_workers = 4).

#### 이벤트 타입 검증
- `event_type`이 `task_registry`에 등록되지 않은 경우, 메시지를 즉시 삭제하고 건너뛴다.
- `validate_message_payload`를 통해 페이로드 유효성을 검증한다 (현재는 항상 valid 반환).

#### 작업 등록
- `@AsyncRunner__Registry.register_task("task_name")` 데코레이터를 사용하여 태스크 함수를 등록한다.
- 등록된 태스크는 `task_registry` 딕셔너리에 저장된다.

#### 대기(Waiting) 메커니즘
- 대기 메시지 수가 `waiting_limit`(300)을 초과하거나, 소수 업체(4개 미만)에서 대량 메시지를 보낸 경우 대기 모드에 진입한다.
- 대기 모드에서는 큐 크기가 절반으로 줄어들 때까지 10초 간격으로 대기한다.

#### 정리(Cleanup) 로직
- 30회 소비 주기마다 빈 큐와 미사용 락을 정리한다.
- 20분 주기로 큐 현황 로그를 Datadog에 전송한다.

#### Graceful Shutdown
- SIGTERM 수신 시 `shutdown_flag`를 True로 설정한다.
- 워커는 현재 처리 중인 메시지 완료 후 종료한다.
- shutdown 시 미처리 메시지의 SQS visibility timeout을 0으로 변경하여 즉시 재처리 가능하게 한다.

#### 에러 처리
- 워커에서 예외 발생 시 Datadog에 에러 로그를 전송하고, 메시지를 삭제하지 않는다 (SQS 재시도).
- SQS ClientError 발생 시에도 에러를 기록하고 반환한다.
- 각 워커는 처리 전 `close_old_connections()`를 호출하여 Django DB 연결을 정리한다.

### 3.2 크론잡 스케줄 및 처리

크론잡은 Django Management Command로 구현되며, ECS 스케줄러 또는 외부 스케줄러에 의해 주기적으로 실행된다. 주요 크론잡은 복합 커맨드(다수의 하위 커맨드를 순차 실행)와 단일 커맨드로 나뉜다.

#### Singleton Task Lock 메커니즘
- `@singleton_task_lock(keyname)` 데코레이터를 통해 동일한 태스크가 동시에 실행되지 않도록 보장한다.
- S3의 키 파일을 락으로 사용한다 (`S3Lock__Feature`).
- 락 획득 실패 시 "already running" 로그를 남기고 종료한다.
- 운영 환경과 개발 환경의 락 키를 구분한다 (`dev-` 접두사).
- 예외 발생 시에도 반드시 락을 해제한다 (finally 블록).

#### 복합 크론잡

##### task_1day (매일 00시)
다음 하위 작업을 순차적으로 실행한다:
1. **혜택회수** (`every_00am_refund_test`): 환불/교환/반품으로 회수예정인 리뷰의 혜택 회수
2. **뱃지 갱신** (`every_00am_badge_date_refresh`): 뱃지 조건에 맞는 리뷰-뱃지 관계 재설정
3. **VOC 리포트 통계 생성** (`daily_00am_create_report_stat_daily`): 리뷰 분석 리포트 일간 통계 생성
4. **VOC 리포트 메시지 생성** (`daily_00am_create_report_message`): 부정리뷰 리포트 알림 메시지 생성

주의: VOC 리포트의 두 작업은 반드시 순서가 보장되어야 한다 (통계 생성 후 메시지 생성).

##### task_1day_data (종일 반복)
매 5분 간격으로 반복 실행하며, 22시 초과 또는 날짜 변경 시 종료된다. 각 쇼핑몰에 대해 다음 작업을 순차 실행한다:
1. **Order Tracking** (`daily_00am_task`): 주문 상태 추적, 알림톡 발송, 혜택/리뷰 권한 처리
2. **Facebook Refresh** (`every_1hour_review_facebook`): SNS 리뷰 인증 상태 확인 및 갱신
3. **Review Upload** (`every_1hour_review_upload`): 카페24로 리뷰 동기화 업로드
4. **Review Number Update** (`every_1hour_review_number_update`): 상품별 리뷰 개수 갱신
5. **Review Tracking** (`every_1hour_review_tracking`): 리뷰 혜택 지급 추적 및 비회원 포인트 처리

##### every_15min_tracking_concurrency (15분 주기, 병렬 처리)
`task_1day_data`와 동일한 5개 하위 작업을 수행하되, `ThreadPoolExecutor`를 사용하여 쇼핑몰 단위로 병렬 처리한다.
- 최대 동시 쓰레드: 2개 (`MAX_THREAD_COUNT = 2`)
- 최대 동시 작업 수: 4개 (`MAX_IN_FLIGHT_TASKS = 4`)
- Bounded Queue 패턴: 작업 큐가 꽉 차면 하나가 완료될 때까지 대기한다.
- `singleton_task_lock`으로 중복 실행을 방지한다.
- 각 작업의 소요 시간이 60초 이상이면 경고 로그를 남긴다.

##### every_15min_tracking_1~4 (15분 주기, 분산 처리)
환경변수 `MIN_SHOP_ID`와 `MAX_SHOP_ID`를 통해 쇼핑몰 ID 범위를 지정하여 처리 대상을 분산한다. 동일한 5개 하위 작업을 순차 실행한다.

##### task_every_09am (매일 09시)
- VOC 리포트 메시지 발송 (`daily_09am_send_report_message`): 알림센터를 통한 부정리뷰 분석 리포트 전송

### 3.3 시스템 사이클(ECS Task) 관리

SystemCycle은 장기 실행이 필요한 작업을 AWS ECS Fargate 태스크로 분리하여 실행하는 시스템이다.

#### 실행 흐름
1. `RunnerData__Feature.run_task_by_shop()`이 호출되면 `RunnerData` 인스턴스를 생성한다.
2. `ECS__TaskRunner.run_task()`가 AWS ECS Fargate 태스크를 기동한다.
3. ECS 컨테이너에서 `python manage.py ecs_task --runner_id {id}` 커맨드가 실행된다.
4. `ecs_task` 커맨드는 `RunnerData`를 조회하여 `task_type`에 따른 프로세스를 실행한다.

#### ECS 태스크 유형별 처리

| Task__Type | 메서드 | 설명 |
|------------|--------|------|
| 테스트 (0) | `test_process` | 60초 대기 후 종료하는 컨테이너 동작 테스트 |
| 리뷰_최초이관 (100) | `review_initial_transfer_process` | 카페24 리뷰를 알파리뷰로 이관 |
| 리뷰_해지이관 (200) | `review_termitate_transfer_process` | 알파리뷰 리뷰를 카페24로 이관 |
| 리뷰_리뷰개수갱신 (300) | `review_number_update_process` | 상품별 리뷰 개수 필드 갱신 |

#### ECS 인프라 구성
- `ECS__TaskRunner`는 boto3 ECS 클라이언트를 사용한다.
- 네트워크: VPC 내 서브넷, 보안 그룹, Public IP 할당
- 실행 방식: Fargate 런치 타입
- 컨테이너 오버라이드: 커맨드 및 환경변수(DD_PROFILING_ENABLED=false) 재정의
- Datadog 프로파일링은 ECS 태스크에서 비활성화된다.

## 6. 마이크로서비스 구조

### 6.1 AI Review Analyzer

리뷰 텍스트에 대한 AI 분석을 수행하는 독립 서비스이다.

- **위치**: `services/ai_review_analyzer/`
- **실행 방식**: asyncio 기반 비동기 SQS 폴링
- **입력**: SQS 큐 (`AWS_REVIEW_AI_SQS_V2_NAME`)에서 `ReviewAnalyzeRequest` 수신
- **출력**: SNS 토픽 (`AWS_REVIEW_ENHANCEMENT_SNS_NAME`)으로 분석 결과 발행

#### 핵심 기능
- **구문 분리(Phrase Segmentation)**: 리뷰를 의미 단위 구문으로 분리
- **감성 분석(Sentiment Analysis)**: 각 구문의 부정 여부 판단
- **태그 부착(Tagging)**: 쇼핑몰별 활성 태그를 구문에 부착

#### 아키텍처 상세
- **AgentSelector**: 모델별 Rate Limit을 관리하며 사용 가능한 LLM을 선택
  - 모델 우선순위: 부정분석만일 경우 `gpt-5-mini`, 태그분석 포함 시 `haiku-4.5`
  - S3에서 모델 설정(config.json)을 15분 캐싱
  - `MultiResourceRateLimiter`로 모델별 요청/토큰 한도 관리
- **ReviewAnalyzer**: 분석 요청의 전체 라이프사이클 관리
  - 최대 3회 재시도 (일시적 오류: 429, 500, 502, 503, 504)
  - 지수 백오프 적용
- **프롬프트 시스템**: `ReviewAnalysisNegativePrompt` (부정분석) / `ReviewAnalysisNegativeAndTagPrompt` (통합분석)
  - 반어법/강조 처리 규칙 (Emphatic Positive)
  - 과거/일반적 불만 예외 규칙
  - 태그 Hallucination 방지 규칙

#### 에러 처리
- 503 에러 시 Redis 기반 재시도 카운트 관리 (최대 3회, 1시간 TTL)
- 최대 재시도 초과 시 메시지 삭제 후 실패 결과 SNS 발행
- 일반 에러 시 에러 결과 SNS 발행 후 메시지 삭제

#### 동시성 제어
- 폴링 대기 시간: 15초
- 폴링당 최대 메시지 수: 10개
- 최대 동시 처리 수: 30개 (`asyncio.Semaphore`)
- Logfire를 통한 분산 추적(distributed tracing) 계측

### 6.2 Keyword Service

- **위치**: `services/keyword_service/`
- **프로토콜**: gRPC (포트 50051)
- **최대 워커 수**: 10개 쓰레드

#### 제공 API
- `ExtractKeywords(shop_id, sentence)`: 리뷰 텍스트에서 키워드를 추출하여 반환
  - 반환값: keywords 목록, keyword_version, keyword_status, keyword_library

#### 내부 구현
- `WordCloud__Feature.extract_keywords()`: 형태소 분석 기반 키워드 추출
- 실패 시 빈 키워드 목록과 `FAIL` 상태를 반환

### 6.3 Web Crawler Service

네이버 블로그 등 외부 콘텐츠를 크롤링하는 gRPC 서비스이다.

- **위치**: `services/web_crawler_service/`
- **프로토콜**: gRPC (포트 50051)

### 6.4 Review Persistence Service

리뷰 생성/분석 이벤트를 처리하여 데이터를 영속화하는 서비스이다. Django ORM에 직접 접근한다.

- **위치**: `services/review_persistence_service/`
- **실행 방식**: 동기 SQS 폴링 (while True 루프)
- **입력**: SQS 큐 (`AWS_REVIEW_ENHANCEMENT_SQS_NAME`)에서 이벤트 수신
- **출력**: SNS 토픽 (`AWS_REVIEW_AI_SNS_NAME`)으로 분석 요청 발행

#### 처리하는 이벤트 유형

| 이벤트 | 프로세서 | 설명 |
|--------|----------|------|
| `ReviewCreatedEvent` (CREATE) | `ReviewCreatedEventProcessor` | 리뷰 생성 시 AI 분석 요청 생성 |
| `ReviewBatchAnalysisEvent` (BATCH_ANALYZE) | `ReviewBatchAnalysisProcessor` | 배치 AI 분석 요청 생성 |
| `ReviewAnalyzeSuccessResult` (ANALYZE_COMPLETE) | `ReviewAnalysisCompleteEventProcessor` | 분석 성공 결과 DB 저장 |
| `ReviewAnalyzeErrorResult` (ANALYZE_FAILED) | - | 분석 실패 로그 기록 |
| `ReviewTagDeleteEvent` (DELETE) | `ReviewTagDeleteEventProcessor` | 태그 삭제 이벤트 처리 |

#### 핵심 비즈니스 규칙
- **처리 대상 필터링**: 일반리뷰만 분석 대상, 최근 1개월 이내 작성된 리뷰만 처리
- **중복 요청 방지**: Redis 캐시(TTL 15분)로 동일 리뷰에 대한 중복 분석 요청 차단
- **멱등성 보장**: 분석 완료 이벤트는 Redis 캐시(TTL 15분)로 중복 처리 방지
- **이전 데이터 정리**: 새 분석 결과 저장 전 기존 Phrase, NegativePhrase, TagRel 삭제
- **트랜잭션 보장**: 분석 결과 저장은 `@transaction.atomic`으로 원자성 보장
- **삭제 중 태그 보호**: Redis 캐시로 삭제 진행 중인 태그에 대한 관계 생성 방지
- **배치 분석 진행 관리**: `BatchAnalysisCompletionHandler`로 배치 분석 진행 상태 추적

#### 에러 처리
- 메시지 처리 성공 시에만 SQS 메시지 삭제
- 실패 시 메시지를 유지하여 SQS 재시도 메커니즘 활용

### 6.5 Init Transfer Worker

코어 서비스에서 전달되는 최초이관 메시지를 처리하여 상품, 주문, 리뷰를 이관하는 서비스이다.

- **위치**: `services/init_transfer_worker/`
- **실행 방식**: 동기 SQS 폴링 (while True 루프)
- **입력**: SQS 큐 (`AWS_CORE_TO_REVIEW_MIGRATE_SQS_NAME`)
- **출력**: SQS 큐 (`AWS_REVIEW_TO_CORE_MIGRATE_SQS_NAME`)로 완료 알림

#### 처리하는 이벤트 유형

| 이벤트 | 설명 |
|--------|------|
| `MIGRATION__PRODUCT_START` | 상품 이관 시작 - ShopAutocrawling 상태 변경 |
| `상품생성` (90001) | 개별 상품 데이터 이관 |
| `MIGRATION__PRODUCT_END` | 상품 이관 완료 - 리뷰 크롤링 설정 수행 |
| `MIGRATION__ORDER_START` | 주문 이관 시작 |
| `주문생성` (90023) | 개별 주문 데이터 이관 |
| `MIGRATION__ORDER_END` | 주문 이관 완료 |
| `최초이관` (1) | 리뷰 크롤링 실행 (카페24 게시글 -> 알파리뷰) |

#### 핵심 비즈니스 규칙
- 형식에 맞지 않는 메시지는 즉시 삭제한다.
- 리뷰 크롤링 타겟은 `ShopAutocrawlingReviewTarget` 모델로 관리한다.
- 좀비 타겟 정리: 2시간 이상 "진행" 상태인 타겟은 자동으로 "실패" 처리한다.
- 모든 타겟 처리 완료 시 코어 서비스에 완료 메시지를 전송한다.
- `mall_id`와 `shop_no` 기반 쇼핑몰 조회 결과를 인스턴스 캐시에 저장한다.
