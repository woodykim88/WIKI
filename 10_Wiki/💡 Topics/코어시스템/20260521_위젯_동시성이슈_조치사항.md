# 위젯 세트 다중 생성 동시성 이슈 및 조치 사항 (2026-05-21)

## 1. 이슈 개요

알파리뷰 위젯(키워드 위젯, 작성버튼 위젯) 활성화 상태에서 특정 고객사의 상품 상세페이지 조회 시 위젯이 화면에 비정상적으로 미노출되는 현상이 발견되었습니다.

*   **With yeny** (고객사 ID: 8105) | 베이직 플랜 | **작성버튼 위젯 미노출** (https://withyeny.com/)

## 2. 발생 원인 (Root Cause Analysis)

위젯 세트(Widget Set)를 생성하는 시점의 **느슨한 초기화(Lazy Initialization) 정책**과 **동시성 제어(Concurrency Control) 부재**가 결합하여 발생한 레이스 컨디션(Race Condition) 에러입니다.

1.  **동적 생성 트리거의 문제**: 
    어드민 대시보드에서 위젯 사용 설정을 켜는 시점(Toggle On)에 위젯 세트 레코드를 즉시 DB에 생성하는 것이 아니라, 사용자가 해당 위젯이 포함된 쇼핑몰 상세페이지에 최초로 접근하여 위젯 요청 API가 발생하는 시점에 **"조회 후 없을 때 생성"**하는 조치 방식을 취하고 있었습니다.
2.  **동시성 충돌 (Race Condition)**: 
    위젯 세트 생성이 데이터베이스에 정상적으로 반영(Commit)되기 전 극히 짧은 찰나에, 상품 상세페이지로부터 여러 개의 위젯 조회 요청($N$개)이 비동기적으로 동시에 유입되었습니다.
3.  **데이터 무결성 훼손**: 
    그 결과 $N$개의 중복된 위젯 세트 레코드가 DB에 삽입되었고, 이후 정상적인 위젯 조회 쿼리 실행 시 단일 레코드가 아닌 여러 레코드가 반환되었습니다. 
4.  **시스템 에러 전환**: 
    조회 API가 "단 하나의 활성 위젯 세트"를 기대하는 상황에서 복수 개의 데이터가 검출되자, 백엔드는 "요청 대상이 불명확함"을 알리는 익셉션을 발생시키고 응답에 실패하여 위젯이 완전히 미노출되었습니다.

---

## 3. 조치 사항 (Resolution & Prevention Plan)

동시성 이슈를 근본적으로 차단하고 기존의 오작동 데이터를 구제하기 위해 두 가지 단계적 조치를 수행합니다.

### 1단계: 임시 디펜시브 코드 적용 (조회 단 레이스 컨디션 완화)
*   **작업 내용**: 위젯 조회 요청 처리 시, 데이터베이스 내에 동일 타입의 위젯 세트가 실수로 여러 개($N$개) 존재하더라도 API가 크래시되지 않고 **최신 생성된 1개의 위젯 세트만 안전하게 추출하여 반환**하도록 쿼리 조회 조건 및 예외 처리를 디펜시브하게 보강합니다.
*   **목적**: 동시성 이슈 발생 시에도 사용자 화면상의 위젯 노출이 깨지는 최악의 고객 경험(UX) 방지.

### 2단계: 위젯 세트 생성 시점 전환 (근본 원인 제거)
*   **작업 내용**: 키워드 위젯 및 작성버튼 위젯의 생성 시점을 **"쇼핑몰 페이지 첫 요청 시점"**에서 **"어드민 대시보드 내 사용 설정 토글 On 시점"**으로 완전히 변경합니다.
*   **목적**: 설정 시점에 동기적(Synchronous)으로 단 1회만 단일 위젯 세트가 생성되도록 보장하여, 동시 다발적인 클라이언트 요청으로 인한 중복 데이터 삽입 가능성을 원천 봉쇄합니다.
