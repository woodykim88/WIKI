# [[코어 데이터베이스 스키마 가이드]]

## 📌 한 줄 통찰 (The Karpathy Summary)
> 코어 시스템은 shop_id를 축으로 Shop-Subscription-Bill-Payment 계층 구조를 갖고, 현재 상태만 저장하며(이력은 History 테이블만), 소프트 딜리트를 기본으로 한다.

## 📐 도메인 모델 개요 (Domain Model Overview)

| # | 도메인 | 핵심 개념 |
|---|--------|-----------|
| 1 | 기본 개념 | 테이블=엑셀시트, 1:1/1:N/N:M 관계, shop_id 중심 조인 |
| 2 | 쇼핑몰(Shop) | Company→Shop(1:N), main_status 9단계, sub_status JSON, 보조 테이블 4종 |
| 3 | 앱/서비스 | App(제품단위) vs Service(과금단위), ShopApp/ShopService 시점 필드 |
| 4 | 설치/제거 요청 | InstallRemoveRequest 6종 유형, 6단계 상태 전이 |
| 5 | 구독/결제 | Bill→Subscription→Payment 계층, BillServiceItem/Discount 포함 |
| 6 | 월간 사이클 | MonthlySystemCycle: 주문 한도 관리, 일 2회 집계, 105% 초과 규칙 |
| 7 | 후불비용/예치금 | DefferedAmount(선사용후정산), PaidCash(선불잔액) |
| 8 | 이커머스 데이터 | Product/Order/Customer + 배송/쿠폰/포인트, 소프트딜리트 |
| 9 | 데이터 정책 | 현재 상태만 저장, updated_at 주의, 하드딜리트 예외 케이스 |

## 📖 핵심 도메인별 상세 (Domain Deep-Dive)

### 📖-1. 기본 개념 및 데이터 정책

**테이블 관계:**
- 1:1 (One-to-One): Shop ↔ ShopDetail
- 1:N (One-to-Many): Shop → Subscription (가장 흔함)
- N:M (Many-to-Many): 중간 테이블로 연결

**핵심 원칙: shop_id 중심**
- 거의 모든 테이블이 `shop_id` 필드를 가짐
- 쇼핑몰 ID만 알면 관련 데이터를 쉽게 조인 가능

**⚠️ 데이터 변경 이력 한계:**
- DB는 **현재 상태만 저장** — 특정 필드가 언제, 무엇에서 무엇으로 바뀌었는지 이력 없음
- `updated_at`은 아무 필드 수정 시 갱신 → 관심 필드 변경 시점 추적 불가
- **이력 추적 가능한 경우:** 별도 History 테이블이 있는 경우만 (DefferedAmountHistory, PaidCashHistory, SubscriptionExecuteHistory 등)

**데이터 삭제 정책:**
- 기본: **삭제하지 않음** — 상태 변경 또는 소프트딜리트(`is_deleted`)
- ⚠️ 하드딜리트 예외:
  - Management Command (`delete_orders`, `delete_product`, `delete_customer`, `delete_category`)
  - CASCADE 삭제 (FK에 `on_delete=CASCADE` 설정 — Shop 삭제 시 연쇄, 단 정상 운영에서 Shop 삭제 없음)
  - 이커머스 플랫폼 데이터: `is_deleted` 소프트딜리트

### 📖-2. 쇼핑몰(Shop)

**Company → Shop (1:N, 멀티쇼핑몰):**

| 필드 | 설명 |
| --- | --- |
| mall_id | 몰 ID |
| platform_id | 이커머스 플랫폼의 회사 ID |
| ecommerce_platform | "cafe24" 또는 "imweb" |

**Shop 주요 필드:**

| 필드 | 설명 |
| --- | --- |
| main_status | 쇼핑몰 메인 상태 (9단계) |
| sub_status | 세부 상태 JSON |
| service_started_at | 서비스 시작일 |
| service_stopped_at | 서비스 종료일 (운영 중이면 null) |

**main_status 상태 전이:**
```
계정활성화(100) → 앱설치(200) → 사이트생성(300) → 필수세팅완료(400)
→ 라이브(500) → 구독종료(600) → 해지완료(700)
분기: 리드이탈(800), 앱삭제(900)
```

**sub_status 키:**

| 키 | 값 |
| --- | --- |
| 재도입여부 | 최초도입=101, 재도입=102 |
| 기본정보등록여부 | 미등록=201, 등록완료=202 |
| 필수정보등록여부 | 미등록=301, 등록완료=302 |
| 카드등록여부 | 미등록=401, 등록완료=402, 카드등록면제=403 |
| 라이브유형 | 슈퍼관리자_구독예약중=501, 유료구독중=502, 체험구독중=503, 무료구독중=504, 계약구독중=505 |
| 구독종료유형 | 일시중지=601, 해지요청=602, 결제실패이탈=603, 기타=604, 앱삭제=605 |

**보조 테이블 (Shop과 1:1 조인: Shop.id = 테이블.shop_id):**

**ShopDetail:**
- number_of_monthly_orders, category, terms_and_privacy_agreement, webhook_active_status

**ShopSystemInfo (현재 상태 빠른 조회용):**
- plan_id (현재 플랜 FK), cur_subscription_id (현재 구독 FK), cur_monthly_system_cycle_id, latest_bill_number, plan_limit_option

**ShopStatistics (주 1회 갱신):**
- previous_months_order_count/revenue (설치시점 30일), current_months_order_count/revenue (이번달 30일)

**ShopCard (1:N):**
- nickname, company, number(16자리), is_business, customer_uid(빌링키)

### 📖-3. 앱(App)과 서비스(Service)

**앱 = 제품 단위** (웹훅 라우팅, 스크립트 설치, 대시보드 권한)
**서비스 = 과금 단위** (가격 책정, 구독/청구서/설치요청 연결)

관계: Service 테이블에 `app_id` FK

**ShopApp (쇼핑몰-앱 연결) 시점 필드:**

| 필드 | 업데이트 시점 | 의미 |
| --- | --- | --- |
| installed_at | 메인 서비스가 "프로덕트온보딩중" 전환 시 **최초 1회만** | 최초 도입일 (재설치 시 불변) |
| main_status | ShopService 상태 변경에 연동 자동 갱신 | ShopService와 동일 값 |

**ShopService (쇼핑몰-서비스 연결) 시점 필드:**

| 필드 | 업데이트 시점 | 의미 |
| --- | --- | --- |
| installed_at | "프로덕트온보딩중" 전환 시 **매번 갱신** | 마지막 설치일 |
| live_at | "라이브" 전환 시 **매번 갱신** | 마지막 라이브일 |
| stopped_at | "서비스중단" 전환 시 **매번 갱신** | 마지막 중단일 |

**⚠️ 주의:** ShopApp.installed_at은 최초 1회만, ShopService 시점 필드들은 매번 덮어씌워짐 (과거 이력 미보존)

**ShopService 상태 흐름과 시점 필드:**
```
구독없음(100) → 프로덕트온보딩중(200) [installed_at 갱신]
→ 라이브(300) [live_at 갱신] → 서비스중단(500) [stopped_at 갱신]
```

### 📖-4. 설치/제거 요청(InstallRemoveRequest)

조인: `Shop.id = InstallRemoveRequest.shop_id`, `Service.id = InstallRemoveRequest.service_id`

**요청 유형:**

| 코드 | 유형 | 언제 생성 |
| --- | --- | --- |
| 101 | 자동설치 | 시스템 자동 |
| 102 | 직접설치 | 관리자 직접 → 즉시 진행중 |
| 103 | 설치요청 | 쇼핑몰 요청 → 접수 대기 (알파리뷰 설치 시 주로 이 유형) |
| 201 | 자동제거 | 구독 종료 등 → 즉시 완료 |
| 202 | 직접제거 | 관리자 직접 → 즉시 진행중 |
| 203 | 제거요청 | 쇼핑몰 요청 → 접수 대기 |

**상태 전이:**
```
대기(100) ─── "접수" ───→ 진행중(200) ─── 완료 ───→ 완료(300)
    │                          │
    ├── "취소" → 취소(400)     ├── 실패 → 실패(600) → 재시도 → 완료(300)
    │                          └── "취소" → 취소(400)
    └── "삭제" → 제거(500) (연관 청구서도 취소)
```

**"접수" 시 동작:**
1. 상태 → 진행중(200)
2. 이커머스 플랫폼에 스크립트 설치/삭제
3. 과거 데이터 마이그레이션 (14~30일치)
4. 알림톡 발송
5. 웹훅 활성화

**설치 가능 조건:**
- 설치: "구독없음", "프로덕트온보딩중", "서비스중단" 중 하나
- 제거: "라이브" 또는 "제거중"
- ⚠️ 계약구독 중에는 자동설치/자동제거 불가

### 📖-5. 구독/결제 (Bill → Subscription → Payment)

**핵심 계층 구조:**
```
Bill (청구서) ← 먼저 생성, 독립 존재 가능
├── Subscription (구독) ← bill_id FK (Bill 필수)
│   ├── SubscriptionSection (기간 구간)
│   └── SubscriptionMonthlySystemCycle (월간 사이클 연결)
├── BillServiceItem (서비스별 금액)
│   └── ServiceItem → Service + Plan
├── BillDiscount (할인)
└── Payment (결제) ← 1:1 관계
    └── CardPaymentHistory
```

**Bill 주요 필드:**
- type: 구독="Subscription", 보조="Assistant", 임의="Custom"
- total_original_price, total_combine/yearly/contract/additional_discount_amount, total_price

**Subscription 주요 필드:**
- type: 체험구독=1, 무료구독=2, 정기구독=3, 계약구독=4
- period_type: 월간=1, 월간_3/6/9개월=11/12/13, 연간=2
- **현재 활성 구독:** main_status=201 또는 ShopSystemInfo.cur_subscription_id

**Payment 주요 필드:**
- cash(결제액), paid_cash(예치금결제액), method(카드=0, 계좌이체=1)
- main_status: 대기=100, 성공=200, 실패=400, 미수=402, 실패확정=403, 환불=600

**현재 플랜 확인 — 두 가지 방법:**
1. 빠른: ShopSystemInfo.plan_id → Plan 조인
2. 상세: Subscription → Bill → BillServiceItem → ServiceItem → Plan

### 📖-6. 월간 사이클(MonthlySystemCycle)

| 필드 | 설명 |
| --- | --- |
| start_date / end_date | 사이클 기간 |
| plan_id | 적용 플랜 FK |
| order_limit | 주문 한도 (Plan.order_limit × rate) |
| order_count | 현재 주문 수 |
| rate | 일할 비율 (기본 1.0) |

**생성:** 구독 시작 시 + 매일 00:00 배치 자동 갱신
**order_count 업데이트:** 매일 00:00, 10:00 — Order 테이블에서 해당 기간 주문 수 **처음부터 재집계** (Cafe24 주문만)
**한도 초과 규칙:** 105% 초과 시 plan_limit_option이 "안전"이면 알림만, "일반"이면 서비스 제한

### 📖-7. 후불비용 / 예치금

**후불비용(DefferedAmount) 흐름:**
```
① 포인트 사용 (잔액 차감, 마이너스 가능)
→ ② DefferedAmountHistory 기록 (type=사용)
→ ③ 구독 종료 시 AssistantBill 생성
→ ④ 결제 처리
→ ⑤ 잔액 정산 (type=충전)
```

**DefferedAmountHistory 필드:**
- type: 사용=100, 충전=200, 소멸=300
- source: API=1, QUEUE=5, 월간=100, 체험=200, 청구서=300
- amount, before_price, after_price, summary

**예치금(PaidCash):** Shop과 1:1, amount=현재 잔액
**PaidCashHistory:** before_price, amount(양수=충전, 음수=사용), after_price, summary

### 📖-8. 이커머스 플랫폼 데이터

연동 플랫폼: Cafe24, Imweb. 모든 테이블 `platform_id`(원본 ID) + `shop_id` 조인 가능.

**소프트 딜리트:** Product, Customer, CustomerGroup, Category 등 → `is_deleted=True`로 관리

**상품:** Product, ProductDescription, ProductStatistics, ProductOption, ProductOptionValue, ProductVariant, ProductCategory
**주문:** Order, OrderItem, OrderBenefit, OrderReceiver, OrderBuyer
**고객:** Customer, CustomerGroup, ShopCustomerOptOutHistory
**배송:** Shipping, ShippingRates, Supplier, SupplierShippingRates
**플랫폼 쿠폰/포인트:** ECommercePlatformNewBenefit, ECommercePlatformCoupon, ECommercePlatformPointAutoExpiration

### 📖-9. 자주 쓰는 조회 시나리오

| 시나리오 | 조인 경로 |
| --- | --- |
| 쇼핑몰 현재 상태 | Shop (main_status, sub_status) + ShopDetail + ShopStatistics |
| 현재 플랜 | Shop → ShopSystemInfo.plan_id → Plan |
| 서비스 및 요금 | ShopSystemInfo.cur_subscription_id → Subscription → Bill → BillServiceItem → ServiceItem → Service + Plan |
| 설치/제거 이력 | Shop → InstallRemoveRequest (type, main_status) → Service |
| 결제 실패 여부 | Shop → Subscription → Bill → Payment (main_status) |
| 주문 한도 초과 | ShopSystemInfo.cur_monthly_system_cycle_id → MonthlySystemCycle (order_count vs order_limit) |
| 후불비용 | Shop → DefferedAmountHistory |
| 적용 쿠폰 | Shop → ShopCoupon (main_status=201) → Coupon |
| 상품 판매 내역 | Product.sale_total 또는 Product → OrderItem |
| 고객 주문 이력 | Customer → Order → OrderItem + OrderReceiver |
