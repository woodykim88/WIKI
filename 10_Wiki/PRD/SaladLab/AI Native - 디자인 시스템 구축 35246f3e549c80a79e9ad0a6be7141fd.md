> *지금 우리고객은 어떤 문제를 겪고 있을까. 구체적인 사례와 데이터를 포함한다.*

## 1.1. 지금 우리가 풀어야 할 것

- 과거 워크 플로우 : 기획 → 디자인 hand-off → 개발 순으로 각 담당자별 역할과 절차대로 업무 진행
- 현재 워크 플로우 : 기획자 혹은 개발자가 AI로 먼저 디자인 → 디자이너가 이후에 피그마로 hand-off 문서를 작성하거나, 구두로 개선을 논의하며 진행
    - **🚨 근본적인 문제점**
        - **비효율** : 설계보다 구현이 먼저 일어나면서 이미 만들어진 결과물을 다시 정리하는 반복 문서 작업이 쌓인다.
        - **일관성 붕괴** : 컴포넌트 분리 없이 스타일이 산발적으로 적용된다. 컴포넌트 하나가 바뀌어도 전체에 반영되지 않아, 일관성을 유지하려면 일일이 찾아 수동으로 수정해야 한다.
            - 실측 데이터 사례
                
                현재 색상이 3곳에서 각각 다른 문법으로 정의되어 어느 것이 Single Source of Truth인지 불분명하다. 간격은 정의 자체가 없고 컴포넌트마다 `padding: 20px 24px` 등으로 하드코딩되어 있다. z-index는 1000, 10600 등 임의값이 혼재한다. `@saladlabinc/ui`의 50개+ 컴포넌트가 단일 모듈로 번들링되어 tree-shaking이 불가능하다. 앱별 로컬 컴포넌트도 중복 구현되어 있다 (alpha-review `shared/ui/` 15개, core-dashboard `shared/ui/` 2개). Storybook에도 기존 컴포넌트를 수정한 버전과 새로 만든 버전이 혼재하며 둘 다 실무에 쓰이고 있어, 하나를 수정해도 나머지에 반영되지 않는다.
                
        - **퀄리티 저하** : UIUX 전문성 없이 진행된 설계가 제품에 그대로 남아 완성도 낮은 사용자 경험으로 이어진다.
        - AI 네이티브 개발 불가 : 현재 구조의 가장 큰 문제는 UI가 불편한 것이 아니라, AI가 이해하고 생성하고 수정하기 좋은 형태로 UI 규칙이 구조화되어 있지 않다는 점이다. "버튼의 primary 색상은 무엇인가"라는 질문에 답하려면 `base.less`, `@saladlabinc/ui` CSS Variables, 앱별 `styles.less`를 모두 추적해야 한다. 사람은 암묵지로 처리하지만 AI에게는 명시적이고 단일한 소스가 있어야 한다. Figma MCP를 연결해도, 코드가 받아들일 구조가 없으면 자동화는 성립하지 않는다.
        
        → 일관성이 무너지면 수정 비용이 올라가고, 수정 비용이 올라가면 퀄리티 개선을 포기하게 된다.
        
    - 👍 **현재 방식이 가져온 장점** : 초안 속도가 빠르고 커뮤니케이션 비용이 줄었다. 이 장점은 유지한다.
- 🌸 **이상적인 워크플로우 — 우리가 만들고 싶은 상태**
    
    
    | 유형 | 현재 | 목표 |
    | --- | --- | --- |
    | 퀄리티 | AI가 디자인 시스템을 참고하지 않고 만든 결과물이 제품에 반영 | 디자인 시스템 안에서 일관된 UIUX로 완성 |
    | 효율 | 컴포넌트 수정 → 일일이 찾아 수동 반영 | 컴포넌트 하나 수정 → 제품 전체 즉시 반영 (SSOT) |
    | 속도 | AI로 빠르게 초안 생성, 이후 수정 비용 큼 | AI가 디자인 시스템 규칙 안에서 동작 → 초안 속도 유지 + 수정 비용 감소 |
    | AI Native | 규칙이 암묵지에 의존, AI가 일관된 결과 생성 불가 | 규칙이 구조화되어 AI가 토큰, 상태, 이벤트를 참조해 일관된 코드 생성 |

# 2️⃣. Solution

> *문제 해결을 위한 핵심 아이디어를 적는다.*
> 
- (필수) Breadboarding : 핵심구성요소 흐름을 텍스트로 설명

## 2.1. 핵심 아이디어

### **AI가 디자인 시스템 안에서만 창작하게 만든다.**

현재 문제의 본질은 AI 코딩 도구가 제약 없이 동작한다는 것이다. 디자인 시스템은 이 제약을 만드는 인프라다. 누가 아이디어를 내든, 어떤 도구로 구현하든, 결과물이 동일한 토큰·컴포넌트 위에서 나오도록 강제하는 구조를 만든다.

**① 단일 진실 공급원 확립** — 토큰이 Figma와 코드에서 동일한 값을 참조한다. 디자이너가 Figma 혹은 원본 Code에서 토큰값을 바꾸면 다른 코드에도 반영된다.

**② 컴포넌트 목록의 가시화** — Storybook이 "우리가 가진 컴포넌트 목록"을 공개한다. AI는 새 컴포넌트를 만들기 전에 이 목록을 먼저 확인하도록 프롬프트로 제약된다.

**③ AI 진입점 통제** — DESIGN.md와 AI 프롬프트 가이드라인이 AI 코딩 도구의 context에 항상 포함된다. AI는 이 파일을 읽고 토큰·컴포넌트 규칙 안에서만 코드를 생성한다.

## 2.2. 브레드보딩 및 시나리오

**시나리오 A — 기획자가 새 기능 아이디어를 AI로 구현할 때**

```haskell
기획자
  → Cursor·Antigravity에 UI 구현 요청
  → AI가 DESIGN.md를 context로 읽음
  → Semantic Token으로만 스타일 적용 (하드코딩 없음)
  → 기존 Standalone Component 재사용
  → 디자이너가 Figma에서 검토
  → 개발자가 headless 상태 모델 기반으로 구현
```

**시나리오 B — 디자이너가 토큰을 변경할 때**

```haskell
디자이너가 DESIGN.md에서 color.primary 값 수정
  → IDE에서 AI한테 "DESIGN.md 바뀐 값으로 토큰 파일 업데이트해줘"
  → AI가 tokens/ 파일 수정
  → CSS Variables 자동 생성
  → Git 브랜치 반영 → 개발자 확인 → 마스터 커밋
  → Cursor·터미널·Antigravity 전체 자동 반영
  → 수동 수정 0건
```

**시나리오 C — 개발자가 기존 컴포넌트에 새 상태를 추가할 때**

```haskell
개발자
  → headless/button/button.state.ts 열기
  → loading 상태 추가 (기존 상태 머신 확인하며 안전하게 확장)
  → adapter-angular가 자동으로 Angular Signal로 변환
  → 해당 컴포넌트를 쓰는 모든 화면에 자동 반영
```

## 버전2 (Big Batch — 6주 (5월 4일 ~ 6월 12일))

이번 베팅은 디자인 시스템 전체를 만드는 것이 아니라, **토큰 · Headless · Adapter** 3축이 하나의 핵심 컴포넌트와 신규 화면 1개를 통해 **수직으로 관통하는지** 검증하는 것이다.

### 3.1. 베팅 범위 안 (이 6주 안에 끝낼 것)

- **Foundation** — 토큰(Primitive → Semantic) 단일 정의 + CSS Variables 자동 생성
- **핵심 컴포넌트** — Headless + Adapter-Angular + Standalone Component (5/29까지 완료)
    - Select · Radio · Switch · Badge · Tag · Card · Tab · Tooltip 등
- **수직 관통 검증** — 신규 화면 1개에 새 시스템 적용, 토큰 1개 변경이 *수동 수정 0건*으로 반영됨을 확인
- **공존 검증** — 기존 `@saladlabinc/ui`를 쓰는 화면들이 *영향받지 않음* (회귀 테스트 baseline 대조)
- **AI 진입점 통제** — DESIGN.md 작성 + 팀 IDE 규칙 파일 등록 + lint(토큰 외 직접 입력 시 경고)

### 3.2. 베팅 범위 밖 (다음 베팅으로 미루는 것)

- SSOT
- adapter-lit / 위젯 환경 확장
- Figma ↔ 코드 자동 동기화 (옵션 A)
- 기존 `@saladlabinc/ui` 마이그레이션
- 조합 패턴(Patterns) 정의
- 디자인 시스템 독립 패키지화
- 전사 온보딩·확산
- lint (디자인 규칙 어기면 막는 도구 / 토큰 외 색·간격 직접 입력 시 경고)

### 3.3. 제약 사항 (감수하는 것)

- 기존 구조를 즉시 교체하지 않고 *신규 코드부터 점진적으로* 적용한다
- 초기 단계에서는 모든 UI에 완전한 일관성을 요구하지 않고, *일부 불일치와 임시 구조*를 감수한다
- 기존 `@saladlabinc/ui`와 새 구조가 *공존*하면서 발생하는 *중복과 운영 복잡성*을 감수한다
- Figma 완전 동기화는 *장기 목표*이며 이번 베팅 범위가 아니다

### 3.4. 스프린트 구조

### Anthropic 내부 디자인팀은 어떻게 일하고 있을까?

디자인 팀이 Claude Code를 써서 에러 상태·로직 흐름·시스템 상태를 파악하고, 개발 단계가 아닌 디자인 단계에서 엣지 케이스를 발견해 초기 디자인 품질을 근본적으로 개선하고 있어요. [Anthropic](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf)

그리고 "research preview" 문구를 전체 코드베이스에서 제거해야 할 때 Claude Code로 모든 인스턴스를 찾고, 주변 카피를 검토하고, 법무팀과 실시간으로 조율하고, 업데이트를 구현했는데 — 원래 일주일 걸리던 작업이 30분짜리 미팅 두 번으로 끝났어요. [Anthropic](https://www-cdn.anthropic.com/58284b19e702b49db9302d5b6f135ad8871e7658.pdf)

- **Claude.md 파일** — 팀마다 워크플로우·규칙·기대값을 Claude.md에 문서화하고 Claude Code가 읽어서 따르게 해요. 우리 DESIGN.md랑 완전히 같은 개념이에요
- **디자이너가 직접 코드 탐색** — **디자이너가 Claude Code로 코드베이스를 직접 읽고 엣지 케이스를 디자인 단계에서 파악**해요. hand-off 없이 디자이너-개발자 경계가 흐려지는 구조예요
- **Claude Design** — 코드베이스를 읽어서 디자인 시스템을 자동 추출하고 모든 프로젝트에 적용하는 방식
